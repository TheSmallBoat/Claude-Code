# 查询状态机与消息管线设计

## 文档状态

- 状态：Proposed
- 日期：2026-04-02
- 对应上游：ADR-002、模块接口清单

## 目标

定义 query loop 的内部状态、事件模型、流式处理阶段、工具回注逻辑和落盘边界，确保实现团队可以直接把“消息驱动内核”写成可测试的 reducer + driver 结构。

## 设计原则

- 状态机负责合法迁移，driver 负责副作用
- 所有外部输入先转成 `QueryEvent`
- 所有可持久化输出都先转成 `Message`
- 模型流、工具结果、权限决议、压缩结果都属于同一事件域

## 推荐状态集合

```rust
pub enum QueryState {
    Idle,
    PreparingInput,
    BuildingContext,
    StreamingModel,
    AwaitingPermission(PendingPermission),
    ExecutingTools(PendingToolBatch),
    ApplyingToolResults,
    Compacting(CompactionTicket),
    Stopping,
    Persisting,
    Completed(QueryResult),
    Failed(QueryError),
}
```

## 事件流原则

所有输入统一进入 mailbox，再由 reducer 推进状态：

```text
+------------------------+
| User / Host Event      |
+------------------------+
           |
           v
+------------------------+
| QueryEvent Mailbox     |
+------------------------+
           |
           v
+------------------------+
| Query Reducer          |
+------------------------+
   |                |
   |                +--> +------------------------+
   |                     | Side Effect Driver     |
   |                     +------------------------+
   v
+------------------------+
| State Snapshot         |
+------------------------+
```

## Reducer 与 Driver 分工

Reducer 只做以下工作：

- 校验当前状态是否接受某事件
- 生成新状态
- 产生命令型 effect 描述
- 决定哪些消息需要进入 transcript

Driver 只做以下工作：

- 调模型
- 执行工具
- 请求权限
- 触发压缩
- 写 transcript 与 session snapshot

## 模型流处理

模型流建议拆成三层：

1. 原始 provider 事件流
2. provider adapter 归一化后的 `ModelStreamEvent`
3. query reducer 可消费的 `QueryEvent::ModelStream`

实现约束如下：

- provider 特有字段不得直接泄漏到 query reducer
- tool use、plain text、thinking、stop reason 必须先归一化
- stream chunk 可增量写入 host sink，但 transcript 应以可重建单元落盘

## 工具回注循环

一次典型回合的执行语义如下：

1. query 接收用户输入。
2. context builder 构建输入上下文。
3. reducer 进入 `StreamingModel`。
4. 如果出现 tool use，则进入权限评估或工具执行。
5. 工具结果回注为 `MessageBlock::ToolResult`。
6. reducer 回到 `StreamingModel` 继续采样。
7. 终止条件满足后进入 `Persisting` 和 `Completed`。

## Transcript 写入边界

建议使用以下落盘原则：

- 用户输入：进入 query 前落盘
- 模型增量：在形成稳定 assistant message 后落盘
- tool call：在确定 tool batch 后落盘
- tool result：在执行完成并归一化后落盘
- stop / failure：写 session terminal event 与 final summary

## Continuation 与重入

query engine 应支持以下重入场景：

- 工具完成后的继续采样
- 权限批准后的继续执行
- 压缩完成后的继续采样
- 进程崩溃后的 session resume

因此需要区分：

- 内存态 `QueryState`
- 可恢复的 `PersistedQueryCheckpoint`

## 最小公开接口

```rust
pub trait QueryRuntime {
    fn submit(&self, input: QueryInput) -> Result<QueryTicket, QueryError>;
    fn dispatch(&self, query_id: QueryId, event: QueryEvent) -> Result<(), QueryError>;
    fn snapshot(&self, query_id: QueryId) -> Result<QuerySnapshot, QueryError>;
}
```

## 测试要求

至少应覆盖以下测试：

1. 无工具调用的直达回合。
2. 单轮工具调用后回注继续。
3. 多轮工具调用循环。
4. 权限拒绝后的停机语义。
5. 压缩插入后的继续执行。
6. query checkpoint 恢复。
