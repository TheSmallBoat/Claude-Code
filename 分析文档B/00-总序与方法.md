# 新一代 Code Agent 技术报告系列：总序与方法

## 报告目标

这套报告不是 TypeScript 到 Rust 的逐文件翻译，也不是对现有实现做语法层面的“语言迁移说明”。它的目标更高一层：

- 从第一性原理抽象 Claude Code 这类 Code Agent 的本质运行模型
- 从实际代码证据中提炼稳定的工程机制，而不是依赖产品宣传或外部传闻
- 将系统拆解为未来可在 Rust 中重建的核心构件、边界条件、约束规则与演进路径
- 区分“必须继承的架构资产”与“可以被替换的实现偶然性”

因此，报告聚焦的是工程要素之上的结构性知识：运行主线、控制平面、上下文工程、权限与安全、扩展体系、分布式能力、自治运行、多 Agent 编排，以及适合 Rust 重构的模块切分方式。

## 审计方法

本系列采用“证据优先、抽象后置”的方法。

### 方法一：先还原软件本体，再讨论语言迁移

对 Code Agent 而言，真正的系统本体不是某个 React 组件，也不是某个具体工具实现，而是以下七个层面共同构成的运行体：

1. 会话与消息状态机
2. 工具执行与权限决策内核
3. 上下文构造、压缩与记忆机制
4. 任务、子代理与后台执行模型
5. 扩展体系与外部能力接入协议
6. 远程会话与桥接分布式能力
7. 观测、恢复、门控与产品线控制

Rust 重构必须继承的是这些层面的不变量，而不是 TypeScript 的模块布局。

### 方法二：只接受代码证据支撑的结论

本次分析不以 README 的功能清单为准，而以源码中的实际入口、状态、调用链、边界对象和门控逻辑为准。文档内容仅作为辅助索引，不能单独作为结论来源。

### 方法三：优先抽取“长期稳定机制”

以下内容在本系列中会被弱化：

- 具体 UI 组件排版
- 个别命令的交互细节
- TypeScript 类型体操本身
- 只服务于当前框架生态的临时兼容代码

以下内容会被强化：

- 事件驱动和状态机边界
- 上下文与记忆的生命周期设计
- 多工具并发与串行的调度规则
- 本地/远程/后台/多代理的一致抽象
- 权限、安全、恢复、门控的系统化设计

## 审计边界

本仓库是恢复版源码树，不是官方上游仓库；同时，服务端实现并未完整包含在本地工作区。因此本系列的结论边界如下：

- 可以准确分析客户端运行时、协议使用方式、状态机和本地工程体系
- 可以依据客户端协议推断服务端接口契约与分布式协同模型
- 不会假定未出现在仓库中的服务端内部实现细节
- 对部分仅由 feature gate 或内部用户类型控制的能力，只讨论其客户端可见结构与工程意图

这意味着本系列适合作为“重构新一代本地 Code Agent”的技术储备，但不应被误读为对 Anthropic 内部全量平台的完整逆向说明。

## 核心结论摘要

在完成对关键入口、主循环、工具系统、任务系统、上下文工程、MCP、Bridge、Remote、KAIROS、Coordinator 及相关文档的交叉审计后，可以先给出九条顶层结论：

1. Claude Code 的本体不是“聊天 CLI”，而是一个本地优先、状态化、可持续运行的 agent runtime。
2. 系统的最核心资产不是命令集合，而是围绕“消息流 + 工具流 + 任务流”建立的统一执行语义。
3. Query 循环不是普通 API 调用封装，而是一个会反复进入“模型输出 -> 工具执行 -> 结果回注 -> 再推理”的回合机。
4. 上下文工程是系统中最关键的竞争力之一，包含提示构造、记忆注入、压缩、缓存稳定性和预算管理多个子系统。
5. 权限并不是 UI 层弹窗逻辑，而是一个独立的策略内核，直接决定 agent 是否安全可控。
6. 工具、任务、子代理、远程会话之间并不是彼此孤立的特性，而是同一执行内核在不同生命周期和隔离级别下的表现形态。
7. MCP、技能、插件、工作流共同构成了“能力扩展面”，说明系统的长期方向不是固化功能，而是平台化装配。
8. Bridge、Remote、KAIROS、Coordinator 说明该系统已经从单终端交互体进化为分布式、自治化、可远控、可编排的 agent 平台。
9. 如果要用 Rust 重构，真正应该重写的是“运行时内核”，而不是优先重写 CLI 外壳或 UI 组件。

## 总体判断

从工程视角看，Claude Code 已经不是“带几个工具调用的 LLM 客户端”，而是一个具备操作系统雏形的 agent runtime。它有自己的控制平面、上下文内存层次、权限内核、任务模型、扩展协议和远程协同能力。真正值得复用和重构的，是这套运行哲学与架构骨架。
