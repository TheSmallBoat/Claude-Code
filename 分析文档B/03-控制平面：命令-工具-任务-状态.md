# 03. 控制平面：命令、工具、任务、状态

## 本章定位

如果说 query loop 是执行内核，那么命令、工具、任务、状态就是控制平面。它们决定：

- 系统有哪些可调用能力
- 能力以什么边界暴露给模型或用户
- 长短生命周期工作如何区分
- UI 状态、会话状态、后台状态如何协同

在重构时，这一章直接决定 Rust 项目的核心对象模型。

## 命令系统：用户入口的语义层

[src/commands.ts](../src/commands.ts) 展示了命令系统的本质：

- 命令是用户显式触发的控制接口
- 命令可分为 prompt 型、本地执行型、内部命令和 feature-gated 命令
- 命令注册表可以汇聚内建命令、技能命令、插件命令和隐藏命令

命令系统的职责不是执行底层工作，而是：

- 对用户输入做高层语义路由
- 触发特定执行模式
- 为 agent 构造前置上下文或专用 prompt

因此命令层更像 shell 的“控制命令层”，而不是能力内核本身。

### 对 Rust 的含义

命令系统应与工具系统分离。命令负责 orchestration intent，工具负责 capability execution。

## 工具系统：模型可调度的能力层

[src/Tool.ts](../src/Tool.ts) 与 [src/tools.ts](../src/tools.ts) 共同定义了工具系统的边界。

### Tool 的工程角色

工具在系统中承担四类职责：

1. 向模型公开能力与输入 schema
2. 在执行前接受权限决策约束
3. 在执行中读写统一上下文 `ToolUseContext`
4. 以消息结果或上下文修正反馈给 query loop

这意味着工具不是简单函数，而是“受策略控制的外部能力端点”。

### ToolUseContext 的重要性

`ToolUseContext` 暗示了 Claude Code 的真实运行模型：

- 工具不直接依赖全局单例
- 工具通过上下文访问 app state、messages、permission、MCP、agent definitions
- 工具可携带 query tracking、content replacement state、requestPrompt 等复杂运行时信息

这是一种轻量依赖注入模型。对 Rust 来说，最合适的对应物不是全局 mutable state，而是带 trait 的执行上下文对象。

### 工具注册表是平台能力总表

`getAllBaseTools()` 清楚表明，Claude Code 把所有潜在能力都汇聚到统一注册表中，再按 feature、环境、权限模式做筛选。这样做的好处是：

- 能力发现统一
- prompt cache 前缀可稳定控制
- agent/worker/SDK 可以共享同一能力定义源

## 任务系统：长生命周期执行单元

[src/Task.ts](../src/Task.ts)、[src/tasks.ts](../src/tasks.ts) 与 [src/tasks/types.ts](../src/tasks/types.ts) 说明，Claude Code 明确区分了两种执行对象：

- Tool：回合内短生命周期调用
- Task：跨回合或后台持续运行单元

### 为什么需要 Task

因为很多工作不适合被建模为一次工具调用：

- 长时间 Bash 作业
- 后台 agent
- remote agent
- workflow
- monitor
- dream consolidation

Task 的存在，是系统从“工具调用 agent”演进为“持续运行 agent runtime”的分水岭。

### Task 的统一字段说明了什么

`TaskStateBase` 包含：

- id / type / status
- description
- start / end time
- output file / offset
- notified

这说明任务系统不是 UI 辅助结构，而是具备持久可观察性的正式运行对象。

### LocalAgentTask 揭示的后台 agent 模型

[src/tasks/LocalAgentTask/LocalAgentTask.tsx](../src/tasks/LocalAgentTask/LocalAgentTask.tsx) 体现出后台 agent 的真实语义：

- 每个后台 agent 都有独立 task state
- 可记录 progress、message transcript、pending messages
- 具备 foreground/background/retain/evict 生命周期
- 完成后以 `task-notification` 形式回注主会话

这里非常关键的一点是：

> 后台代理不是与主会话分裂的“另一个世界”，而是通过任务状态和消息回注重新回到主控制平面。

## 双层状态模型

Claude Code 没有把所有状态都塞进同一个 store，而是形成了双层状态结构。

### 第一层：Bootstrap State

[src/bootstrap/state.ts](../src/bootstrap/state.ts) 维护的是进程级和会话级全局状态，例如：

- sessionId
- projectRoot / cwd
- telemetry counters
- main loop model
- kairosActive
- prompt cache 和 tracing 相关状态
- session-only flags

这层更像 runtime kernel state。

### 第二层：AppState

[src/state/AppStateStore.ts](../src/state/AppStateStore.ts) 维护的是交互态、任务态和 UI 可观察态：

- toolPermissionContext
- tasks
- MCP clients/tools/resources
- notifications / elicitation queue
- bridge / remote 状态
- plugin / agent definitions
- todo / attribution / fileHistory

这层更像 runtime view-model state。

### 为什么双层结构成立

因为两类状态的生命周期、更新频率、访问对象并不相同：

- Bootstrap State 更偏进程基础设施
- AppState 更偏交互和任务编排

Rust 重构时，建议显式拆为：

- `RuntimeState`
- `SessionState`
- `ViewState`

而不是混成一个超级结构体。

## 消息是控制平面的统一语义

虽然本章主题是控制平面，但真正把命令、工具、任务、状态串起来的是消息。

系统中至少有四类关键消息用途：

1. 用户输入
2. 模型输出
3. 工具结果
4. 任务通知与系统事件

LocalAgentTask 通过 XML task notification 回注主线程；bridge/remote 通过 SDKMessage 映射到本地 message；slash command 会生成 local command/system message。所有这些都最终回到消息流。

这是一种非常强的统一抽象：

> 控制平面通过不同对象组织能力，但执行语义最终统一沉淀为消息流。

## 控制平面的边界划分

可以把 Claude Code 的控制平面总结为四条边界。

| 边界 | 问题 | Claude Code 的答案 |
| --- | --- | --- |
| 用户入口边界 | 用户如何表达意图 | Commands |
| 能力执行边界 | 模型如何调用外部能力 | Tools |
| 长期工作边界 | 长耗时或后台工作如何运行 | Tasks |
| 运行时观察边界 | 系统当前状态如何被查看与修改 | AppState + Bootstrap State |

这四条边界清晰，是系统可扩展的核心原因。

## 对 Rust 重构的对象模型建议

### 建议的核心模块

1. `command-core`
2. `tool-core`
3. `task-core`
4. `runtime-state`
5. `session-state`
6. `message-ir`
7. `agent-context`

### 建议的关键 trait

1. `CommandHandler`
2. `ToolExecutor`
3. `TaskRunner`
4. `PermissionEvaluator`
5. `ContextAccessor`
6. `TranscriptStore`

### 不要做的事

1. 不要把 Task 简化成异步 Tool。
2. 不要让 Tool 直接依赖全局状态单例。
3. 不要把 UI 状态与会话内核状态硬耦合。
4. 不要把 Commands 与 Tools 混成统一注册表。

## 本章结论

Claude Code 的控制平面相当成熟。它清楚地区分了：

- 谁负责接受用户意图
- 谁负责暴露可执行能力
- 谁负责承载长生命周期工作
- 谁负责维护运行时状态

这种边界清晰度，是重构时最应该继承的资产之一。
