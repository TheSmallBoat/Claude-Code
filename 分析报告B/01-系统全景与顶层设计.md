# 01. 系统全景与顶层设计

## 一句话定义

Claude Code 的工程本体可以定义为：

> 一个以本地终端为主要宿主、以消息流为统一控制语义、以工具和任务为执行载体、以内存与压缩为上下文维持机制、以 MCP/插件/远程桥接为扩展外壳的状态化 Code Agent 运行时。

这一定义比“CLI 产品”更准确，因为它解释了为什么仓库同时存在以下能力：

- 交互式 REPL
- 非交互式 SDK/打印模式
- 后台任务与持久会话
- 子代理与多代理编排
- MCP 外部能力接入
- 远程桥接和云端会话
- 自动记忆、自动压缩和自治行为

这些功能并非附属品，而是同一个运行时架构在不同执行场景中的展开。

## 顶层设计目标

从源码可以归纳出系统的七个顶层设计目标。

1. 让模型输出具备可执行性，而不是停留在文本建议。
2. 让执行过程保持连续会话状态，而不是一次性请求响应。
3. 让上下文在长时间运行中可控、可压缩、可恢复。
4. 让能力面可扩展，而不是把所有功能硬编码在主程序中。
5. 让敏感操作受权限与策略内核约束，而不是由提示词“自觉遵守”。
6. 让同一运行时同时支持本地、人机交互、远程控制和后台自治。
7. 让产品功能可以通过编译、用户类型和远程门控进行产品线分层。

这些目标分别映射到代码中的不同层次，而不是集中在某一个模块中实现。

## 分层架构

从工程结构看，系统可以抽象为六个平面。

```text
+--------------------------------------------------------+
| Bootstrap Plane                                         |
| entrypoints / init / setup / main                       |
+--------------------------------------------------------+
                                 |
                                 v
+--------------------------------------------------------+
| Control Plane                                           |
| commands / tools / tasks / state                        |
+--------------------------------------------------------+
    |                         |
    |                         +--> +--------------------------------------+
    |                              | Extension Plane                      |
    |                              | MCP / plugins / skills / workflows   |
    |                              +--------------------------------------+
    v
+--------------------------------------------------------+
| Execution Plane                                         |
| query loop / model stream / tool orchestration          |
+--------------------------------------------------------+
    |
    +--> +--------------------------------------+
    |    | Distribution Plane                   |
    |    | bridge / remote / subagent /         |
    |    | coordinator                          |
    |    +--------------------------------------+
    v
+--------------------------------------------------------+
| Context Plane                                           |
| prompt / memory / compaction / cache                    |
+--------------------------------------------------------+

All planes are constrained by:

+--------------------------------------------------------+
| Governance Plane                                        |
| feature gates / policy / telemetry / recovery           |
+--------------------------------------------------------+

Bootstrap  ----+
Control    ----+
Execution  ----+--> Governance Plane
Context    ----+
Extension  ----+
Distribution --+
```

### 1. Bootstrap Plane

负责把进程从“一个 CLI 程序”提升为“一个可运行的 agent 环境”。

核心职责：

- CLI 参数与特殊 fast path 分流
- 配置、证书、代理、认证、遥测、策略等底层环境初始化
- 预读与懒加载的性能编排
- REPL、headless、daemon、remote-control 等不同入口的模式切换

### 2. Control Plane

负责定义系统可做什么、以什么边界做、如何对外暴露。

核心对象：

- Command
- Tool
- Task
- AppState
- Bootstrap State
- Session/Transcript

这是系统的“语义边界层”。

### 3. Execution Plane

负责驱动一次实际 agent 回合的执行。

核心机制：

- 输入解析
- 系统上下文装配
- 模型流式调用
- 工具调用检测
- 工具执行与回注
- 继续/终止状态转移

这一层是系统最像“操作系统内核”的部分。

### 4. Context Plane

负责管理模型看到什么，以及什么时候忘、什么时候压缩、什么时候把经验写回系统记忆。

核心机制：

- system prompt section 组装
- 用户/系统上下文注入
- CLAUDE.md 和记忆文件装配
- Session Memory
- Auto Dream
- 自动压缩与 reactive compact
- prompt cache 稳定性控制

### 5. Extension Plane

负责让系统具备平台化扩展能力，而不是被主仓库代码封死。

扩展来源：

- MCP servers
- skills
- plugins
- workflow scripts
- built-in commands

这使 Claude Code 更接近一个 agent 平台，而不是固定功能产品。

### 6. Distribution Plane

负责把同一运行时扩展到不同执行地点和不同角色。

表现形态：

- 本地主线程 agent
- forked subagent
- local background agent task
- remote session viewer
- bridge-controlled local runtime
- coordinator/worker 多代理编排
- assistant/kairos 持续运行代理

这一层解释了系统为何天然适合演化成跨设备、跨会话、跨角色的 agent 平台。

## 核心对象模型

系统虽然模块很多，但抽象中心并不复杂。它围绕七类对象组织。

| 对象 | 工程含义 | 在系统中的位置 |
| --- | --- | --- |
| Session | 一个持续的 agent 工作上下文 | transcript、history、remote session、assistant mode |
| Turn | 一轮用户输入驱动的推理与执行闭环 | QueryEngine + query loop |
| Message | 系统统一语义载体 | 用户、助手、系统、进度、工具结果 |
| Tool Use | 模型触发的外部能力调用 | tools.ts、tool orchestration |
| Task | 长生命周期后台执行单元 | LocalShell、LocalAgent、RemoteAgent、Dream 等 |
| Memory | 系统的长期和中期上下文储备 | CLAUDE.md、session memory、auto memory |
| Transport | 本地与远程之间的状态同步通道 | bridge、remote、MCP、WS/SSE/HTTP |

未来 Rust 重构时，优先保持这七类对象的语义稳定，比保持现有目录结构更重要。

## 设计风格特征

### 特征一：本地优先，而不是云优先

即便存在 Ultraplan、Bridge、Remote Session 等能力，系统的基本身份依然是本地进程。大量设计选择都围绕“本地作为可信执行宿主”展开：

- cwd、git、worktree、terminal、permission 都以本地环境为中心
- Bridge 是把远端输入映射到本地执行，而不是把本地完全让位给云端
- Remote viewer 也是对远端会话的消费，而不是替代本地内核

### 特征二：状态驱动，而不是函数式调用驱动

代码中几乎所有关键能力都依赖持续状态：

- 会话消息历史
- 工具权限上下文
- 任务状态
- 记忆提炼状态
- auto compact 跟踪状态
- remote/bridge 连接状态

这说明新系统若想具备同等级能力，必须围绕显式状态机设计，而不是把 agent 当成一组 stateless RPC handler。

### 特征三：产品线化，而不是单版本化

从 [docs/07-feature-gates.md](../docs/07-feature-gates.md) 与多处 `feature()` / `USER_TYPE` / GrowthBook 代码可见，系统从一开始就不是单一产品，而是一个受多层门控控制的代码平台。

这一点对新一代项目非常关键：

- 能力门控要从架构层设计，而不是后期补 if/else
- 内部试验、灰度发布、企业策略限制都应被视为一等公民
- Rust 重构时应保留“能力裁剪”与“远程配置”这两类机制

### 特征四：性能意识非常强

从 `main.tsx` 和 `entrypoints/init.ts` 可以看到大量显式性能设计：

- 顶层 fast path
- 动态导入重模块
- telemetry 延迟初始化
- keychain/mdm 预取并行化
- query config 快照化
- prompt cache 稳定性优化
- 工具并发执行上限控制

这说明系统设计目标不是“能跑即可”，而是“高频交互下的持续低延迟”。

## 新一代 Rust 设计应继承什么

### 必须继承

1. 会话化消息状态机
2. 工具/任务统一执行语义
3. 上下文构造与压缩内核
4. 权限与安全策略内核
5. 扩展协议层
6. 本地与远程统一运行模型

### 可以重构或替换

1. React/Ink UI 细节
2. TypeScript 模块边界和文件布局
3. 现有部分历史兼容逻辑
4. 前端式 store 组织形式
5. Bun/Node 特定加载与 bundling 习惯

## 本章结论

Claude Code 的工程重心并不在某个单一功能，而在于它已经形成了一个相对完整的 agent operating model。未来用 Rust 重建时，真正需要复制的是这种 operating model 本身：

- 以消息为统一语义
- 以工具和任务为执行单元
- 以上下文工程维持长期可用性
- 以权限与门控维持可控性
- 以扩展和平面化分层支持长期演化
