# ADR-003：上下文与记忆系统

## 状态

Proposed

## 日期

2026-04-02

## 背景

Code Agent 的真正瓶颈通常不在模型调用，而在上下文管理。项目规则、用户偏好、会话历史、长期记忆、压缩结果、缓存稳定性都会影响实际行为。如果缺少分层的上下文与记忆体系，系统会快速退化为短上下文工具调用器。

## 决策

新一代系统采用“分层上下文 + 分层记忆 + 策略化压缩 + 缓存稳定器”的上下文架构。

### 上下文分层

1. 基础系统提示层
2. 项目上下文层
3. 用户上下文层
4. 会话上下文层
5. 相关记忆注入层
6. 模式附加上下文层
7. 临时覆盖层

### 记忆分层

1. `ProjectMemory`
2. `RelevantMemorySelection`
3. `SessionMemory`
4. `LongTermMemory`
5. `DreamConsolidation`

### 压缩分层

1. `SessionMemoryCompaction`
2. `ConversationCompaction`
3. `ReactiveCompaction`
4. `ManualCompaction`

## 决策理由

### 理由一：不同上下文有不同生命周期

项目规则、当前会话摘要、长期记忆和临时模式覆盖不应混为一体，否则缓存和更新策略会紊乱。

### 理由二：上下文是预算资源

上下文必须服从 token window、输出保留、warning threshold、blocking threshold 等预算约束。

### 理由三：长期记忆维护应异步化

Session Memory 与 Dream Consolidation 本质上都是后台 agent 任务，不应阻塞主交互。

### 理由四：缓存稳定性是一级目标

只有显式维护 cache-safe prefix，系统才能在长会话、多代理和多宿主条件下保持可接受成本。

## 方案细节

### 核心抽象

```rust
pub struct ContextBundle {
    pub system_prompt: Vec<PromptSection>,
    pub user_context: Vec<ContextLayer>,
    pub system_context: Vec<ContextLayer>,
    pub memory_artifacts: Vec<MemoryArtifact>,
    pub cache_safe_prefix: CacheSafePrefix,
}

pub trait ContextBuilder {
    fn build(&self, request: &ContextBuildRequest) -> Result<ContextBundle, ContextError>;
}

pub trait MemorySelector {
    fn select(&self, query: &str, inventory: &[MemoryHeader]) -> Result<Vec<MemoryArtifact>, MemoryError>;
}

pub trait CompactionPolicy {
    fn should_compact(&self, state: &BudgetState, transcript: &TranscriptView) -> CompactionDecision;
}
```

### 强制约束

1. 所有 prompt section 必须有独立 identity。
2. 会破坏 cache 的 section 必须显式标记。
3. 任何 compact 都必须生成可审计的边界或摘要产物。
4. Session Memory 和 Dream 任务必须能在无 UI 情况下运行。

## 被拒绝的备选方案

### 方案 A：把所有上下文拼成一个字符串再缓存

拒绝原因：无法表达生命周期与 cache-break 细粒度差异，也不利于增量重建。

### 方案 B：只保留会话摘要，不保留长期记忆

拒绝原因：会丢失跨会话知识积累能力，系统难以从一次性交互进化到持续助手。

### 方案 C：只在 prompt too long 时做紧急压缩

拒绝原因：这会把上下文治理降格成故障处理，而不是运行时资源管理。

## 后果

### 正面后果

1. 上下文治理将更稳定、可预测。
2. 长会话和多会话质量更高。
3. 可逐步演进出更强的长期记忆系统。

### 负面后果

1. 系统实现复杂度提高。
2. 需要额外维护 memory inventory、selector、compaction artifacts。

## 实施建议

1. 第一阶段先实现 ContextBundle、BudgetState 与 basic compaction。
2. 第二阶段实现 SessionMemory。
3. 第三阶段实现 Dream Consolidation 与记忆索引体系。
