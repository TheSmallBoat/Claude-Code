# Code Agent Runtime Core 萃取范围与文档路线图

## 文档状态

- 状态：In Progress
- 日期：2026-04-03
- 目标：从 TS-Source 中萃取可用于 Rust 基线版本实现的 Runtime Core 规格体系

## 1. 本轮审计的第一性目标

本轮工作不是复刻现有 TypeScript 工程，也不是做逐文件语言迁移，而是从现有代码资产中逆向提炼出一套更稳定、更可迁移、更适合 Rust 实现的内核级规格。

本次要保留的，是系统的长期稳定机制：

1. 会话与回合状态机。
2. 模型调用与工具回注闭环。
3. 上下文装配、压缩、记忆和预算控制。
4. 工具调度、权限约束、任务抽象。
5. 能力扩展与领域适配的核心接口。

本次要主动剥离的，是当前产品形态中的偶然性：

1. 界面排版、交互细节、TUI/REPL 外观。
2. 命令包装、展示消息、提示词文案细枝末节。
3. 平台化附属能力、品牌化表达、陪伴式功能。
4. 与 Runtime Core 解耦的宿主逻辑和外围集成。

## 2. 本次萃取对象

### 2.1 纳入范围

| 层面 | 内容 | 审计目的 |
| --- | --- | --- |
| Agent Runtime Core | Query 主循环、消息状态机、恢复路径、停止条件 | 还原内核执行语义 |
| Context Plane | system prompt、user/system context、记忆装配、缓存安全前缀 | 还原上下文工程 |
| Budget & Compact | token 预算、microcompact、autocompact、reactive compact | 还原成本与稳定性机制 |
| Tool Plane | Tool 抽象、调度、并发/串行规则、上下文突变 | 还原领域能力接入面 |
| Task Plane | 后台任务的最小抽象、任务生命周期、与 query 的耦合点 | 还原长期运行能力 |
| Session Plane | Conversation 级状态、转录持久化、resume 所需的不变量 | 还原会话内核 |
| Programming Adaptation | 文件读写、搜索、Shell、Notebook 等编程向能力的共性接口 | 为 Rust 能力层定义边界 |

### 2.2 暂不纳入范围

| 类别 | 当前处理方式 | 原因 |
| --- | --- | --- |
| TUI/REPL 组件 | 排除 | 不属于内核能力本体 |
| Buddy、宠物、视觉反馈 | 排除 | 与基线 Runtime Core 无关 |

| 插件市场、安装器、渠道装配 | 暂缓 | 属于平台分发面，不是最小内核 |
| Chrome/computer-use 等特化能力 | 暂缓 | 不属于编程内核最小集合 |

1. 先读最小必要代码闭环。
2. 立即沉淀工作笔记，记录证据、术语和关键不变量。

## 5. 当前对基线架构的初步判断

从已审计代码看，最值得继承的不是 CLI 命令集合，而是如下结构：

1. `Conversation Engine` 与 `Turn Executor` 的两层分工。
2. “预处理上下文 -> 流式采样 -> 工具执行 -> 附件/记忆回注 -> 再采样”的回合闭环。
3. 以消息数组为中心的统一事实源，而不是四散的状态对象。
4. 将预算、压缩、恢复、停止钩子作为 query 内核的标准阶段，而不是异常旁路。
5. 将工具系统视为内核扩展协议，而不是若干散落命令实现。

这意味着 Rust 版本的第一优先级不应是 UI 或命令系统，而应是以下几个 crate 或模块：

1. `agent-message`
2. `agent-query`
3. `agent-context`
4. `agent-tools`
5. `agent-permissions`
6. `agent-session-store`
7. `agent-tasks`

## 6. 文档验收标准

当 `分析文档C` 的文档集满足以下条件时，才算具备“可支撑基线版本实现”的成熟度：

1. 能明确回答每个内核模块的职责、边界、输入、输出和失败模式。
2. 能给出回合状态机、消息流和工具流的精确定义。
3. 能说明预算、压缩、记忆、权限、持久化之间的先后顺序与耦合点。
4. 能把 TS 中的动态对象语义转写为 Rust 中可实现的数据结构与 trait 边界。
5. 能给出可执行的测试矩阵，而不仅仅是架构图。

## 7. 本轮输出定位

本轮不是终局，而是 Phase 1 的落点：

1. 固定萃取边界，避免后续文档发散。
2. 建立首个真正可实现的模块规格样本。
3. 用评审清单提前暴露缺口，防止“写了很多但还不能开工”。

后续每完成一个核心模块，都会同步补充对应工作笔记、模块规格和评审状态。
