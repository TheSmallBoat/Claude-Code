## 2. 第一判断：Tool 不是“函数”，而是能力描述符

`Tool.ts` 里的 `Tool` 接口暴露的信息量远超一个可调用函数。它至少同时承担：

1. 能力元数据描述。
2. 输入 schema 与结果映射。
3. 并发安全性声明。
4. 只读/破坏性/交互性声明。
5. 权限检查入口。
6. 结果序列化与宿主渲染适配。

因此，在 Rust 中必须拆开处理：

1. Runtime Core 只保留能力描述、输入校验、执行和权限元数据。
2. UI/TUI/SDK 展示渲染全部下沉到宿主层。

## 3. buildTool 暴露出的默认策略很重要

`buildTool()` 给 Tool 注入默认行为，透露出系统的保守设计：

1. `isConcurrencySafe` 默认 `false`。
2. `isReadOnly` 默认 `false`。
3. `isDestructive` 默认 `false`。
4. `checkPermissions` 默认返回 allow，由通用权限引擎补足最终判定。

其中最值得继承的是前两项：

1. 未声明并发安全，就按不安全处理。
2. 未声明只读，就按可能写处理。

这对 Rust 基线版很关键，说明 scheduler 与 safety classifier 应该 fail-closed。

## 4. tools.ts 表明“工具清单装配”本身是独立平面

`tools.ts` 不是简单导出数组，而是：

1. 汇聚全部 base tools。
2. 按 feature gate、环境变量、用户类型、工作模式过滤。
3. 形成系统 prompt 要暴露给模型的能力集合。

因此必须把以下两层分开：

1. `ToolRegistry`：系统内存在的全部工具描述。
2. `ToolSetResolver`：根据 runtime mode、feature gates、permission context 得到当前 turn 的可见工具集。

## 5. 工具调度器的关键语义

`services/tools/toolOrchestration.ts` 很清楚地说明：

1. 工具不会逐个盲目执行。
2. 先按 `isConcurrencySafe(input)` 切分批次。
3. 并发批次允许并发跑多个只读工具。
4. 并发批次中的 context modifier 会被暂存，待批次完成后再按原顺序提交。
5. 非并发安全工具始终串行执行。

这说明工具调度器真正要解决的是“副作用序控制”，而不是线程池调度。

## 6. ToolUseContext 的另一层意义

对工具系统来说，`ToolUseContext` 不是普通参数，而是：

1. 会话状态访问器。
2. 权限上下文读取入口。
3. 中断句柄。
4. 进度/通知/SDK 输出接口。
5. 当前消息视图。

这意味着 Rust 中的工具执行接口不能只长这样：

```rust
fn call(input) -> output
```

至少应长这样：

```rust
async fn execute(input, exec_ctx, progress_sink) -> ToolOutcome
```

## 7. 权限内核的边界非常清晰

`hooks/useCanUseTool.tsx` 展示了一个非常重要的边界：

1. `hasPermissionsToUseTool()` 负责生成核心权限决策。
2. `useCanUseTool()` 只是把该决策接到不同宿主路径：
   - 直接 allow
   - 直接 deny
   - coordinator worker 自动检查
   - swarm worker 转发
   - interactive dialog

因此，真正属于 Runtime Core 的，是权限求值引擎，不是权限弹窗。

## 8. 权限流水线的顺序

从 `permissions.ts` 的 `hasPermissionsToUseToolInner()` 和 `checkRuleBasedPermissions()` 看，权限求值顺序大致如下：

1. 整个工具级 deny rule。
2. 整个工具级 ask rule。
3. tool-specific `checkPermissions()`。
4. tool-specific deny 结果。
5. `requiresUserInteraction()` 强制 ask。
6. 内容级 ask rule 不可被 bypass 覆盖。
7. safety-check ask 不可被 bypass 覆盖。
8. permission mode 决定是否 bypass。
9. 整个工具级 allow rule。
10. 若仍为 passthrough，则转成 ask。

这套顺序说明：

1. deny/ask 优先于 allow。
2. 路径安全与显式 ask 规则是 bypass-immune 的。
3. 工具可以提供内容级权限子规则，但最终仍归入统一权限流水线。

## 9. 文件系统权限规则体现出“多层安全”

`filesystem.ts` 的读写权限逻辑很值得单独保留，因为它不是简单的路径 allowlist。

### 9.1 read 权限的关键顺序

1. 先做 UNC 路径和可疑 Windows 路径拦截。
2. 先查 read-specific deny rule。
3. 再查 read-specific ask rule。
4. 再判断“edit access implies read access”。
5. 再判断是否位于 working directories。
6. 再判断是否为 internal readable path。
7. 再查 allow rule。
8. 最后默认 ask。

### 9.2 write 权限的关键顺序

1. deny rule 优先。
2. internal editable path 早于危险路径检查。
3. `.claude/**` 的 session 级 allow 有特殊豁免路径。
4. 其后才进入更严格的 auto edit safety checks。

这说明 Rust 中应把文件系统权限当作独立子策略，而不是散进单个 FileEditTool 里。

## 10. permission mode 不是枚举值，而是带副作用的状态机

`permissionSetup.ts` 的 `transitionPermissionMode()` 表明：

1. mode 切换会触发 side effects。
2. plan mode 进出会影响附件和恢复状态。
3. auto mode 进入时会 strip dangerous permissions。
4. auto mode 退出时会 restore dangerous permissions。
5. `prePlanMode` 等字段承载模式切换前的历史。

这意味着 Rust 中不能只定义：

```rust
enum PermissionMode { Default, Auto, Plan, Bypass }
```

还必须定义：

1. `PermissionModeManager`
2. `ModeTransitionEffect`
3. dangerous rule stash / restore 机制

## 11. 任务抽象与工具系统的关系

`Task.ts` 显示任务系统的最小公共基线非常薄：

1. `TaskType`
2. `TaskStatus`
3. `TaskStateBase`
4. `Task.kill()`
5. task id 生成

这说明任务系统虽然在产品层看起来很大，但它进入 Runtime Core 后，首先只是“长生命周期副作用句柄”。

对工具系统而言，后台 Shell、Agent、Workflow 更像是特殊工具结果：

1. 启动一个长期运行单元。
2. 返回 task handle。
3. 允许后续查询读取输出或停止任务。

## 12. 对 Rust 设计的抽象结论

当前可以先把这一层拆成以下组件：

1. `ToolDescriptor`
   - 名称、schema、只读性、并发安全性、危险等级、是否需要交互。
2. `ToolExecutor`
   - 真正执行工具副作用。
3. `ToolScheduler`
   - 负责批次规划和上下文突变提交。
4. `PermissionEngine`
   - 统一做规则、模式、tool-specific checks 的决策。
5. `PermissionModeManager`
   - 负责模式切换副作用。
6. `PermissionPromptAdapter`
   - 宿主层对 ask 决策的呈现和回填。
