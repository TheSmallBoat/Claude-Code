## 2. 第一判断：编程能力不是 Runtime Core，而是一组宿主适配包

从 `Tool.ts`、`tools.ts`、`types/command.ts` 看，运行时真正稳定的抽象只有：

1. tool contract
2. command contract
3. permission contract
4. task / query / transcript contract

而“编程相关能力”是通过这些通用契约挂进去的一组 host adapters：

1. 工作区文件系统
2. 搜索与索引
3. shell / 进程执行
4. git / repo 语义
5. LSP / IDE 协同
6. skills / workflows / coding commands

因此 Rust 版不应把这些能力混进 `agent-runtime-core`，而应把它们放到独立的 `code-adapter layer`。

## 3. 通用接口和领域实现是分离的

### 3.1 `buildTool()` 只是通用包装器

`Tool.ts` 的 `buildTool()` 只提供：

1. `isEnabled`
2. `isConcurrencySafe`
3. `isReadOnly`
4. `isDestructive`
5. `checkPermissions`
6. `toAutoClassifierInput`
7. `userFacingName`

这说明 tool system 本身不关心“编程”是什么；编程语义发生在具体 tool 实现里。

### 3.2 `commands` / `skills` 也是通用载体

`types/command.ts` 说明命令框架也只是统一承载：

1. `prompt`
2. `local`
3. `local-jsx`

但 skill frontmatter 可以挂上：

1. `allowedTools`
2. `context = inline | fork`
3. `agent`
4. `paths`
5. `hooks`

这正是“领域能力编排层”搭在通用 runtime 之上的证据。

## 4. 文件系统适配层的真实复杂度

### 4.1 `FileReadTool` 已经不是简单读文本

`FileReadTool.ts` 支持的输出形态包括：

1. `text`
2. `image`
3. `notebook`
4. `pdf`
5. `parts`

并且它还处理：

1. device file blocklist
2. PDF page range
3. image resize / token budget
4. notebook cell map
5. session memory / transcript 文件识别
6. line numbering

这说明“工作区读取”在 coding agent 里必须是 typed read API，而不能只是 `read_to_string()`。

### 4.2 `FileEditTool` / `FileWriteTool` / `NotebookEditTool` 共享一组强不变量

三者都体现了相同策略：

1. 路径先 `expandPath()` 归一化。
2. 先做权限判定，再做真正 I/O。
3. UNC / network path 在验证前直接短路，避免意外交网。
4. mutating 操作要求 read-before-write。
5. 文件若在读取后被用户或 linter 改过，必须拒绝并要求重读。

这些不是某一个 tool 的偶然逻辑，而是 coding host 的共享写入契约。

### 4.3 写路径还附带了很多 coding-specific side effects

以 `FileWriteTool` / `FileEditTool` 为例，写入后还会：

1. 记录 file history 备份
2. 计算 diff / git diff
3. 通知 LSP `didChange` / `didSave`
4. 通知 VSCode `file_updated`
5. 更新 `readFileState`
6. 激活 conditional skills / discover nested skill dirs

这说明对 coding agent 来说，“写文件”本质上是一次工作区状态迁移，不只是字节覆盖。

## 5. 搜索层是单独的适配面，不该并入文件读取

### 5.1 `GlobTool` 和 `GrepTool` 是两种不同语义

1. `GlobTool` 负责路径枚举。
2. `GrepTool` 负责内容匹配。

两者都在 `tools.ts` 中被独立注册，且都标记为 read-only / concurrency-safe。

### 5.2 `GrepTool` 的真实功能比“执行 rg”更高一层

它额外定义了：

1. `output_mode`
2. context lines
3. `head_limit` / `offset`
4. `type`
5. `multiline`
6. 默认 limit 与分页提示
7. VCS 目录排除

说明搜索适配层输出的是“可被模型消费的分页搜索结果”，而不是裸终端输出。

### 5.3 搜索也被权限与 token 预算包裹

1. 搜索基路径要过 read permission。
2. 结果会相对化以节省 token。
3. 大结果默认截断，要求显式分页。

这说明 search adapter 是 context engineering 的一部分。

## 6. `BashTool` 是最重的 coding host adapter

`BashTool.tsx` 的职责远超“运行命令”：

1. shell AST / operator parsing
2. sandbox 选择
3. read/search/list command 语义识别
4. permission / safety / classifier 协作
5. 大输出落盘与 `TaskOutput` 对接
6. 前台转后台 / 自动后台化
7. shell 输出图片化 / 持久化
8. 特殊 `sed` 编辑预览后直接转为安全文件写入

这里可以看出一个核心判断：在 coding agent 里，shell executor 是带语义和策略的“工作区过程适配器”，而不是 runtime core 里一个普通 subprocess helper。

## 7. `LSPTool` 说明代码智能是可选 sidecar，不是核心状态机的一部分

`LSPTool.ts` 的关键信号：

1. 只有连接了 LSP manager 才启用。
2. 调用前会等待初始化。
3. 若文件未 open，会先把文件内容送进 LSP server。
4. 默认 10MB 上限。
5. 对 definitions / references / symbols 还会过滤 gitignored locations。

这说明 code intelligence 层应被建模为可选 sidecar backend，挂在 runtime 外侧，而不是 query 核心的一部分。

## 8. repo / workspace 语义也属于领域适配，不属于核心 runtime

### 8.1 `context.ts` 把 git 快照注入系统上下文

`getSystemContext()` 会按需注入：

1. branch
2. main branch
3. `git status --short`
4. recent commits
5. git user

这显然是 coding-domain prompt context，而不是一般 agent runtime 必需的内容。

### 8.2 `utils/git.ts` 还处理 worktree / canonical root 语义

它不只是找 `.git`：

1. 识别 `.git` directory 与 file
2. 解析 worktree `gitdir` / `commondir`
3. 计算 canonical repo root
4. 防御恶意 repo 借用受信任 worktree 的路径欺骗

说明 repo adapter 必须是一个独立的安全敏感子系统。

## 9. IDE bridge 也是编程领域适配层的一部分

`services/mcp/vscodeSdkMcp.ts` 说明系统会把 VSCode 当成一个宿主协同端：

1. 写文件后发 `file_updated` notification
2. 把实验 gates 下发给 VSCode
3. 接受 VSCode 发回的 telemetry/log 通知

因此 coding adaptation 不只包含本地文件和 shell，还包括 IDE 协同桥。

## 10. 技能、命令、工作流构成了“编程工作流编排层”

### 10.1 `skills/loadSkillsDir.ts` 暗示了技能不是静态列表

它支持：

1. frontmatter 解析
2. `allowed-tools`
3. `context: fork`
4. `agent`
5. `paths`
6. `hooks`
7. shell interpolation

而且技能会：

1. 从 `.claude/skills` 动态发现
2. 随文件触摸动态激活
3. 根据路径模式成为 conditional skill

这说明编程领域适配层并不只提供 primitives，还负责把这些 primitives 组织成经验化工作流。

### 10.2 `commands.ts` 把多来源工作流统一挂到命令面

命令来源至少包括：

1. bundled skills
2. skill dir commands
3. plugin skills
4. workflow commands
5. built-in commands

同时还有：

1. remote-safe command filtering
2. bridge-safe local command filtering
3. MCP skill filtering

这再次证明：领域层负责根据宿主能力存在与否装配可用工作流。

## 11. 一个重要结论：remote-safe / bridge-safe 过滤就是边界证明

`commands.ts` 对 remote 和 bridge 场景专门过滤命令，保留那些不依赖本地：

1. filesystem
2. git
3. shell
4. IDE
5. MCP 本地宿主状态

这实际上从反面证明了：这些能力都不是 runtime core 的一部分，而是本地 coding host adapters。

## 12. 对 Rust 的第一层抽象判断

建议把这层明确拆成：

1. `CodeWorkspaceAdapter`
2. `CodeSearchAdapter`
3. `CodeExecutionAdapter`
4. `CodeIntelAdapter`
5. `CodeRepoAdapter`
6. `CodeWorkflowCatalog`

通用 runtime core 只知道“有一组 capability providers 被装配进来”，不直接知道 `rg`、`git`、`.ipynb`、LSP 或 VSCode。

## 13. 当前已知缺口

本轮已经足够写 Rust-facing 规格，但仍有几个待补点：

1. `review`、`commit`、`pr` 等高阶编程工作流还未单独审计。
2. browser / container / remote coding host 的适配边界还未单列。
3. semantic search / tool search / embedded search 的统一搜索抽象仍可再收束。
