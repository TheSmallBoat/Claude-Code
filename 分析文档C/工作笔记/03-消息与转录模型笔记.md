## 2. 第一判断：`types/message.ts` 不是完整真相

`types/message.ts` 里的类型定义比较宽松，很多字段被恢复成 `[key: string]: unknown` 或弱化结构。真正稳定的运行时语义更多分散在：

1. `utils/messages.ts` 的构造函数。
2. `utils/sessionStorage.ts` 的持久化过滤与 transcript 链逻辑。
3. `types/logs.ts` 的 `SerializedMessage` / `TranscriptMessage` 定义。
4. `utils/messages/mappers.ts` 的 SDK 映射。

因此，Rust 规格不能直接照抄 `types/message.ts`，而应以构造器和持久化层定义为准。

## 3. Message 体系的运行时角色

当前实现的 `Message` 联合体承载多种角色：

1. 用户与模型的主对话事实。
2. 工具执行与工具结果配对载体。
3. 系统生命周期事件。
4. 进度和附件这类中间态或辅助态数据。
5. compact boundary 等恢复关键点。

从内核角度，最值得保留的是以下几类：

1. `user`
2. `assistant`
3. `system`
4. `attachment`
5. `progress`
6. `tombstone`
7. `tool_use_summary`

## 4. 基础字段不变量

从 `MessageBase`、各构造器和 transcript 写入逻辑看，以下字段应视为稳定不变量：

1. `uuid`
2. `timestamp`
3. `parentUuid`
4. `logicalParentUuid`
5. `isMeta`
6. `isVirtual`
7. `origin`

其中：

1. `uuid` 是消息身份与去重基础。
2. `parentUuid` 是 transcript 链关系基础。
3. `logicalParentUuid` 用于 compact 边界后的逻辑续链。
4. `isMeta` 决定消息是否是用户/系统生成的辅助性提示。
5. `isVirtual` 用于“存在于运行态但对外部 transcript 需要降格或改写”的消息。

## 5. 消息构造器揭示的真实语义

### 5.1 `createUserMessage`

这个构造器说明 user message 并不等价于“人类键盘输入”。它也用于：

1. tool_result 回注。
2. synthetic meta continuation 提示。
3. compact summary 相关消息。
4. command breadcrumb。

这意味着 Rust 中 `UserMessage` 更准确的含义应是“user-role message”，而不是“human-authored only”。

### 5.2 `createAssistantMessage` / `createAssistantAPIErrorMessage`

assistant message 同时承载：

1. 正常模型输出。
2. synthetic API error。
3. 中断或恢复相关的占位输出。

因此 Rust 中不能把 API error 放到完全独立的事件流里，否则会破坏当前“错误也作为消息参与恢复与 transcript 逻辑”的一致性。

### 5.3 `createSystemMessage` 与 system subtype

system message 不是一个单一类型，而是一族生命周期事件。当前关键 subtype 包括：

1. `informational`
2. `compact_boundary`
3. `microcompact_boundary`
4. `stop_hook_summary`
5. `turn_duration`
6. `api_error`
7. `local_command`

其中对基线 Runtime Core 最关键的是 `compact_boundary` 和 `stop_hook_summary`。

## 6. compact boundary 是恢复语义锚点

`createCompactBoundaryMessage()` 说明 compact boundary 不是普通提示文本，而是带 metadata 的关键锚点。它至少包含：

1. `trigger`
2. `preTokens`
3. `userContext`
4. `messagesSummarized`
5. 可选 `logicalParentUuid`

而 `utils/messages/mappers.ts` 进一步表明，在 SDK 侧最稳定的 compact metadata 结构是：

1. `trigger`
2. `pre_tokens`
3. `preserved_segment.{head_uuid, anchor_uuid, tail_uuid}`

这说明 Rust 中 compact boundary 必须是一种强类型消息，而不是自由结构体挂载到 system message 上。

## 7. `TranscriptMessage` 暴露出持久化层真实结构

`types/logs.ts` 中的 `TranscriptMessage` 才是真正进入 JSONL 的核心类型。它在普通 Message 基础上新增：

1. `cwd`
2. `userType`
3. `sessionId`
4. `version`
5. `parentUuid`
6. `logicalParentUuid`
7. `isSidechain`
8. 可选 `agentId`
9. 可选 `teamName`
10. 可选 `promptId`

因此 transcript 不是“直接把 Message JSON.stringify”，而是附加了 session/logging 元数据后的序列化消息。

## 8. transcript 并不持久化所有消息

`isTranscriptMessage()` 和 `isLoggableMessage()` 一起暴露出两个层次：

1. transcript entry 类型只承认 `user/assistant/attachment/system`。
2. 运行时消息中 `progress` 明确不进入 transcript 链。
3. 对外部用户，大多数 attachment 也会被过滤掉。

这说明需要区分：

1. `RuntimeMessage`
2. `TranscriptMessage`

不能把它们混为同一个集合。

## 9. `cleanMessagesForLogging` 的真正作用

`cleanMessagesForLogging()` 不是简单过滤，而是一个 transcript 正规化步骤：

1. 过滤掉非 loggable 消息。
2. 对 external transcript 做 REPL wrapper 去除。
3. 把 `isVirtual` 消息必要时降格成真实 transcript 消息。

这说明 transcript 持久化前存在一次专用变换：

1. 运行态消息。
2. transcript 规整消息。
3. 再进入 JSONL 序列化。

Rust 版本也应保留这层，而不是直接写内存消息。

## 10. `recordTranscript` 的几个关键不变量

`recordTranscript()` 暴露出一组非常重要的持久化语义：

1. 先用 `cleanMessagesForLogging()` 得到可落盘消息。
2. 基于 `messageSet` 去重，只写此前未持久化的消息。
3. 只把已经存在且位于“前缀”的消息作为新的 `startingParentUuid`。
4. 新消息通过 `insertMessageChain()` 写成 parentUuid 链。
5. 返回最后一个真正写入的 chain participant uuid。

这里最关键的是“前缀”逻辑：

1. 已记录消息只有在形成前缀时，才能参与新的 parent 链续接。
2. compact 后保留消息出现在新 boundary 之后，因此不会错误地变成新链父节点。

## 11. sidechain transcript 是一等公民

`recordSidechainTranscript()` 表明 subagent/background task 并不是写同一份 transcript，而是写独立 sidechain transcript。

这意味着 Rust 版至少要区分：

1. main transcript
2. sidechain transcript

同时共享：

1. 同一消息序列化规则
2. 同一 parentUuid 链规则
3. 但不同文件路径和 agentId 命名空间

## 12. compact 恢复依赖 `applyPreservedSegmentRelinks`

这个函数揭示了 compact 后 resume 的关键机制：

1. 找到绝对最后一个 compact boundary。
2. 若其 `preservedSegment` 存在，则验证 tail -> head 链是否完整。
3. 若链不完整，则放弃 relink，并保留完整 pre-compact 历史。
4. 若链完整，则把 preserved segment 的 head 重新挂接到 anchor。
5. 并裁掉 boundary 之前的大部分历史，只保留 preserved 部分。

这说明 compact 不是“简单 summary 替换”，而是一个受 preserved segment 元数据驱动的链重写过程。

## 13. `ensureToolResultPairing` 是消息完整性防线

这个函数极其关键，因为当前系统大量依赖 tool_use / tool_result 配对合法性。它做了以下防御：

1. 修复 assistant 中重复 `tool_use id`。
2. 剥离没有前置 assistant 的 orphaned tool_result。
3. 为缺失结果的 tool_use 识别问题并修补或至少清理坏块。
4. 对 server-side tool use 也做同类一致性检查。

这意味着 Rust 中需要一个单独的 `MessageIntegrityNormalizer`，不能把配对检查散在 API client 中。
