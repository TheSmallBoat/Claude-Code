## 2. 第一判断：这不是单一 compact 功能，而是分层的上下文治理梯子

`query.ts` 里真实存在的顺序不是“超长了就摘要”，而是一个从便宜到昂贵、从局部到全局的治理链：

1. `applyToolResultBudget()` 先对超大的 tool result 做预算替换。
2. 可选 `snipCompactIfNeeded()` 先做历史裁剪。
3. `microcompactMessages()` 先处理旧 tool result。
4. `contextCollapse.applyCollapsesIfNeeded()` 在更高层做折叠投影。
5. `autoCompactIfNeeded()` 再决定是否进入真正 compaction。
6. 真正 API 采样失败后，才进入 withheld 413 / media error 的恢复链。

这说明 Rust 版不应把 compact 写成单个函数，而应当建成 `LongContextGovernor` 一类的阶段式管线。

## 3. microcompact 的真实语义

### 3.1 它优先处理的是“旧工具结果”，不是整段历史

`microCompact.ts` 只关心可压缩工具集合：

1. FileRead
2. Shell 类工具
3. Grep / Glob
4. WebSearch / WebFetch
5. FileEdit / FileWrite

它不是摘要消息，而是优先降低这些工具结果带来的上下文膨胀。

### 3.2 存在两条路径

第一条是 time-based microcompact：

1. 若距离上次 assistant 主线程消息超过阈值，判定服务端 cache 已冷。
2. 直接把旧 `tool_result` 内容清空为占位文本。
3. 保留最近 `N` 个可压缩工具结果。
4. 清理 cached microcompact 的模块级状态，避免下一轮删错。

第二条是 cached microcompact：

1. 只允许主线程使用。
2. 只在支持 cache editing 的模型上启用。
3. 不直接改本地 message content。
4. 而是登记要删掉的 `tool_use_id`，交给 API 层注入 `cache_edits`。

### 3.3 这层的本质是“保留工作轨迹、去掉旧负载”

也就是说 microcompact 不是 lossless，但它尽量不破坏近期会话结构，只优先把最昂贵、最容易陈旧的工具结果内容抹平。

## 4. autocompact 的真实语义

### 4.1 阈值不是直接拿模型上下文窗

`autoCompact.ts` 先算：

1. `effectiveContextWindow = contextWindow - reservedSummaryOutput`
2. `autoCompactThreshold = effectiveContextWindow - 13000`

还存在：

1. warning threshold
2. error threshold
3. blocking limit

说明系统把“还能继续跑”“该警告”“该主动 compact”“已经接近阻塞”分成了不同层级。

### 4.2 它有很多抑制条件

以下场景会直接跳过 proactive autocompact：

1. `querySource === 'session_memory'`
2. `querySource === 'compact'`
3. context collapse 已启用
4. reactive-only mode 启用
5. auto-compact 被全局或用户配置关闭
6. 特定 querySource 可能会污染主线程共享状态

这说明 autocompact 只是治理梯子的一层，而不是总控开关。

### 4.3 它有失败断路器

`MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3`。

也就是：

1. 不是每轮都无脑重试 compact。
2. 若上下文已经不可恢复，会停止继续浪费调用。

这在 Rust 版里必须保留为显式 tracking state，而不是日志层面的优化。

## 5. proactive compaction 的真实执行路径

`autoCompactIfNeeded()` 的顺序是：

1. 先判断要不要 compact。
2. 若要 compact，先尝试 `trySessionMemoryCompaction()`。
3. 若 session memory compaction 失败，再走 `compactConversation()`。
4. 成功后统一跑 `runPostCompactCleanup()`。
5. 失败则累加 consecutive failures。

这里很关键：session memory compaction 并不是 session memory 的附属玩法，而是 autocompact 的优先路径。

## 6. full compact 的真实语义

### 6.1 它不是“清空历史后塞一个 summary”这么简单

`compactConversation()` 的关键步骤是：

1. 先跑 pre-compact hooks。
2. 构造 compact prompt。
3. 优先尝试 prompt cache sharing 的 forked-agent 路径。
4. 失败后再退回常规 streaming summarize 路径。
5. 如果 compact 请求自己又撞上 prompt-too-long，则按 API round group 截断最老历史重试。

### 6.2 summarize 前会先做输入清理

为了让 compaction API 自己不被撑爆，compact 会先：

1. strip images / documents
2. strip 一些会在 compact 后重新注入的 attachments

说明 compact 过程本身也有自己的输入治理。

### 6.3 compact 后不是简单保留 summary，而是重建工作集

`compact.ts` 里 compact 成功后会重新生成：

1. post-compact file attachments
2. async agent attachments
3. plan file attachment
4. plan mode attachment
5. invoked skills attachment
6. deferred tools delta
7. agent listing delta
8. MCP instructions delta
9. session start hook messages

也就是说，compact 之后的新上下文是“summary + working set reconstruction”，不是纯摘要上下文。

## 7. compact 结果对象的稳定结构

`CompactionResult` 的稳定成员包括：

1. `boundaryMarker`
2. `summaryMessages`
3. `attachments`
4. `hookResults`
5. 可选 `messagesToKeep`
6. pre/post token 统计

而 `buildPostCompactMessages()` 明确规定了顺序：

1. boundary
2. summaries
3. preserved tail
4. attachments
5. hook results

这个顺序本身就是 transcript 恢复和 resume 的契约。

## 8. preserved segment 的语义非常关键

`annotateBoundaryWithPreservedSegment()` 与 `sessionStorage.ts` 的 relink 逻辑说明：

1. compact 并不会重写所有保留消息的 parent uuid。
2. 它只在 boundary metadata 里记录 `headUuid / anchorUuid / tailUuid`。
3. transcript loader 再把 preserved tail 重新 splice 回链表。

因此 compact boundary 不是一条展示消息，而是 transcript 结构重建锚点。

## 9. session memory 的真实定位

### 9.1 它是后台维护的会话笔记文件

`initSessionMemory()` 在 `setup.ts` 注册 post-sampling hook，但前提是：

1. 不是 remote mode
2. auto-compact enabled

说明 session memory 不是独立产品功能，而是 compact 体系的一部分。

### 9.2 它不是每轮都更新，而是阈值驱动

默认阈值来自 `sessionMemoryUtils.ts`：

1. 初始化阈值：10000 tokens
2. 两次更新最小增量：5000 tokens
3. 两次更新之间至少 3 次 tool calls

`shouldExtractMemory()` 还要求：

1. token threshold 必须满足
2. tool call threshold 满足，或者最近 assistant turn 没有 tool call

也就是说 session memory 倾向在自然断点更新，而不是每轮抖动。

### 9.3 它的执行是串行、隔离、限权的

`extractSessionMemory` 被 `sequential()` 包住，并且：

1. 只允许主线程触发
2. 使用 `createSubagentContext()` 隔离读写状态
3. setup 阶段先建立 memory file，再用 FileReadTool 读当前内容
4. forked agent 只允许对这个精确文件执行 `Edit`

因此 session memory 本质上是“后台维护单文件记忆账本”的子代理流程。

## 10. session memory 文件本身也受预算治理

`services/SessionMemory/prompts.ts` 规定：

1. 文件模板有固定 section 结构
2. 总 token 预算大约 12000
3. 单 section 大约 2000 token 上限
4. 更新 prompt 强制保持标题和斜体描述不变

这意味着 session memory 自己也是一个被预算控制的外部记忆体，而不是无限增长的日志。

## 11. session memory compaction 的真实语义

### 11.1 它不是读取 memory 后简单替换历史

`trySessionMemoryCompaction()` 的真实步骤是：

1. 等待进行中的 extraction 完成，最多 15 秒；超过 1 分钟视为 stale。
2. 加载 session memory 文件。
3. 若文件不存在或仍只是模板，则放弃。
4. 若 `lastSummarizedMessageId` 存在，按它找边界；否则走 resumed session 分支。
5. 用 `calculateMessagesToKeepIndex()` 反向扩展 preserved tail。
6. 运行 session start hooks。
7. 生成 boundary、summary、plan attachment。

### 11.2 preserved tail 的选择规则比看上去复杂得多

`calculateMessagesToKeepIndex()` 同时满足：

1. 至少保留一定 tokens
2. 至少保留一定数量的 text-block messages
3. 不超过最大 tokens
4. 不跨过最近 compact boundary
5. 不拆开 tool_use / tool_result 对
6. 不拆开同一 assistant `message.id` 的 thinking / tool_use 分片

这层说明 preserved tail 不是“保留最后 N 条消息”，而是“保留一段满足 API 不变量的最小工作集”。

## 12. relevant memories 与 session memory 不是一回事

### 12.1 relevant memories

这是 `utils/attachments.ts + memdir/findRelevantMemories.ts` 那条链：

1. 每个用户 turn 启动一次异步 prefetch。
2. 根据最后一个真实 user message 选相关 memory files。
3. 选择最多 5 个文件。
4. 每个文件最多 200 行 / 4096 bytes。
5. 整个 session 累计 surfacing 超过 60KB 后停掉。
6. consume 点在工具执行之后，若 prefetch 尚未 settle 就跳过，不阻塞本轮。

它是“当前回合的相关记忆注入”。

### 12.2 nested memory

`memoryFilesToAttachments()` 说明另一条链是固定 memory / CLAUDE.md 级别注入：

1. 用 `loadedNestedMemoryPaths` 做非驱逐去重。
2. 再用 `readFileState` 做跨轮 dedupe。
3. 如果注入内容和磁盘原文不完全一致，会记录 `isPartialView`。

这更像持久规则层，而不是 relevant memories 那种回合级 recall。

### 12.3 结论

可以把“记忆”拆成三层：

1. nested memory：稳定规则与项目记忆
2. relevant memories：回合级 recall 附件
3. session memory：后台维护、用于 compaction 的外部会话摘要

## 13. post-compact cleanup 的意义

`runPostCompactCleanup()` 说明 compact 不只是替换消息数组，还要清理共享状态：

1. reset microcompact state
2. main-thread 场景下 reset context collapse
3. 清 `getUserContext` cache
4. reset memory file cache
5. clear system prompt sections
6. clear classifier approvals / speculative checks
7. clear session message cache

同时它明确避免清理 invoked skills，因为那是后续 compact reconstruction 还要用的状态。

## 14. reactive compact 在当前恢复树中的状态

这里有一个必须诚实记录的点：

1. `query.ts` 与 `/compact` 命令都清楚接入了 reactive compact。
2. `services/api/errors.ts`、`query.ts` 注释也明确描述了它应处理 prompt-too-long 和 media-size recovery。
3. 但当前恢复树中的 `services/compact/reactiveCompact.ts` 只有一个占位 stub。

因此目前只能较可靠地确认它的接口语义：

1. 真实 413 要先给 context collapse drain 一次机会。
2. drain 后若仍失败，再尝试 reactive compact。
3. media-size rejection 走 strip-retry 路径，不走 collapse drain。
4. 同一轮只允许尝试一次，避免死循环。

但其内部摘要/保留策略在当前代码树中无法完整复原，后续规格必须把这部分标成“根据调用点反推的接口契约”。

## 15. 当前对 Rust 设计的抽象结论

这一层建议至少拆出：

1. `MicrocompactEngine`
2. `AutoCompactController`
3. `CompactionExecutor`
4. `SessionMemoryManager`
5. `RelevantMemoryPrefetcher`
6. `PostCompactCleaner`
7. `PreservedSegmentRelinker`

## 16. 当前已知缺口

虽然材料已经足够写出 Rust-facing 规格，但仍有两个缺口要显式保留：

1. reactive compact 的内部实现缺失，只能先固化接口语义。
2. context collapse 的完整提交/恢复机制尚未单独成文，本笔记只记录了和 compact 的接缝。
