# 上下文装配与缓存前缀笔记

## 1. 证据源

本笔记主要基于以下代码：

1. `TS-Source/context.ts`
2. `TS-Source/constants/prompts.ts`
3. `TS-Source/utils/queryContext.ts`
4. `TS-Source/utils/forkedAgent.ts`
5. `TS-Source/QueryEngine.ts`

## 2. 第一判断：上下文不是单个 prompt，而是三层前缀

`utils/queryContext.ts` 直接把 cache-key 前缀定义为三部分：

1. `systemPrompt`
2. `userContext`
3. `systemContext`

这意味着系统真正缓存和复用的不是“最终字符串 prompt”，而是一个多层上下文束。

## 3. `fetchSystemPromptParts()` 的语义

这个函数说明基础上下文装配顺序是：

1. 生成 `defaultSystemPrompt`。
2. 读取 `userContext`。
3. 读取 `systemContext`。

但当 `customSystemPrompt` 存在时，行为会改变：

1. 跳过默认 `getSystemPrompt()`。
2. 跳过 `getSystemContext()`。
3. 只保留 `userContext`。

这说明 custom prompt 的语义不是“在默认 prompt 外再叠一层”，而是“替换默认 system prompt 主体”。

## 4. `context.ts` 揭示的 context 来源

### 4.1 `getUserContext()`

其主要来源包括：

1. `CLAUDE.md` / memory files 汇总内容。
2. `currentDate`。

而且它有两个重要特性：

1. 使用 memoize 做会话级缓存。
2. CLAUDE.md 读取会受 `--bare`、`CLAUDE_CODE_DISABLE_CLAUDE_MDS`、`--add-dir` 等条件影响。

### 4.2 `getSystemContext()`

其主要来源包括：

1. git status 快照。
2. cache breaker 注入。

它同样是 memoized，会在 system prompt injection 变化时清 cache。

判断：这两者都不是“每轮实时重建的一次性字符串”，而是带生命周期缓存的 context providers。

## 5. `getSystemPrompt()` 的真正结构

`constants/prompts.ts` 表明 system prompt 本身也不是单段文本，而是 section 数组：

1. 静态 section。
2. 动态 section。
3. 以 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 分隔两段缓存策略。

静态部分包含：

1. intro
2. system
3. doing tasks
4. actions
5. using tools
6. tone and style
7. output efficiency

动态部分通过 section registry 生成，当前包括：

1. `session_guidance`
2. `memory`
3. `ant_model_override`
4. `env_info_simple`
5. `language`
6. `output_style`
7. `mcp_instructions`
8. `scratchpad`
9. `frc`
10. `summarize_tool_results`
11. `token_budget`
12. `brief`

## 6. 动态边界的真正意义

`SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 不是注释性标记，而是 cache scope 分割点：

1. marker 之前的内容可以走更稳定的全局缓存范围。
2. marker 之后的内容视为用户/会话相关动态内容。

这说明 Rust 版不能只生成单块 system prompt string，而应保留 section 列表和边界信息。

## 7. `resolveSystemPromptSections()` 暗示 section registry 是长期正确方向

虽然本轮未深入到 section registry 实现细节，但从 `getSystemPrompt()` 的使用方式可以确认：

1. section 是惰性计算的。
2. 部分 section 可缓存。
3. 部分 section 是 `DANGEROUS_uncachedSystemPromptSection`，明确告诉系统“不要把这块当稳定缓存项”。

这说明 Rust 的 `ContextBuilder` 最好不是直接拼字符串，而是先生成 `PromptSection` 列表。

## 8. QueryEngine 的最终装配顺序

`QueryEngine.ts` 中真正发给 query 的 system/context 组合逻辑是：

1. 先取 `defaultSystemPrompt`、`baseUserContext`、`systemContext`。
2. 再把 coordinator user context 合并进 `userContext`。
3. 若 `customSystemPrompt` 且 `hasAutoMemPathOverride()`，再补一层 `memoryMechanicsPrompt`。
4. 最终 `systemPrompt = custom/default + memoryMechanicsPrompt + appendSystemPrompt`。

这里有两个重要结论：

1. `userContext` 允许在 fetch 完成后再被宿主模式扩展。
2. `appendSystemPrompt` 不是替换，而是尾部附加策略位。

## 9. cache-safe params 的真实边界

`utils/forkedAgent.ts` 中的 `CacheSafeParams` 明确包含：

1. `systemPrompt`
2. `userContext`
3. `systemContext`
4. `toolUseContext`
5. `forkContextMessages`

注释说明其目标是保证 fork 与 parent API 请求在 cache-critical 参数上保持一致，以共享 prompt cache。

这意味着 cache-safe prefix 并不只是文本层面，还包括：

1. tools
2. model
3. thinking config
4. messages prefix

其中部分信息通过 `toolUseContext` 间接携带。

## 10. `saveCacheSafeParams()` 的作用

`query/stopHooks.ts` 在 turn 结束后保存 cache-safe params，供：

1. prompt suggestion
2. post-turn summary
3. `/btw`
4. side_question fallback

等 fork 行为复用。

这说明 cache-safe params 是一个会话级最近快照，而不是临时局部变量。

## 11. `buildSideQuestionFallbackParams()` 暗示的降级策略

这个函数表明，当正常路径拿不到上一次 stop-hook 快照时，系统会尝试重新构建一个“尽可能接近主线程”的 cache-safe prefix，用于 side question。

它的价值不在于绝对一致，而在于：

1. 没有快照时仍能工作。
2. 尽量命中与主线程接近的 prompt cache。

Rust 中也应有类似 fallback builder。
