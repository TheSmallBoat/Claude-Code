## 2. 第一判断：文本资产要按“注入平面”分类，而不是按文件扩展名分类

单看扩展名会把关键机制混在一起：

1. `SKILL.md` 既可能是用户可调用 slash command 的正文，也可能只是 bundled 文档的一部分。
2. `.md` 既可能进入主会话 system prompt，也可能进入 meta user message，也可能根本不会自动注入。
3. `.txt` 既可能是远端计划 prompt，也可能是权限 classifier 的内部系统提示模板。

因此，真正稳定的分类标准不是 `.md` / `.txt`，而是：

1. 谁负责发现和加载这段文本。
2. 它被注入到哪一层上下文。
3. 注入后是否可缓存、可恢复、可覆盖。
4. 文本来源是否可信，是否允许做 shell interpolation 或权限提升。

本仓库至少存在六种注入平面：

1. 主会话 `system prompt`。
2. 主会话 `user context`。
3. slash command / skill 形成的 `meta user message`。
4. agent markdown 形成的 `agent-local system prompt`。
5. classifier / dream / session-memory / magic-docs 一类内部维护型 prompt。
6. output style / attachment / task metadata 一类辅助保持行为一致性的上下文附着物。

## 3. 主会话 system prompt 的主干是代码拼装，而不是外部 markdown 直读

### 3.1 默认 system prompt 主体来自 `constants/prompts.ts`

`getSystemPrompt()` 不是简单返回一个长字符串，而是把 system prompt 切成静态段和动态段：

1. intro/system/doing-tasks/actions 等静态规范。
2. `session_guidance`、`memory`、`language`、`output_style`、`mcp_instructions` 等动态 section。
3. `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 用来把全局可缓存前缀和动态后缀分开。

这说明主会话 system prompt 已经是一套“结构化 prompt section 协议”，而不是一段不可分解的文案。

### 3.2 `QueryEngine` 只在最后阶段决定替换还是追加

`fetchSystemPromptParts()` 会并行拉取：

1. `defaultSystemPrompt`
2. `userContext`
3. `systemContext`

随后 `QueryEngine` 再决定最终 system prompt：

1. 若用户传入 `customSystemPrompt`，则整个默认 prompt build 被替换。
2. 若启用了特定 memory override，还会插入 `memoryMechanicsPrompt`。
3. `appendSystemPrompt` 总是在最后追加。

因此 CLI flag / file 不是“额外提示”，而是可直接改变主系统提示装配语义的强覆盖入口。

### 3.3 真正送往模型前还会被切 block 并附 cache scope

`services/api/claude.ts` 的 `buildSystemPromptBlocks()` 会把 system prompt 数组转为 API `text blocks`，并在合适时打上 `cache_control`。

这意味着：

1. system prompt 在 API 层仍是结构化数组，而不是最终拼平的单字符串。
2. `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 不是文档注释，而是缓存协议的一部分。

## 4. `CLAUDE.md` / `.claude/rules/*.md` 属于 user context，而不是默认 system prompt

### 4.1 `getUserContext()` 负责把 `CLAUDE.md` 变成 `claudeMd`

`context.ts` 中 `getUserContext()` 会在非 bare 且未显式关闭时读取 `CLAUDE.md` 相关内容，并把结果放入 `claudeMd` 字段。

这类文本不是 system prompt hardcode，而是“用户给代理的附加指令上下文”。

### 4.2 `utils/claudemd.ts` 有一套独立的发现顺序和优先级

`getMemoryFiles()` / `getClaudeMds()` 体现了另一条完全不同于 `loadMarkdownFilesForSubdir()` 的发现机制：

1. managed memory，例如 `/etc/claude-code/CLAUDE.md`。
2. user memory，例如 `~/.claude/CLAUDE.md`。
3. project memory，例如项目根上的 `CLAUDE.md`、`.claude/CLAUDE.md`、`.claude/rules/*.md`。
4. local memory，例如 `CLAUDE.local.md` 一类本地私有文件。
5. 附加目录与 nested directory 也可继续叠加。

这条链条还支持：

1. frontmatter `paths` 规则。
2. `@include` 指令递归展开。
3. 文本类型白名单和循环引用保护。
4. 对超长 `MEMORY.md` 入口文件做截断。

### 4.3 classifier 还会复用同一份 `claudeMd`

`getUserContext()` 读取出来的 `claudeMd` 会缓存到 bootstrap state，`yoloClassifier.ts` 再把它包成独立 user message 注入权限分类器请求。

这说明 `CLAUDE.md` 不是只对主对话有效，而是会跨到内部 side-query / classifier 流程。

## 5. 文件型 `.claude/*` 资产有统一发现器，但 `CLAUDE.md` 不走这条链

### 5.1 `loadMarkdownFilesForSubdir()` 是 `.claude/commands|agents|output-styles` 的统一入口

`utils/markdownConfigLoader.ts` 会：

1. 从 managed、user、project 三类目录加载 markdown。
2. project 路径从当前目录向上走到 git root，避免仓库外父目录泄漏。
3. 在 git worktree 场景下为 `.claude/<subdir>` 提供 canonical root fallback。
4. 对 inode 去重，防止符号链接路径重复装载。

这条发现器主要服务于：

1. commands
2. agents
3. output styles

### 5.2 `CLAUDE.md` 采用另一套专用发现器

`CLAUDE.md` 不走 `loadMarkdownFilesForSubdir()`，而是完全由 `utils/claudemd.ts` 自己负责路径遍历、frontmatter path 过滤、include 扩展和文本截断。

所以 Rust 版不能把所有 markdown 资产都塞到一个 loader 里，否则优先级、边界和过滤行为会被混淆。

## 6. skills / slash commands 的 markdown 会被包装成 `meta user message`

### 6.1 本地 skills 的正文来自 `skills/loadSkillsDir.ts`

这里把 `.claude/skills/<name>/SKILL.md` 或兼容目录中的 markdown 解析成 `PromptCommand`，并允许 frontmatter 提供：

1. `allowed-tools`
2. `when_to_use`
3. `context = inline | fork`
4. `agent`
5. `paths`
6. `hooks`
7. `model` / `effort`
8. `shell`

正文进入 `getPromptForCommand()` 前还会经历：

1. 参数替换。
2. `${CLAUDE_SKILL_DIR}` 替换。
3. `${CLAUDE_SESSION_ID}` 替换。
4. 对本地可信 skill 执行 `!\`...\`` 形式的 shell interpolation。

### 6.2 skill 的真正注入位置不是 system prompt，而是 meta user message

`processSlashCommand.tsx` 在处理 prompt 型命令时，会：

1. 调用 `command.getPromptForCommand()` 生成文本 blocks。
2. 记录 skill invocation 供 compaction 恢复。
3. 包装一条 metadata user message。
4. 把 skill 正文作为 `isMeta: true` 的 user message 注入。
5. 解析 skill 正文中的附件引用。
6. 再附上 `command_permissions` attachment。

所以 skill / slash command 的本质是“把一段 workflow 文本注入对话 transcript”，而不是直接改主 system prompt。

### 6.3 MCP skills 明确是远端不可信文本

`skills/loadSkillsDir.ts` 对 `loadedFrom === 'mcp'` 的 skill 禁止执行 shell interpolation。

这说明文本资产的 trust boundary 已经进入 prompt 装配协议本身，而不是事后再统一做安全扫描。

## 7. remote canonical skills 和 plugin markdown 也是同一类资产，但加载边界不同

### 7.1 `SkillTool.executeRemoteSkill()` 直接注入远端 `SKILL.md`

`tools/SkillTool/SkillTool.ts` 会把 remote canonical skill 的 `SKILL.md` 正文拉回本地，然后：

1. strip frontmatter。
2. 注入 `Base directory for this skill`。
3. 替换 `${CLAUDE_SKILL_DIR}` / `${CLAUDE_SESSION_ID}`。
4. 直接作为 meta user message 注入当前会话。

这条链和本地 skill 的差异主要在发现路径，而不是最终注入平面。

### 7.2 plugin markdown commands 是另一套 discovery，但最终仍回到 `PromptCommand`

`utils/plugins/loadPluginCommands.ts` 会从插件的 `commands/` 或额外路径读取 markdown，并转换成 `PromptCommand`。

其共同点：

1. 正文仍走 `getPromptForCommand()`。
2. 仍支持参数替换和变量替换。
3. 仍可生成 allowed tools、fork context、model、effort 等元数据。
4. 最终仍通过 `processSlashCommand.tsx` 变成 meta user message。

因此，plugin command 不是新的 prompt plane，只是新的 discovery source。

## 8. agent markdown 的正文会变成 agent-local system prompt

### 8.1 本地 agent markdown 的 body 就是 `getSystemPrompt()` 返回体

`tools/AgentTool/loadAgentsDir.ts` 的 `parseAgentFromMarkdown()` 显示：

1. frontmatter `name` 和 `description` 决定 agent identity。
2. markdown `content.trim()` 本身就是 agent system prompt body。
3. `memory` 启用时会在尾部追加 agent memory prompt。

frontmatter 还能附着：

1. `model`
2. `background`
3. `memory`
4. `isolation`
5. `effort`
6. `permissionMode`
7. `maxTurns`
8. `tools` / `disallowedTools`
9. `skills`
10. `initialPrompt`
11. `mcpServers`
12. `hooks`

### 8.2 plugin agent 明确是阉割版 agent prompt 资产

`utils/plugins/loadPluginAgents.ts` 允许插件提供 agent markdown，但会显式忽略：

1. `permissionMode`
2. `hooks`
3. `mcpServers`

也就是说 plugin agent 仍然是 system prompt 资产，但其可携带的控制面能力被故意收缩，以维持插件信任边界。

## 9. build-time bundled 文本资产同样是正式 prompt 资产

### 9.1 `claudeApiContent.ts` / `verifyContent.ts` 把 `.md` 在构建期内联

这类文件说明 prompt 资产不一定来自运行时文件系统：

1. `skills/bundled/claudeApiContent.ts` 直接 import 多份 `.md` 字符串。
2. `skills/bundled/verifyContent.ts` 也把 `SKILL.md` 与示例文档内嵌。
3. `claudeApi.ts` / `verify.ts` 再按命令调用时动态拼装成实际 prompt。

因此系统至少有两种文本资产供应链：

1. 运行时发现本地文件。
2. 构建期内嵌字符串。

### 9.2 `ultraplan` 说明 `.txt` 资产也可能是工作流主提示词

`commands/ultraplan.tsx` 通过 `require('../utils/ultraplan/prompt.txt')` 读入默认 instructions，并允许 ant-only 环境变量 `ULTRAPLAN_PROMPT_FILE` 覆盖。

这说明：

1. `.txt` 也会成为 workflow prompt source of truth。
2. 环境变量级覆盖也是正式装配路径之一。

### 9.3 classifier `.txt` 模板是内部 side-query 的 system prompt

`utils/permissions/yoloClassifier.ts` 会装载：

1. `auto_mode_system_prompt.txt`
2. `permissions_external.txt`
3. `permissions_anthropic.txt`

然后根据用户类型与 settings 组装 classifier system prompt，并把 transcript/action 作为 user prompt 发给 side-query。

在当前 restored tree 中，这些 `.txt` 只剩占位文本，这意味着接口路径已可证实，但正文内容本身缺失。

## 10. 内部维护型 prompt 不是主对话提示，却同样改变系统行为

### 10.1 session memory 有独立模板和更新 prompt

`services/SessionMemory/prompts.ts` 说明 session memory 维护至少有两份文本协议：

1. `template.md` 决定 notes 文件结构。
2. `prompt.md` 决定更新 notes 时的维护型 instructions。

两者都允许用户从 `~/.claude/session-memory/config/` 覆盖。

### 10.2 magic docs 也有同样的可覆盖维护 prompt

`services/MagicDocs/prompts.ts` 同样允许从 `~/.claude/magic-docs/prompt.md` 加载自定义提示词，用于指导文档更新代理。

### 10.3 auto-dream consolidation prompt 属于后台维护子代理

`services/autoDream/consolidationPrompt.ts` 定义了 orient / gather / consolidate / prune 等阶段性文本。

这类 prompt：

1. 不直接暴露给主对话用户。
2. 也不属于默认 system prompt。
3. 但会驱动后台 forked agent 的行为和 memory consolidation 结果。

## 11. output style 不是纯 UI，它会被注入主 system prompt

### 11.1 output style markdown 也走文件发现器

`outputStyles/loadOutputStylesDir.ts` 会从：

1. `~/.claude/output-styles/*.md`
2. 项目目录向上遍历到 git root 的 `.claude/output-styles/*.md`

读取用户和项目 output styles。

`utils/plugins/loadPluginOutputStyles.ts` 还允许插件提供 output style markdown。

### 11.2 `constants/outputStyles.ts` 把正文作为 prompt section

每个 output style 都含有：

1. `name`
2. `description`
3. `prompt`

`constants/prompts.ts` 的 `getOutputStyleSection()` 会把 `outputStyleConfig.prompt` 插入主 system prompt 的 `# Output Style` section。

这意味着 output style 不只是展示偏好，而是会实际改变模型响应策略。

### 11.3 transcript 里还会额外附上 output style attachment

`utils/attachments.ts` 会在非 default style 时附加 `output_style` attachment。

这说明 output style 还有第二重职责：在 transcript / compact 过程中维持行为语义不丢失。

## 12. 并不是所有 co-located markdown 都会自动进入上下文

当前代码里有几条显式的“不要误注入”规则：

1. 没有 agent frontmatter 的 markdown 在 `parseAgentFromMarkdown()` 中会被跳过。
2. plugin/skill 目录中如果存在 `SKILL.md`，则该目录其他 markdown 不自动充当 skill 正文。
3. slash command 的附件解析会 `skipSkillDiscovery`，防止把 skill 正文再次当成用户意图。
4. MCP skill 禁止 shell interpolation，避免把远端文档正文当成本地命令执行说明。

因此文本资产不是“读到 markdown 就注入”，而是有明确 loader、frontmatter、source 和 trust gate。

## 13. 当前可稳定抽象出的文本资产类别

综合以上证据，当前仓库至少存在以下七类正式文本资产：

1. 主会话 system prompt sections。
2. `CLAUDE.md` / rules / memory 指令块。
3. slash command / skill markdown 正文。
4. agent markdown 正文。
5. output style markdown 正文。
6. classifier / remote workflow / internal maintenance `.txt` / `.md` 模板。
7. build-time bundled 文档字符串。

真正应该被 Rust 版继承的，不是这些文案本身，而是：

1. discovery path
2. precedence
3. injection plane
4. cache boundary
5. trust policy
6. override mechanics

## 14. 对 Rust 版的初步判断

Rust 版不应只做一个 `PromptBuilder`，而应至少拆成三层：

1. `PromptAssetDiscovery`：发现用户文件、插件文件、bundled 资源、CLI 覆盖文本。
2. `PromptAssetCatalog`：把这些文本标准化为带 source / plane / trust / priority 的资产对象。
3. `PromptAssembler`：按 query type、agent type、workflow type 把资产装配成最终上下文。

否则后续会把：

1. `CLAUDE.md` 和 system prompt 混在一起。
2. skill markdown 和 agent system prompt 混在一起。
3. internal maintenance prompt 和用户可见主对话 prompt 混在一起。
4. trusted local assets 和 remote untrusted assets 混在一起。

这会直接破坏缓存、安全和可恢复性边界。
