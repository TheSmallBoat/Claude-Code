## 2. 第一判断：Task Plane 和 Subagent Plane 不是一回事

从代码看，系统其实分了两层：

1. `Task` / `TaskState` 是控制平面的可见记录。
2. `runAgent()` / `runForkedAgent()` 才是真正执行 query loop 的运行时。

也就是说，task 不是执行器本身，而是：

1. 一个可观察状态壳。
2. 一个通知与恢复入口。
3. 一个 UI / SDK / message-queue 能消费的控制面对象。

Rust 版不应把 task 和 agent runtime 混成一个对象。

## 3. `Task.ts` 暗示的极简控制面抽象

`Task.ts` 非常克制：

1. `TaskType`
2. `TaskStatus`
3. `TaskStateBase`
4. `Task` 只暴露 `kill()`

这说明当前系统已经默认放弃“所有 task 统一 spawn/render 多态”这类复杂抽象，真正统一的只有：

1. 生命周期状态
2. 输出文件
3. 通知/回收语义

这是一个很好的信号：Rust 版也应该让 task registry 只做统一控制面，不强制所有任务共享同一执行接口。

## 4. `TaskStateBase` 的含义

所有任务共有的稳定字段是：

1. `id`
2. `type`
3. `status`
4. `description`
5. `toolUseId`
6. `startTime` / `endTime`
7. `outputFile` / `outputOffset`
8. `notified`

其中最值得保留的是：

1. `outputFile`：让任务输出总能被统一引用。
2. `notified`：防止重复消息回注。
3. `outputOffset`：支持增量读取。

## 5. task framework 的真实职责

`utils/task/framework.ts` 说明 task framework 负责：

1. 注册任务到 `AppState.tasks`
2. 对指定任务做类型安全更新
3. 合并 offset patch 和 eviction
4. 发出 `task_started` / `task_notification` 事件

特别值得注意的是：

1. `registerTask()` 在 replacement 场景会保留 `retain`、`messages`、`diskLoaded` 等 UI-held 状态。
2. framework 本身不负责所有任务的 completion notification；某些任务类型自行通知。

这说明 task framework 是状态总线，不是完整生命周期协调器。

## 6. 输出文件抽象的意义

`utils/task/diskOutput.ts` 表明任务输出不是随便写磁盘：

1. 输出目录按 session 隔离。
2. `getTaskOutputDir()` 第一次调用后固定，避免 `/clear` 或 session 切换把旧任务路径搞丢。
3. 对 shell 输出有 5GB 上限。
4. 对 agent/main-session 任务，输出文件可以是 transcript 的 symlink。

因此“任务输出文件”其实是统一的恢复与观察接口，而不是单纯日志。

## 7. sidechain transcript 是真正的恢复基础设施

`recordSidechainTranscript()` 与 agent transcript path/mata file 的组合说明：

1. 每个可恢复 agent / background session 都有独立 transcript 文件。
2. transcript 记录沿用 message chain 语义，带 `startingParentUuid`。
3. metadata 额外持久化 `agentType`、`worktreePath`、`description`。

因此 resume 不是从 task state 恢复，而是从 sidechain transcript + metadata 恢复。

## 8. `runForkedAgent()` 的定位

这是内部维护型 fork 运行器，典型用途包括：

1. session memory 提取
2. compact summarize cache-sharing path
3. prompt/skill/command 级别的内部 fork

它的几个关键语义：

1. 强依赖 `CacheSafeParams` 以共享主线程 prompt cache。
2. 总是创建隔离的 subagent context，避免污染主线程可变状态。
3. 可选跳过 transcript，因此可以是完全 ephemeral 的内部 fork。
4. 自己累计 usage，而不是依赖主线程计数。

结论：它不是后台 task，而是 query 内核的内部子执行器。

## 9. `runAgent()` 的定位

和 `runForkedAgent()` 相比，`runAgent()` 是全功能子代理运行器：

1. 会构造 agent-specific system prompt。
2. 会重建 agent-specific tool pool。
3. 会处理 agent 前置 hooks、frontmatter hooks、skills、MCP servers。
4. 会把消息持续写入 sidechain transcript。
5. 会在 finally 里做比较重的生命周期清理。

也就是说，Rust 版应该把它当作 `SubagentRuntime`，而不是 task framework 的一部分。

## 10. `runAgent()` 暗示的 agent 隔离维度

### 10.1 工具池隔离

`AgentTool.tsx + agentToolUtils.ts` 显示：

1. worker tool pool 用 agent 自己的 permission mode 重建。
2. async agent 只能拿到受限工具集。
3. fork path 为了 prompt cache 命中，可以直接继承 parent exact tools。

### 10.2 权限模式隔离

`runAgent.ts` 会：

1. 根据 agent frontmatter 覆盖 permission mode。
2. 对 async agents 默认设置 `shouldAvoidPermissionPrompts`。
3. 对某些 background agents 允许先等自动化检查再决定是否弹权限。

### 10.3 可变状态隔离

`createSubagentContext()` 会克隆：

1. `readFileState`
2. content replacement state
3. abort controller
4. options / messages / agentId

async agent 默认不共享 `setAppState`，但仍保留一个 `rootSetAppState` 通道给 task/control-plane 写状态。

## 11. AsyncLocalStorage 的关键意义

`utils/agentContext.ts` 把 agent identity 放进 `AsyncLocalStorage`，原因非常明确：

1. 多个 background agents 可以并发跑在同一进程里。
2. `AppState` 是共享的，不能可靠承载每条异步链路的 agent identity。

这说明 Rust 版也必须给并发 agent 运行链路提供显式的 execution-local identity，而不是只靠全局状态。

## 12. 三种不同的子代理运行形态

### 12.1 异步从启动即后台化

`registerAsyncAgent()` + `runAsyncAgentLifecycle()` 这条路径的特征是：

1. 一开始就是 background task。
2. 立即注册 task state。
3. 默认不绑定 parent abort controller，因此主线程 ESC 不会杀掉它。
4. 最终通过 task notification 回流结果。

### 12.2 前台运行，途中切到后台

同步子代理会先：

1. `registerAgentForeground()`
2. 在 while-loop 中把 `agentIterator.next()` 和 `backgroundSignal` 竞态起来

一旦 background signal 触发：

1. 先关闭 foreground iterator，让其 finally 清理本轮资源。
2. 用同一个 agentId 和 task abort controller 重新起一个 async `runAgent()`。
3. 前台立即返回 `async_launched`。

这说明前后台切换不是“同一个 iterator 继续跑”，而是“同一 agent identity 下切换运行壳”。

### 12.3 主会话整体后台化

`LocalMainSessionTask.ts` 又是一种不同路径：

1. 用户当前主 query 被 background。
2. 新建 taskId，前缀是 `s`。
3. 复用 `local_agent` task type，但 `agentType = 'main-session'`。
4. 当前消息先写入 isolated sidechain transcript。
5. 再在后台继续跑主 query。

这里 TS 为了复用状态结构把 main-session task 塞进 `local_agent` 类型里，但 Rust 版最好单独显式建模。

## 13. `runAsyncAgentLifecycle()` 的核心语义

这个函数是后台 agent 的真正生命周期包装器：

1. 起 summarization sidecar。
2. 流式消费 `runAgent()`。
3. 更新 progress tracker 和 task.progress。
4. 一旦结束，先 `completeAsyncAgent()` / `failAsyncAgent()` / `killAsyncAgent()`。
5. 然后再跑 handoff classifier、worktree cleanup、enqueue notification。

这个“先切 terminal state，再做昂贵后处理”的顺序是非常关键的内核不变量。

## 14. task progress 与 notification 的关系

从 `LocalAgentTask.tsx` 和 `agentToolUtils.ts` 看：

1. progress 是持续增量更新的。
2. summary 是另一个后台 summarization 服务写入的，不应被普通 progress 覆盖。
3. terminal notification 是单次 XML/system event。

这三者必须分开：

1. progress stream
2. state snapshot
3. terminal notification

## 15. main-session background task 的特殊性

`LocalMainSessionTask.ts` 最有价值的几个点是：

1. 输出 symlink 指向 isolated transcript，而不是主 session transcript。
2. 这样 background session 可以跨 `/clear` 存活。
3. `foregroundedTaskId` 允许把某个 background session 拉回前台视图。
4. foregrounded 时完成任务不发 XML notification，但仍要给 SDK 发送终止事件。

因此 main-session backgrounding 本质上是“把主 query 暂时降格成 sidechain agent”。

## 16. resume 机制的真实语义

`resumeAgent.ts` 表明 resume 不是简单重放：

1. 先读 transcript + metadata。
2. 清理 whitespace-only assistant、orphaned thinking-only、unresolved tool uses。
3. 重建 content replacement state。
4. 若是 fork agent resume，还要重建 parent system prompt，并使用 exact tools。
5. 重新注册 async task，但复用同一个 agentId。

所以 resume 的核心不是 task state hydration，而是 transcript normalization + context reconstruction。

## 17. `runAgent()` 的清理职责

finally 中会清理：

1. agent-specific MCP servers
2. agent session hooks
3. prompt cache tracking state
4. cloned readFileState
5. transcript subdir mapping
6. agent todo entry
7. agent 派生出的 shell / monitor tasks

这说明子代理不是只运行 query loop，它还会注册很多外围资源，必须有集中 cleanup。

## 18. 对 Rust 的第一层抽象判断

建议至少拆成四层：

1. `TaskRegistry`：控制面可见任务与状态
2. `ForkedAgentRunner`：内部临时 fork 执行器
3. `SubagentRuntime`：全功能可恢复子代理运行时
4. `SidechainStore`：per-agent transcript + metadata + resume 入口

## 19. 当前已知缺口

虽然本轮材料已经足够写 Rust-facing 规格，但仍有几个缺口要保留：

1. `remote_agent`、`local_workflow`、`dream`、`monitor_mcp` 还未逐个细读。
2. in-process teammate 的特殊规则只在工具过滤和 ALS 上看到了接缝，尚未单独成文。
3. task output 的 shell/file 双模式尚未独立整理为规格。
