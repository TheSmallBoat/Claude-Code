# 宿主装配与扩展边界设计

## 文档状态

- 状态：Draft
- 日期：2026-04-03
- 作用：定义 Runtime Core 如何被 CLI/TUI/SDK/Remote host 装配，以及 tools/skills/plugins/MCP/hooks 如何接入

## 1. 设计目标

本设计要解决三个中层问题：

1. 谁负责把 core、code adapters、tasks、subagents、MCP、skills 装配到一个可运行会话里。
2. 外部扩展应通过哪些正式边界接入，而不是直接侵入 query 内核。
3. CLI/TUI/SDK/Remote host 如何共享同一 runtime，而不是各自复制一套控制逻辑。

## 2. 核心原则

### 2.1 Host 是装配者，不是内核实现者

CLI、TUI、SDK、Remote、IDE 集成都只能：

1. 提供 transport / UI / configuration / policy。
2. 选择启用哪些 adapters 与 workflows。
3. 创建并驱动同一个 runtime core。

它们不应各自实现一套 query / task / transcript 逻辑。

### 2.2 扩展只能通过 registry 接入

允许接入的扩展面应收敛到：

1. Tool registry
2. Command / workflow catalog
3. Agent catalog
4. Hook registry
5. MCP server set
6. Context contributors

除这些正式入口外，不允许扩展直接篡改 core 的状态结构。

### 2.3 Host-specific side effects 必须停留在外层

例如：

1. 终端备份恢复
2. keychain / auth 预取
3. UDS server
4. IDE bridge
5. plugin cache maintenance

都应停留在装配层或 sidecar 层，不得成为 query 内核的直接依赖。

## 3. 现有 TS 的装配启示

### 3.1 `main.tsx` 负责“入口级预热与总装”

从 `main.tsx` 可以看出当前系统把以下内容放在入口层：

1. profiling checkpoints
2. settings / policy / auth / telemetry 初始化
3. bundled skills / plugins 初始化
4. tools、commands、MCP、LSP manager 装配
5. app state store 创建
6. render + run

这说明总装根节点确实应位于 host 入口，而不是 runtime core 内部。

### 3.2 `setup.ts` 负责 session-scoped bootstrap

`setup.ts` 的职责包括：

1. cwd / session id / project root 初始化
2. worktree / tmux / hook watcher 装配
3. session memory、context collapse 等后台 sidecar 注册
4. 命令和能力预取

这说明“会话引导”应被建模为 host assembly 的第二阶段，而不是 QueryEngine 构造函数的一部分。

## 4. 建议的两阶段装配模型

建议把宿主装配拆成两个阶段：

### 4.1 Stage A: Process Bootstrap

负责一次进程级初始化：

1. env / settings / policy 加载
2. telemetry / diagnostics 初始化
3. plugin / bundled skills 静态注册
4. global MCP / IDE / auth sidecars 初始化

### 4.2 Stage B: Session Assembly

负责每个 session / conversation 的装配：

1. session id、cwd、project root、git root 决议
2. permission context 构造
3. tool registry / command catalog / agent catalog 实例化
4. context contributors 组装
5. task registry / sidechain store / query engine 创建
6. host transport 绑定

## 5. 建议的装配对象模型

建议显式定义：

```rust
pub struct HostBootstrap {
    pub config: Arc<RuntimeConfig>,
    pub policy: Arc<PolicyState>,
    pub sidecars: SidecarSet,
    pub extension_registry: ExtensionRegistry,
}

pub struct SessionAssembly {
    pub session_id: SessionId,
    pub runtime: RuntimeHandle,
    pub tool_registry: ToolRegistry,
    pub command_catalog: CommandCatalog,
    pub agent_catalog: AgentCatalog,
    pub task_registry: TaskRegistry,
    pub host_bindings: HostBindings,
}

pub struct RuntimeHandle {
    pub query_engine: QueryEngine,
    pub transcript_store: TranscriptStore,
    pub sidechain_store: SidechainStore,
    pub compact_services: CompactServices,
}
```

## 6. 正式扩展边界

### 6.1 Tool Registry

所有可供模型调用的能力必须先映射成 tool descriptor。

来源可以是：

1. built-in coding adapters
2. MCP tools
3. SDK-defined tools
4. synthetic tools

### 6.2 Command / Workflow Catalog

所有 prompt skills、workflow commands、plugin commands、bundled skills 都必须通过 catalog 暴露，而不是直接插入 query 消息流。

### 6.3 Agent Catalog

子代理定义必须独立于 tool registry：

1. agent prompt
2. allowed/disallowed tools
3. model
4. mcp servers
5. preload skills

### 6.4 Hook Registry

hooks 是外围治理与宿主协同机制，不应嵌入 QueryEngine 核心状态。

### 6.5 Context Contributors

git snapshot、memory attachments、plan files、task attachments、MCP instructions 等都应作为可组合的 context contributors 接入。

## 7. Host 类型划分

建议统一支持四类宿主：

1. interactive CLI/TUI host
2. headless SDK host
3. remote/bridge host
4. IDE-assisted host

它们共享同一个 `SessionAssembly`，只在以下方面不同：

1. permission prompt transport
2. rendering / event sink
3. available command subset
4. sidecar availability

## 8. Remote / Bridge 边界

`commands.ts` 中的 `REMOTE_SAFE_COMMANDS` 与 `BRIDGE_SAFE_COMMANDS` 很重要，它们证明 remote host 不是“复制一份本地宿主能力”，而是：

1. 复用同一 command/tool model。
2. 但根据 transport 与 locality 过滤掉依赖本地 shell / filesystem / IDE 的能力。

Rust 版应把这类过滤写成 host policy，而不是散落在 command 实现中。

## 9. MCP 边界

MCP 在装配层的角色应是：

1. 提供额外 tools / resources / prompts。
2. 通过 registry 合并进当前 session。
3. 在 query 结束或 subagent finally 时按作用域清理。

MCP 不应直接拥有 query 生命周期；它只是 capability provider。

## 10. Skills / Plugins / Workflows 边界

这一层应被视为“领域编排资产”，而不是 runtime core 的一部分。

需要支持：

1. bundled skills
2. project skill dirs
3. dynamic path-triggered skills
4. plugin commands / plugin skills
5. workflow scripts

但这些资产只能通过 `CommandCatalog` / `WorkflowCatalog` 暴露，不应直接向核心层写消息或改状态。

## 11. Host 绑定接口

建议把 UI/transport 需求收敛到 `HostBindings`：

```rust
pub struct HostBindings {
    pub event_sink: Arc<dyn EventSink>,
    pub permission_ui: Arc<dyn PermissionPromptSink>,
    pub notification_sink: Arc<dyn NotificationSink>,
    pub file_watch_sink: Option<Arc<dyn FileWatchSink>>,
    pub ide_bridge: Option<Arc<dyn IdeBridgeAdapter>>,
}
```

这样 query / tool / task 层只需向抽象 sink 发事件，而不需要知道当前是 TUI、SDK 还是 remote socket。

## 12. 装配顺序建议

每个 session 的装配顺序建议固定为：

1. 解析 host config 与 session source。
2. 决议 cwd、project root、repo root、worktree state。
3. 构造 permission context 与 host bindings。
4. 收集 built-in tools、MCP tools、SDK tools。
5. 构造 tool registry。
6. 收集 bundled/project/plugin/workflow/remote-safe commands。
7. 构造 command catalog 与 agent catalog。
8. 创建 transcript store、task registry、query engine。
9. 注册 session memory、context collapse、watchers 等 sidecars。
10. 将 session assembly 交给具体 host runner。

## 13. 失败隔离原则

装配层必须保证以下失败不会污染 core：

1. plugin load failure
2. skill directory parse failure
3. MCP registry fetch failure
4. IDE bridge notification failure
5. LSP manager unavailable

这些问题应退化为：

1. 少一个 capability
2. 少一个 workflow
3. 少一个 sidecar

而不是让 session 无法创建或让 transcript 状态损坏。

## 14. Rust 模块建议

建议在 core 之外新增：

1. `agent-host-assembly`
2. `agent-extension-registry`
3. `agent-host-cli`
4. `agent-host-sdk`
5. `agent-host-remote`

其中：

1. `agent-host-assembly` 负责 Stage A / Stage B 装配。
2. `agent-extension-registry` 负责 tools/commands/agents/hooks/MCP 的统一合并。
3. 各 host crate 只负责 transport / UI / policy 差异。

## 15. 验收标准

这份设计只有在以下条件下才算成立：

1. 任一 host 都可以复用同一个 `SessionAssembly`。
2. 扩展新增只需注册到 registry，而不需要修改 QueryEngine。
3. plugin / skill / MCP / IDE sidecar 失败时，session 仍可继续运行。
4. remote / bridge 的命令过滤由 host policy 负责，而不是零散硬编码。
5. host entry 只做装配与 transport，不做核心状态机实现。
