# 工具调度与权限内核规格

## 文档状态

- 状态：Draft
- 日期：2026-04-03
- 对应 Rust 目标模块：`agent-tools`、`agent-permissions`

## 1. 目标

定义一套可在 Rust 中实现的工具系统与权限内核，使其能够复现 TS 版本中最关键的三类能力：

1. 能力描述与工具集装配。
2. 并发安全约束下的工具调度。
3. 规则、模式与内容感知相结合的权限决策。

## 2. 非目标

本规格明确不覆盖：

1. UI 层的权限弹窗、React 组件和视觉呈现。
2. 各具体工具的完整业务实现细节。
3. 插件市场、MCP 管理界面、特定宿主交互。

## 3. 代码证据范围

当前规格主要由以下代码逆向得到：

1. `TS-Source/Tool.ts`
2. `TS-Source/tools.ts`
3. `TS-Source/services/tools/toolOrchestration.ts`
4. `TS-Source/hooks/useCanUseTool.tsx`
5. `TS-Source/utils/permissions/permissions.ts`
6. `TS-Source/utils/permissions/filesystem.ts`
7. `TS-Source/utils/permissions/permissionSetup.ts`
8. `TS-Source/Task.ts`

## 4. 设计原则

### 4.1 Tool 要拆成“描述”和“执行”两部分

TS 的 `Tool` 接口同时塞入了：

1. 运行时执行逻辑。
2. 权限逻辑。
3. 输入/输出 schema。
4. 宿主渲染逻辑。

Rust 版必须拆解。Runtime Core 中只保留前 3 类，渲染相关全部外移。

### 4.2 未声明的安全属性按保守策略处理

默认规则必须是：

1. 未声明并发安全，则视为不安全。
2. 未声明只读，则视为可能写。
3. 未声明危险性，则不自动提升权限，但不得隐式乐观并发。

### 4.3 权限决策引擎与交互提示解耦

`ask` 是一种决策结果，不是 UI 事件。Runtime Core 只负责产生 ask/allow/deny，真正的用户提示由宿主适配层实现。

## 5. Tool 核心对象模型

建议在 Rust 中定义：

```rust
pub enum ToolEffectClass {
    ReadOnly,
    Mutating,
    External,
    Interactive,
    Unknown,
}

pub struct ToolDescriptor {
    pub name: String,
    pub aliases: Vec<String>,
    pub input_schema: JsonSchema,
    pub effect_class: ToolEffectClass,
    pub concurrency_safe: bool,
    pub destructive: bool,
    pub requires_user_interaction: bool,
    pub max_result_size_chars: usize,
}

pub struct ToolCall {
    pub id: ToolUseId,
    pub tool_name: String,
    pub input: serde_json::Value,
}

pub struct ToolOutcome {
    pub blocks: Vec<Message>,
    pub context_patch: Option<ToolContextPatch>,
    pub task_handle: Option<TaskHandle>,
}
```

## 6. Tool 执行接口

建议定义：

```rust
#[async_trait]
pub trait ToolExecutor {
    async fn execute(
        &self,
        call: ToolCall,
        ctx: &mut ToolExecutionContext,
        progress: &mut dyn ToolProgressSink,
    ) -> Result<ToolOutcome, ToolError>;
}
```

`ToolExecutionContext` 至少需要包含：

1. 当前消息视图。
2. 会话状态访问器。
3. 中断信号。
4. 权限上下文访问器。
5. read file cache / memory state。
6. agentId / query tracking 等环境信息。

## 7. ToolRegistry 与 ToolSetResolver

### 7.1 ToolRegistry

职责：

1. 持有全部已编译进系统的 `ToolDescriptor`。
2. 支持按主名或别名查找。

### 7.2 ToolSetResolver

职责：

1. 根据 runtime mode、feature gates、agent type、permission context 解析当前 turn 的可见工具集。
2. 将系统存在的工具集和模型可见的工具集区分开。

这直接对应 TS 中 `tools.ts` 的装配逻辑，但在 Rust 中应做成显式模块。

## 8. ToolScheduler 规范

### 8.1 批次划分算法

对一个 assistant turn 输出的多组 `tool_use`：

1. 逐个读取工具描述与输入。
2. 调用 `is_concurrency_safe(input)` 或等价策略。
3. 连续的并发安全调用组成并发批。
4. 非并发安全调用单独形成串行批。

### 8.2 并发批执行规则

1. 批内调用可并发执行。
2. 每个工具可产生 `context_patch`。
3. 这些 patch 不得即时写入共享上下文。
4. 必须等整批完成后，按原始 tool_use 顺序统一提交。

### 8.3 串行批执行规则

1. 每个工具执行完成后，允许即时更新上下文。
2. 下一个工具必须看到前一个工具已提交的上下文变化。

### 8.4 中断规则

1. 若执行期间 abort，则当前批次必须终止。
2. 对未完成的 tool_use，必须产生可恢复的一致性结束信号。
3. 不能让消息流出现 tool_use 无 tool_result 的悬空状态。

## 9. PermissionEngine 规范

### 9.1 决策结果

建议定义：

```rust
pub enum PermissionDecision {
    Allow {
        updated_input: serde_json::Value,
        reason: PermissionDecisionReason,
    },
    Ask {
        message: String,
        suggestions: Vec<PermissionUpdate>,
        reason: PermissionDecisionReason,
    },
    Deny {
        message: String,
        reason: PermissionDecisionReason,
    },
}
```

### 9.2 决策流水线顺序

权限求值必须按以下顺序：

1. 整个工具级 deny rule。
2. 整个工具级 ask rule。
3. tool-specific `check_permissions(input, context)`。
4. tool-specific deny。
5. `requires_user_interaction` 的强制 ask。
6. content-specific ask rule。
7. safety-check ask。
8. permission mode 是否 bypass。
9. 整个工具级 allow rule。
10. `passthrough -> ask` 收口。

该顺序属于基线语义，Rust 版不得调整优先级。

## 10. Permission Rule 模型

规则模型至少需要：

```rust
pub enum PermissionBehavior {
    Allow,
    Deny,
    Ask,
}

pub struct PermissionRuleValue {
    pub tool_name: String,
    pub rule_content: Option<String>,
}

pub struct PermissionRule {
    pub source: PermissionRuleSource,
    pub behavior: PermissionBehavior,
    pub value: PermissionRuleValue,
}
```

其中 `rule_content` 必须允许：

1. 针对整个工具的规则。
2. 针对子命令或路径模式的内容规则。
3. 针对 agent type 或 MCP server 的扩展规则。

## 11. 文件系统权限子策略

对编程领域工具，文件系统权限必须成为独立子模块。

### 11.1 读权限策略

必须依次执行：

1. 危险路径样式拦截。
2. read-specific deny。
3. read-specific ask。
4. edit access implies read access。
5. working directory allow。
6. internal readable path allow。
7. allow rule。
8. 默认 ask。

### 11.2 写权限策略

必须依次执行：

1. deny。
2. internal editable path allow。
3. 特殊路径豁免规则。
4. auto edit safety checks。
5. allow / ask / 默认收口。

## 12. PermissionModeManager 规范

permission mode 不能只靠枚举值表达，必须有带副作用的 transition manager。

建议定义：

```rust
pub enum PermissionMode {
    Default,
    AcceptEdits,
    Plan,
    Auto,
    BypassPermissions,
}

pub trait PermissionModeManager {
    fn transition(
        &mut self,
        from: PermissionMode,
        to: PermissionMode,
        ctx: PermissionContextState,
    ) -> PermissionContextState;
}
```

transition 期间必须支持：

1. 进入 auto 时 strip dangerous allow rules。
2. 离开 auto 时 restore dangerous allow rules。
3. 进入/退出 plan 时保留 `pre_plan_mode` 或等价状态。
4. 对不允许进入的 mode 做 fail-fast。

## 13. Ask 决策与宿主边界

Runtime Core 在遇到 `Ask` 决策时，不直接弹窗，而是返回一个宿主可消费的请求对象：

```rust
pub trait PermissionPromptAdapter {
    async fn resolve(&self, request: PermissionPromptRequest) -> PermissionPromptResponse;
}
```

CLI、TUI、SDK、background worker 可以分别实现不同的 adapter：

1. 交互式宿主弹窗。
2. headless agent 自动 deny。
3. coordinator worker 转发给 leader。
4. remote host 走控制通道。

## 14. 任务与工具的最小耦合点

对基线 Runtime Core 而言，任务系统在本规格中的最小抽象是：

1. 工具可以启动一个长期执行单元。
2. 启动结果返回 `TaskHandle`。
3. 任务拥有 `pending/running/completed/failed/killed` 终态机。
4. query 内核只关心 task id、状态和输出 artifact，不关心 UI 表现。

## 16. 可直接转化的 Rust 编程任务

1. 定义 `ToolDescriptor`、`ToolCall`、`ToolOutcome`、`ToolEffectClass`。
2. 实现 `ToolRegistry` 与按别名查找。
3. 实现 `ToolScheduler` 的批次划分和 context patch 提交机制。
4. 实现 `PermissionRule`、`PermissionDecision`、`PermissionDecisionReason`。
5. 实现 `PermissionEngine` 的标准流水线。
6. 实现 `FilesystemPermissionPolicy`。
7. 实现 `PermissionModeManager` 的状态迁移和 dangerous rule stash/restore。
8. 预留 `PermissionPromptAdapter`，把交互式 ask 从内核剥离。

## 17. 验收测试矩阵

至少应覆盖以下测试：

1. 并发安全工具被正确并发分批。
2. 非并发安全工具严格串行执行。
3. 并发批中的 context patch 延迟到整批结束后提交。
4. deny rule 高于 ask rule 与 allow rule。
5. content-specific ask rule 在 bypass mode 下仍然生效。
6. safety-check ask 在 bypass mode 下仍然生效。
7. read-specific deny/ask 高于 edit access implies read access。
8. auto mode 进入时危险规则被剥离，退出时被恢复。
9. headless 宿主收到 ask 决策时可自动 deny 而不破坏内核语义。

## 18. 当前已知缺口

本规格已经足够支撑 `agent-tools` 和 `agent-permissions` 的骨架实现，但还需要后续文档补齐：

1. 消息 IR 与 transcript 如何表达 tool result pairing。
2. 任务系统完整模型。
3. MCP 工具与编程领域工具的统一能力分类。
4. auto-mode classifier 的更精细策略接口。
