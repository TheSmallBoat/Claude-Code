# 消息与转录模型规格

## 文档状态

- 状态：Draft
- 日期：2026-04-03
- 对应 Rust 目标模块：`agent-message`、`agent-session-store`

## 1. 目标

定义 Runtime Core 的消息 IR、transcript 持久化模型、compact boundary 恢复锚点，以及 tool-use 配对完整性规则，使 Rust 基线版能够正确支撑：

1. 单回合执行。
2. 跨回合恢复。
3. compact 后 resume。
4. sidechain / subagent transcript。

## 2. 非目标

本规格明确不覆盖：

1. 消息 UI 渲染组件。
2. 移动端或 Web SDK 的序列化兼容细节。
3. 训练数据导出与分享侧的额外格式。

## 3. 设计原则

### 3.1 消息是统一事实源

当前 TS 实现中，用户输入、模型输出、系统生命周期事件、工具结果、compact boundary、stop hook summary 都被尽量重写成消息。因此 Rust 版必须延续“消息优先”的架构，而不是把系统状态散落到多个不可追溯的对象中。

### 3.2 运行态消息与持久化消息必须分层

运行态存在 `progress`、`tombstone` 等消息，但 transcript 只持久化其中一部分。因此必须区分：

1. `RuntimeMessage`
2. `TranscriptEntry`

不能用单一类型硬撑两种语义。

### 3.3 compact 不是删历史，而是重写链关系

compact boundary 与 preserved segment 元数据共同决定 resume 后的可见历史。Rust 版必须保留这种“链重写”语义。

## 4. Runtime Message IR

建议定义：

```rust
pub struct MessageId(pub Uuid);

pub struct MessageMeta {
    pub id: MessageId,
    pub timestamp: DateTime<Utc>,
    pub is_meta: bool,
    pub is_virtual: bool,
    pub origin: Option<MessageOrigin>,
}

pub enum RuntimeMessage {
    User(UserMessage),
    Assistant(AssistantMessage),
    System(SystemMessage),
    Attachment(AttachmentMessage),
    Progress(ProgressMessage),
    Tombstone(TombstoneMessage),
    ToolUseSummary(ToolUseSummaryMessage),
}
```

### 4.1 UserMessage

必须表示“user-role message”，而不是仅表示“人工输入”。它需要支持：

1. 真实用户 prompt。
2. tool_result 回注。
3. continuation meta message。
4. command breadcrumb。

### 4.2 AssistantMessage

必须支持：

1. 正常 assistant content。
2. synthetic API error。
3. 虚拟或恢复性占位消息。

### 4.3 SystemMessage

至少需要支持以下 subtype：

1. `Informational`
2. `CompactBoundary`
3. `MicrocompactBoundary`
4. `StopHookSummary`
5. `TurnDuration`
6. `ApiError`
7. `LocalCommand`

其中 `CompactBoundary` 必须强类型化，不得退化成普通字符串系统消息。

## 5. 内容块模型

assistant/user 的内容块至少需要覆盖：

1. `Text`
2. `ToolUse`
3. `ToolResult`
4. `Thinking`
5. `RedactedThinking`
6. `Image`
7. 预留扩展块类型

建议：

```rust
pub enum MessageBlock {
    Text(TextBlock),
    ToolUse(ToolUseBlock),
    ToolResult(ToolResultBlock),
    Thinking(ThinkingBlock),
    RedactedThinking(RedactedThinkingBlock),
    Image(ImageBlock),
    Other(serde_json::Value),
}
```

## 6. CompactBoundary 模型

建议定义：

```rust
pub struct CompactBoundaryMessage {
    pub meta: MessageMeta,
    pub trigger: CompactTrigger,
    pub pre_tokens: u32,
    pub user_context: Option<String>,
    pub messages_summarized: Option<u32>,
    pub logical_parent: Option<MessageId>,
    pub preserved_segment: Option<PreservedSegment>,
}

pub struct PreservedSegment {
    pub head_id: MessageId,
    pub anchor_id: MessageId,
    pub tail_id: MessageId,
}
```

其中 `preserved_segment` 是 resume 后 relink 的核心，不可省略。

## 7. TranscriptEntry 模型

transcript 层必须区别于 runtime message。建议定义：

```rust
pub struct TranscriptEntry {
    pub message: PersistedMessage,
    pub session_id: SessionId,
    pub cwd: PathBuf,
    pub user_type: UserType,
    pub version: String,
    pub parent_id: Option<MessageId>,
    pub logical_parent_id: Option<MessageId>,
    pub is_sidechain: bool,
    pub agent_id: Option<AgentId>,
    pub prompt_id: Option<String>,
}
```

还需要支持非消息型日志条目：

1. custom title
2. last prompt
3. tag
4. agent metadata
5. attribution snapshot
6. file history snapshot
7. content replacement
8. context collapse commit/snapshot

## 8. 持久化过滤规则

持久化前必须经过 transcript normalization：

1. `progress` 不进入 transcript。
2. 对 external transcript，大多数 attachment 不进入 transcript。
3. 必要时移除 REPL wrapper 的 tool_use/tool_result。
4. `is_virtual` 消息可在 transcript 侧降格为真实消息。

建议定义：

```rust
pub trait TranscriptNormalizer {
    fn normalize(
        &self,
        messages: &[RuntimeMessage],
        full_turn_view: &[RuntimeMessage],
    ) -> Vec<PersistedMessage>;
}
```

## 9. parentUuid 链规则

`recordTranscript()` 表明 transcript 写入不是简单 append，而是维护 parent 链。必须满足：

1. 新写入消息只与“已记录前缀”中的最后 chain participant 续链。
2. 已记录但不在前缀位置的消息，不得成为新链父节点。
3. compact boundary 可通过 `logical_parent_id` 保留逻辑续链信息。

这意味着 transcript writer 需要知道：

1. 已落盘消息集合。
2. 当前待写切片。
3. 哪些消息参与链计算。

## 10. Main Transcript 与 Sidechain Transcript

必须显式区分：

1. 主会话 transcript。
2. subagent/background task 的 sidechain transcript。

二者共享：

1. 同一消息序列化规则。
2. 同一 parent 链逻辑。

但不同点是：

1. 文件路径不同。
2. agentId 命名空间不同。
3. resume 入口不同。

## 11. Tool-use 配对完整性

必须存在一个独立的完整性修复/验证器，执行以下工作：

1. 去除重复 `tool_use id`。
2. 移除 orphaned `tool_result`。
3. 修复或拒绝缺失 `tool_result` 的 assistant trajectory。
4. 校验 server-side tool use/result 对。

建议定义：

```rust
pub trait MessageIntegrityNormalizer {
    fn ensure_tool_result_pairing(
        &self,
        messages: &[RoleMessage],
    ) -> Result<Vec<RoleMessage>, PairingError>;
}
```

对于训练或高保真模式，可提供 strict/fail-fast 策略而不是自动修补。

## 12. Compact Relinker 规范

transcript loader 在恢复 compact 后会话时，必须支持：

1. 找到绝对最后一个 compact boundary。
2. 若其 `preserved_segment` 存在，则执行 tail -> head walk 验证。
3. 若 walk 失败，则不做 relink，并保留完整历史。
4. 若 walk 成功，则把 head 重新挂到 anchor，并裁剪较早历史。

建议定义：

```rust
pub trait CompactRelinker {
    fn relink(
        &self,
        entries: &mut IndexMap<MessageId, TranscriptEntry>,
    ) -> CompactRelinkResult;
}
```

## 13. TranscriptWriter 规范

writer 必须支持：

1. 增量追加。
2. 去重。
3. sidechain 写入。
4. compact boundary 相关补写。
5. flush。
6. tombstone 删除或等价重写。

## 14. TranscriptLoader 规范

loader 必须支持：

1. 从 JSONL 读取消息与非消息元数据。
2. 恢复 parent 链。
3. 应用 compact relink。
4. 跳过 legacy progress entry。
5. 计算 leaf 节点，用于 resume 锚定。

## 15. Rust 实现建议拆分

### 15.1 `agent-message`

1. `message.rs`
2. `blocks.rs`
3. `system.rs`
4. `integrity.rs`
5. `compact.rs`

### 15.2 `agent-session-store`

1. `entry.rs`
2. `writer.rs`
3. `loader.rs`
4. `relink.rs`
5. `metadata.rs`

## 16. 可直接转化的 Rust 编程任务

1. 定义 `RuntimeMessage`、`MessageMeta`、`MessageBlock`。
2. 定义 `CompactBoundaryMessage` 与 `PreservedSegment`。
3. 定义 `TranscriptEntry` 与非消息型日志条目。
4. 实现 `TranscriptNormalizer`。
5. 实现 `TranscriptWriter` 的增量去重与 parent 链维护。
6. 实现 `MessageIntegrityNormalizer::ensure_tool_result_pairing`。
7. 实现 `TranscriptLoader` 与 `CompactRelinker`。

## 17. 验收测试矩阵

至少应覆盖以下测试：

1. 运行态 `progress` 不进入 transcript。
2. external transcript 会去除 REPL 包装层。
3. `recordTranscript` 只把前缀已记录消息作为新链父节点。
4. sidechain transcript 与主 transcript 使用相同序列化规则但不同路径。
5. compact boundary 的 preserved segment 在恢复时可正确 relink。
6. preserved segment 元数据损坏时，恢复逻辑应 fail-open 保留完整历史。
7. orphaned tool_result 会被移除或转为可接受占位。
8. duplicate tool_use id 会被清理或拒绝。

## 18. 当前已知缺口

本规格已经足够支撑 `agent-message` 与 `agent-session-store` 的骨架实现，但后续仍需补齐：

1. attachment 的完整分类与生命周期。
2. context collapse commit/snapshot 的完整 transcript 语义。
3. `insertMessageChain()` 的精确写入策略。
4. 远程/SDK 侧的跨进程 transcript 兼容策略。
