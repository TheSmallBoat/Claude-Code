# 编程领域能力适配层规格

## 文档状态

- 状态：Draft
- 日期：2026-04-03
- 对应 Rust 目标模块：`agent-code-adapters`、`agent-code-workflows`

## 1. 目标

定义 Runtime Core 之上的“编程领域能力适配层”，使 Rust 基线版能够稳定承载：

1. 工作区文件读写与 notebook 编辑。
2. 内容搜索、文件枚举、代码智能查询。
3. shell / 进程执行与背景任务集成。
4. git / repo / IDE 等编程宿主语义。
5. skills / workflows / coding commands 的装配与激活。

## 2. 非目标

本规格明确不覆盖：

1. 具体技能文案和 prompt 模板内容。
2. review / PR / release / browser 等产品工作流细节。
3. VSCode UI 展示与终端面板渲染细节。
4. 特定语言服务器实现的内部协议细节。

## 3. 设计原则

### 3.1 Runtime Core 不直接知道“编程工具”

Core 只知道：

1. tool contract
2. command contract
3. permission contract
4. task / transcript / context contract

`git`、`rg`、`.ipynb`、LSP、VSCode 这类内容必须由独立 adapter 注入。

### 3.2 所有 mutating code operations 必须共享同一安全契约

文件写、文件编辑、notebook 编辑、shell 触发的安全写入，必须统一遵守：

1. path normalization
2. permission check before I/O
3. read-before-write
4. stale detection
5. post-write cache / IDE / LSP sync

### 3.3 搜索、执行、代码智能是不同能力面

不能把：

1. glob / grep
2. shell exec
3. LSP queries

混成一个“workspace access”接口，否则权限、缓存和结果建模都会失真。

### 3.4 skills / workflows 是编排层，不是底层能力层

skills / workflows 只能组合能力 adapter，不能反过来定义 adapter。

### 3.5 IDE / LSP / repo 都应当是可选 sidecar

没有 IDE、没有 LSP、不是 git repo 时，runtime core 仍然应该能工作，只是失去相应 coding augmentation。

## 4. 总体层次模型

建议明确分三层：

1. `agent-runtime-core`
2. `agent-code-adapters`
3. `agent-code-workflows`

其中：

1. core 负责 query / messages / tools / permissions / tasks / compaction。
2. adapters 负责文件、搜索、shell、repo、LSP、IDE 等宿主能力。
3. workflows 负责 skills、commands、workflows、review/commit 类策略拼装。

## 5. 核心适配器接口

建议定义：

```rust
pub trait CodeWorkspaceReader {
    async fn read_text(&self, req: ReadTextRequest) -> ReadTextResult;
    async fn read_binary(&self, req: ReadBinaryRequest) -> ReadBinaryResult;
    async fn read_notebook(&self, req: ReadNotebookRequest) -> NotebookView;
}

pub trait CodeWorkspaceWriter {
    async fn write_file(&self, req: WriteFileRequest) -> WriteFileResult;
    async fn edit_file(&self, req: EditFileRequest) -> EditFileResult;
    async fn edit_notebook(&self, req: EditNotebookRequest) -> EditNotebookResult;
}

pub trait CodeSearchAdapter {
    async fn glob(&self, req: GlobRequest) -> GlobResult;
    async fn grep(&self, req: GrepRequest) -> GrepResult;
}

pub trait CodeExecutionAdapter {
    async fn exec(&self, req: ExecRequest) -> ExecResult;
    async fn spawn_background(&self, req: ExecRequest) -> BackgroundExecHandle;
}

pub trait CodeIntelAdapter {
    async fn query(&self, req: CodeIntelRequest) -> CodeIntelResult;
}

pub trait CodeRepoAdapter {
    async fn snapshot(&self) -> RepoSnapshot;
    async fn file_diff(&self, req: FileDiffRequest) -> FileDiffResult;
    async fn canonical_root(&self) -> Option<PathBuf>;
}

pub trait IdeBridgeAdapter {
    async fn notify_file_updated(&self, event: FileUpdatedEvent);
}

pub trait CodeWorkflowCatalog {
    async fn load_commands(&self, cwd: &Path) -> Vec<WorkflowCommand>;
    async fn activate_for_paths(&self, paths: &[PathBuf]) -> Vec<WorkflowCommand>;
}
```

## 6. 通用写入契约

所有 mutating code adapters 必须共享以下顺序：

1. 归一化输入路径。
2. 在任何真实 I/O 之前执行权限检查。
3. 阻断 UNC / network path 等不应意外交网的路径类型。
4. 若目标已存在，则要求先 read。
5. 比较读取时间戳与当前文件状态，必要时做内容回退比对。
6. 可选记录 file history / undo snapshot。
7. 落盘。
8. 刷新 read-state cache。
9. 通知 LSP / IDE / diagnostics sidecars。

这个契约应当实现为共享 helper，而不是复制到每个 tool 中。

## 7. CodeWorkspaceReader 规范

### 7.1 支持的读取形态

至少应支持：

1. 文本文件按行区间读取。
2. 图片按压缩后 payload 读取。
3. notebook 按 cell 结构读取。
4. PDF 按页区间读取或拆分。

### 7.2 安全与预算要求

1. block device / FIFO / socket / stdin 类路径必须拒绝。
2. 超大文件必须按 token/size 限制拒绝或要求局部读取。
3. 返回值必须带 enough metadata 让上层知道是全文、局部视图还是媒体对象。

## 8. CodeSearchAdapter 规范

### 8.1 glob / grep 分离

必须明确区分：

1. `glob`：路径空间枚举。
2. `grep`：内容空间查询。

### 8.2 grep 能力要求

至少支持：

1. output mode
2. context lines
3. `head_limit` / `offset` 分页
4. file type filter
5. multiline mode

### 8.3 结果治理

1. 默认结果数必须有限制。
2. 结果应相对化以节省 token。
3. VCS 目录、插件缓存等噪音目录应自动排除。

## 9. CodeExecutionAdapter 规范

### 9.1 职责

负责运行 shell / process 语义，但必须带策略层：

1. sandbox 选择
2. permission / safety classifier 协作
3. 背景任务注册
4. 大输出落盘
5. auto-background / manual-background

### 9.2 语义要求

必须支持：

1. 命令语义分类，区分 search/read/list/silent/mutating。
2. 前台执行与后台执行两种路径。
3. 大输出持久化与增量读取。
4. 特殊安全转换路径，例如 sed preview 后转为直接文件写入。

### 9.3 与 task plane 的关系

`CodeExecutionAdapter` 不直接拥有 task registry，但后台进程必须能被 task plane 包装为可观察对象。

## 10. CodeIntelAdapter 规范

### 10.1 定位

代码智能是可选 sidecar，不得成为 query 内核前提。

### 10.2 最低支持能力

至少支持：

1. go to definition
2. find references
3. hover
4. document/workspace symbols
5. implementation
6. call hierarchy

### 10.3 运行约束

1. server 未初始化时可等待初始化，但不能阻塞整个 runtime 启动路径。
2. 对未 open 文件，应先送入当前磁盘内容。
3. 大文件应有限制。
4. 基于位置的结果应支持 gitignore / workspace 过滤。

## 11. CodeRepoAdapter 规范

### 11.1 职责

负责：

1. git root / canonical root 识别
2. worktree 归一化
3. repo snapshot 注入
4. diff / status / commit history 读取

### 11.2 安全要求

worktree / commondir 解析必须防止恶意仓库借用受信任路径，不能只靠字符串拼接。

## 12. IdeBridgeAdapter 规范

这层用于 IDE 协同，而不是 query 核心。

最低要求：

1. 文件写入后可通知 IDE 做 diff / refresh。
2. 可选接收宿主发回的 telemetry / gate 信息。
3. 失败必须 fail-open，不能影响主工作流。

## 13. CodeWorkflowCatalog 规范

### 13.1 工作流来源

必须支持从多来源装配工作流：

1. bundled skills
2. project skill dirs
3. plugins
4. workflows
5. MCP-provided skills

### 13.2 工作流元数据

工作流至少可携带：

1. description
2. allowed tools
3. `context = inline | fork`
4. fork agent type
5. `paths` 条件
6. hooks
7. effort / model

### 13.3 动态发现与激活

必须支持：

1. 从被读写文件路径向上发现 `.claude/skills`。
2. 按路径模式动态激活 conditional skills。
3. 在 remote / bridge 模式下过滤掉依赖本地宿主的工作流。

## 14. Tool Registry 与 Adapter Registry 的关系

`tools.ts` 只是把 adapter 暴露为 model-callable tools 的组装层。

Rust 版建议：

1. 先注册 host adapters。
2. 再由 `ToolRegistryBuilder` 把它们映射成 tool definitions。
3. workflow/skill catalog 再基于可用 tools 生成 prompt commands。

这样 core 不需要知道具体有哪些 coding tools。

## 15. Rust 模块拆分建议

### 15.1 `agent-code-adapters`

1. `workspace_reader.rs`
2. `workspace_writer.rs`
3. `search.rs`
4. `execution.rs`
5. `repo.rs`
6. `intel.rs`
7. `ide_bridge.rs`
8. `mutation_guard.rs`

### 15.2 `agent-code-workflows`

1. `catalog.rs`
2. `skill_loader.rs`
3. `conditional_activation.rs`
4. `command_filter.rs`
5. `tool_projection.rs`

## 16. 可直接转化的 Rust 编程任务

1. 实现统一的 `MutationGuard`，封装 path normalize、permission、stale check、post-write sync。
2. 实现 `CodeWorkspaceReader`，支持 text/image/notebook/pdf 四类读取。
3. 实现 `CodeWorkspaceWriter`，支持 write/edit/notebook edit 三类写入。
4. 实现 `CodeSearchAdapter`，先落 `glob + grep`，保留 semantic/LSP search 扩展位。
5. 实现 `CodeExecutionAdapter`，支持 foreground/background、输出落盘、task handle 对接。
6. 实现 `CodeRepoAdapter`，支持 git snapshot、canonical root、single-file diff。
7. 实现 `CodeIntelAdapter`，支持基础 LSP operations 与 open-before-query。
8. 实现 `IdeBridgeAdapter`，至少支持 file-updated fail-open 通知。
9. 实现 `CodeWorkflowCatalog`，支持 bundled/project/plugin/workflow 多来源装配。
10. 实现基于 file-touch 的动态 skill discovery 与 conditional activation。

## 17. 验收测试矩阵

至少应覆盖以下测试：

1. text/image/notebook/pdf 读取路径都能返回正确的 typed result。
2. device file / UNC path 会在真实 I/O 前被拒绝或短路。
3. 文件写入若未 read 或已 stale，会被拒绝。
4. 文件写入后会同步刷新 read cache，并通知 LSP / IDE sidecars。
5. grep 会执行默认分页，且 offset/head_limit 组合行为稳定。
6. glob/grep 结果会自动剔除噪音目录并相对化路径。
7. shell 后台任务的输出可落盘并由 task plane 持续读取。
8. sed-preview 类安全写入路径会写出与预览一致的结果。
9. worktree / canonical root 解析不会把恶意 repo 误归到受信任仓库。
10. path-triggered skill discovery / activation 会在读写指定文件后生效。
11. `context=fork` 的工作流会转为 subagent 路径，而不是 inline prompt expansion。
12. remote-safe / bridge-safe 过滤会移除依赖本地宿主的命令。

## 18. 当前已知缺口

本规格已经足以启动 `agent-code-adapters` 与 `agent-code-workflows` 的骨架设计，但仍需后续文档继续补齐：

1. review / commit / PR 等高阶编程工作流尚未单列规格。
2. browser / container / remote coding host 的适配边界尚未收束为统一模型。
3. semantic search、tool search、embedded search 的统一抽象仍可进一步归并。
