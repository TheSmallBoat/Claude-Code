# 上下文装配与缓存前缀规格

## 文档状态

- 状态：Draft
- 日期：2026-04-03
- 对应 Rust 目标模块：`agent-context`

## 1. 目标

定义 Runtime Core 的上下文装配机制，使 Rust 基线版能够稳定复现：

1. system prompt 分层装配。
2. user/system context 提供与缓存。
3. custom prompt / append prompt 组合语义。
4. cache-safe params 快照与 fork 复用。

## 2. 非目标

本规格明确不覆盖：

1. session memory 与 compact 的详细规则。
2. 自动记忆提取策略。
3. MCP、skills、brief 等每个动态 section 的业务内容细节。

## 3. 设计原则

### 3.1 上下文是结构化对象，不是单个字符串

必须区分以下对象：

1. `systemPromptSections`
2. `userContext`
3. `systemContext`
4. `cacheSafeParams`

### 3.2 上下文 provider 可以缓存，但装配结果必须可重建

`getUserContext()` 和 `getSystemContext()` 在 TS 中是 memoized。Rust 版也应允许 provider 缓存，但必须保留显式重建能力，以支撑 resume、fork 和 mode change。

### 3.3 cache-safe params 是 fork 共享的协议对象

这不是普通调试快照，而是 prompt cache 共享的输入契约。Rust 版必须把它设计成一等对象。

## 4. ContextBundle 模型

建议定义：

```rust
pub struct ContextBundle {
    pub system_prompt_sections: Vec<PromptSection>,
    pub user_context: BTreeMap<String, String>,
    pub system_context: BTreeMap<String, String>,
    pub rendered_system_prompt: RenderedSystemPrompt,
}

pub struct PromptSection {
    pub key: String,
    pub content: String,
    pub cache_scope: CacheScope,
}

pub enum CacheScope {
    Global,
    Session,
    Uncached,
}
```

## 5. ContextProviders 规范

### 5.1 UserContextProvider

最小职责：

1. 读取 `CLAUDE.md` / memory files。
2. 提供当前日期等用户侧上下文。
3. 允许基于 bare mode / env flags 调整行为。

### 5.2 SystemContextProvider

最小职责：

1. 提供 git status 快照。
2. 提供 cache breaker 等系统注入项。
3. 允许在 remote / disabled-git-instructions 场景跳过重计算。

### 5.3 Session Guidance Provider

最小职责：

1. 根据 enabled tools、skills、runtime mode 生成 session-specific guidance。

## 6. System Prompt Section Registry

system prompt 必须先按 section 组织，再渲染成最终字符串数组。至少应支持：

1. 静态 section。
2. 动态 section。
3. uncached section。
4. dynamic boundary marker。

静态 section 是长期稳定可缓存部分，动态 section 是会话相关或条件相关部分。

## 7. System Prompt 组合顺序

基线顺序必须满足：

1. 先生成默认 system prompt sections。
2. 若 `custom_system_prompt` 存在，则替换默认主体。
3. 若 `custom_system_prompt` 且存在 auto-memory override，则追加 `memory_mechanics_prompt`。
4. 最后追加 `append_system_prompt`。

注意：

1. `custom_system_prompt` 是替换，不是附加。
2. `append_system_prompt` 是尾部附加。
3. `memory_mechanics_prompt` 是特例补丁层。

## 8. User/System Context 组合顺序

### 8.1 userContext

必须支持：

1. base user context provider 输出。
2. 宿主模式扩展项，例如 coordinator user context。

### 8.2 systemContext

基础 system context 来自 provider；当 `custom_system_prompt` 替换默认 system prompt 主体时，TS 实现会跳过默认 `getSystemContext()`，Rust 版应显式保留这一语义开关。

## 9. Dynamic Boundary 规范

必须保留一个明确的 boundary marker，把 system prompt sections 划分为：

1. 可走 global cache scope 的前缀。
2. 仅能按 session 或 turn 处理的动态后缀。

Rust 版不一定要把 marker 直接暴露给模型，但必须在内部保留该边界信息。

## 10. CacheSafeParams 模型

建议定义：

```rust
pub struct CacheSafeParams {
    pub rendered_system_prompt: RenderedSystemPrompt,
    pub user_context: BTreeMap<String, String>,
    pub system_context: BTreeMap<String, String>,
    pub tool_context_snapshot: ToolContextSnapshot,
    pub fork_context_messages: Vec<RuntimeMessage>,
}
```

它必须足以表达与 prompt cache 命中相关的关键维度：

1. system prompt
2. user context
3. system context
4. tools
5. model
6. thinking config
7. messages prefix

## 11. CacheSafe Snapshot Store

系统需要保存“最近一次可复用的 cache-safe params”，供 turn-end fork 行为使用。

建议定义：

```rust
pub trait CacheSafeSnapshotStore {
    fn save(&mut self, params: Option<CacheSafeParams>);
    fn load(&self) -> Option<CacheSafeParams>;
}
```

## 12. Fallback Context Builder

当正常快照不可用时，系统应提供一个 fallback builder，尽量重建与主线程接近的上下文前缀，而不是直接失败。

这对于：

1. side question
2. post-turn fork
3. resume 前的临时分析

都很重要。

## 14. 可直接转化的 Rust 编程任务

1. 定义 `PromptSection`、`CacheScope`、`ContextBundle`。
2. 实现 `UserContextProvider` 与 `SystemContextProvider`。
3. 实现 `SystemPromptSectionRegistry` 与 boundary 划分。
4. 实现 `ContextAssembler` 的默认/custom/append 组合语义。
5. 实现 `CacheSafeParams` 与最近快照存储。
6. 实现 fallback context builder。

## 15. 验收测试矩阵

至少应覆盖以下测试：

1. `custom_system_prompt` 会替换默认 system prompt 主体。
2. `append_system_prompt` 总是附加在最终 prompt 尾部。
3. `custom_system_prompt + auto-memory override` 时会注入 memory mechanics prompt。
4. `userContext` 可被宿主模式额外扩展而不影响基础 provider。
5. dynamic boundary 前后 cache scope 划分正确。
6. 保存的 `CacheSafeParams` 可供 fork 重用。
7. 快照缺失时 fallback builder 仍能生成可用上下文。

## 16. 当前已知缺口

本规格已经足够支撑 `agent-context` 的骨架实现，但仍需后续文档补齐：

1. session memory / nested memory 的详细注入机制。
2. system prompt section registry 的缓存失效与版本化策略。
3. mcp instructions delta 与 late-connect 场景。
4. brief / token budget / scratchpad 等动态 section 的长期演化规则。
