# 查询执行内核规格

## 文档状态

- 状态：Draft
- 日期：2026-04-03
- 对应 Rust 目标模块：`agent-query`、`agent-session-store`、`agent-context` 的交界层

## 1. 目标

定义一个与宿主界面解耦的查询执行内核，使其能够在 Rust 中复现当前 TS 实现最核心的能力：

1. 会话内连续回合执行。
2. 流式模型采样。
3. 工具调用与结果回注。
4. 压缩、恢复、预算、停止钩子的标准化执行阶段。
5. 可恢复、可持久化、可测试的 turn state machine。

## 2. 非目标

本规格明确不覆盖：

1. TUI/REPL 组件与视觉反馈。
2. CLI 命令文案与交互菜单。
3. 插件市场、渠道装配、品牌化能力。
4. Remote/Bridge/Coordinator 等分布式高级能力。

## 3. 代码证据范围

当前规格主要由以下代码逆向得到：

1. `TS-Source/QueryEngine.ts`
2. `TS-Source/query.ts`
3. `TS-Source/query/config.ts`
4. `TS-Source/query/deps.ts`
5. `TS-Source/query/stopHooks.ts`
6. `TS-Source/services/tools/toolOrchestration.ts`
7. `TS-Source/context.ts`
8. `TS-Source/bootstrap/state.ts`
9. `TS-Source/services/compact/autoCompact.ts`
10. `TS-Source/services/compact/microCompact.ts`

## 4. 设计原则

### 4.1 两层引擎分离

必须区分：

1. 会话级引擎 `ConversationEngine`。
2. 回合级执行器 `TurnExecutor`。

原因：TS 中 `QueryEngine` 与 `query()` 已经天然形成这两层语义，Rust 中应显式化，而不是做成单体对象。

### 4.2 单一事实源

当前实现始终围绕消息数组推进，工具结果、compact boundary、stop hook blocking error、附件和中断消息都被重新表达为消息。Rust 基线版必须延续这个策略。

### 4.3 恢复优先而非快速失败

Prompt-too-long、max_output_tokens、流式 fallback、token budget continuation 都表明该系统追求“尽可能把当前回合做完”。Rust 版不能退化成遇错即退出。

### 4.4 宿主接口后置

SDK、CLI、TUI 都只能通过抽象的 sink/adapter 影响内核，不得把宿主 UI 状态直接耦合进 `TurnExecutor`。

## 5. 核心对象模型

建议在 Rust 中定义如下核心对象：

```rust
pub struct ConversationEngine {
    pub history: Vec<Message>,
    pub total_usage: UsageTotals,
    pub permission_denials: Vec<PermissionDenial>,
    pub read_file_cache: ReadFileCache,
    pub transcript_store: Arc<dyn TranscriptStore>,
    pub runtime_handles: RuntimeHandles,
}

pub struct TurnRequest {
    pub prompt: PromptInput,
    pub query_source: QuerySource,
    pub fallback_model: Option<ModelRef>,
    pub max_turns: Option<u32>,
    pub task_budget: Option<TaskBudget>,
}

pub struct TurnState {
    pub messages: Vec<Message>,
    pub tool_context: ToolContext,
    pub auto_compact_tracking: Option<AutoCompactTracking>,
    pub max_output_tokens_recovery_count: u8,
    pub has_attempted_reactive_compact: bool,
    pub max_output_tokens_override: Option<u32>,
    pub pending_tool_use_summary: Option<PendingToolUseSummary>,
    pub stop_hook_active: bool,
    pub turn_count: u32,
    pub transition: Option<ContinueReason>,
}

pub enum ContinueReason {
    CollapseDrainRetry,
    ReactiveCompactRetry,
    MaxOutputTokensEscalate,
    MaxOutputTokensRecovery { attempt: u8 },
    StopHookBlocking,
    TokenBudgetContinuation,
}

pub enum TerminalReason {
    Completed,
    BlockingLimit,
    PromptTooLong,
    ImageError,
    AbortedStreaming,
    AbortedTools,
    HookStopped,
    StopHookPrevented,
    MaxTurnsReached,
    ModelError,
}
```

## 6. 运行时边界

### 6.1 ConversationEngine

职责：

1. 维护会话级历史。
2. 在进入 turn 前处理用户输入、system prompt 组装和 transcript 写入。
3. 消费 `TurnExecutor` 产出的消息流，并回写 usage、history、持久化存储。

### 6.2 TurnExecutor

职责：

1. 驱动单次回合 while-loop。
2. 在每次循环中进行上下文整形、模型调用、工具执行、stop hook、附件回注。
3. 决定继续还是结束。

### 6.3 ContextAssembler

职责：

1. 构造 `systemPrompt`、`userContext`、`systemContext`。
2. 管理 cache-safe prefix。
3. 为 memory injection 和 compact 提供基础上下文。

### 6.4 ToolScheduler

职责：

1. 对工具调用分批。
2. 执行并发批与串行批。
3. 管理上下文突变提交点。

### 6.5 StopHookRuntime

职责：

1. 在回合末执行 stop hooks。
2. 产生 blocking errors 或 continuation stop 信号。
3. 触发 turn-end 自动化能力。

## 7. 规范化执行阶段

每个 turn iteration 必须遵循以下阶段顺序：

1. 读取 `TurnState` 并建立 query tracking。
2. 从 compact boundary 后取有效消息视图。
3. 执行 tool result budget。
4. 执行 snip compact。
5. 执行 microcompact。
6. 执行 context collapse。
7. 执行 autocompact。
8. 调用模型流式采样。
9. 若无后续工具，则执行恢复链、stop hooks、token budget 检查。
10. 若存在工具，则执行工具调度、附件与预取回注。
11. 把新消息合并回 `TurnState.messages`，进入下一迭代或终止。

该顺序属于核心语义，Rust 版不得随意交换。

## 8. 模型采样与恢复规则

### 8.1 模型驱动接口

`TurnExecutor` 不直接依赖具体 API 实现，而是依赖如下抽象：

```rust
pub trait ModelDriver {
    fn stream(&self, request: ModelRequest) -> ModelStream;
}
```

`ModelStream` 需要产出：

1. assistant message block
2. stream event
3. synthetic api error message

### 8.2 Prompt-too-long 恢复顺序

必须严格按以下顺序：

1. 先尝试 collapse drain。
2. 再尝试 reactive compact。
3. 再失败才真正对外暴露错误。

### 8.3 max_output_tokens 恢复顺序

必须支持：

1. 上限升级重试。
2. 注入 continuation meta message 后直接续跑。
3. 限次恢复，避免死循环。

### 8.4 fallback model

当主模型触发 fallback 时：

1. 必须丢弃当前尝试产生的 orphaned tool results。
2. 必须清空该次 assistant/tool state。
3. 必须以 fallback model 从同一 turn 重新执行。

## 9. 工具调度规则

### 9.1 分批规则

工具调度必须先按 `is_concurrency_safe(input)` 分批：

1. 连续只读且并发安全的工具进入并发批。
2. 非并发安全工具单独形成串行批。

### 9.2 上下文突变提交

对并发批：

1. 工具执行期间允许产生 context modifier。
2. modifier 不能立即修改共享上下文。
3. 必须在整批完成后，按工具原始顺序统一提交。

对串行批：

1. 每个工具完成后可立即更新上下文。

这是 Rust scheduler 的硬约束。

### 9.3 中断语义

如果执行期间收到 abort：

1. 必须为尚未配对的 tool_use 生成 synthetic tool_result 或等价结束信号。
2. 必须终止继续迭代。
3. 必须保留 transcript 一致性。

## 10. 停止钩子规则

stop hook 不是附加插件，而是 query 内核阶段：

1. 只在模型成功完成且没有待执行工具时触发。
2. 可产生 blocking errors，把回合重新送入下一次循环。
3. 可阻止 continuation。
4. API error 结尾的回合不得继续执行 stop hooks，防止 error-hook death spiral。

Rust 中建议定义：

```rust
pub trait StopHookRunner {
    fn run(&self, input: StopHookInput) -> StopHookOutcome;
}
```

## 11. 预算与压缩规则

### 11.1 Token budget

软预算控制必须满足：

1. 仅对主线程 turn 生效。
2. 在预算还未接近上限时，允许自动注入 continuation nudge message。
3. 连续继续但增量产出很低时，应提前停止，避免低收益消耗。

### 11.2 Autocompact

必须具备：

1. 阈值判断。
2. 连续失败断路器。
3. 成功后重建 post-compact messages。
4. 对 task budget remaining 的联动扣减。

### 11.3 Microcompact

微压缩属于前置轻量阶段，用于在不触发大规模总结的情况下清理上下文噪音。Rust 版应保留这一层，而不是只保留 autocompact。

## 12. 会话持久化要求

查询执行内核必须依赖 transcript 存储，但不能被具体文件格式绑死。

最低要求：

1. 用户消息在首次模型请求前必须写入存储。
2. assistant/progress/attachment/compact boundary 必须保持顺序化追加。
3. compact boundary 之前需要支持补写尾段。
4. 宿主可选择 lazy flush，但顺序不能错乱。

## 13. Rust 实现建议拆分

### 13.1 crate 或模块建议

1. `agent-query/core`
   - `TurnExecutor`
   - `TurnState`
   - `ContinueReason` / `TerminalReason`
2. `agent-query/model`
   - `ModelDriver`
   - `ModelStreamReducer`
3. `agent-query/hooks`
   - `StopHookRunner`
4. `agent-query/budget`
   - token budget policy
   - max output continuation policy
5. `agent-query/compaction`
   - microcompact policy
   - autocompact policy
   - reactive compact policy

### 13.2 需要外部注入的 trait

1. `ModelDriver`
2. `ToolScheduler`
3. `ContextAssembler`
4. `TranscriptStore`
5. `StopHookRunner`
6. `IdGenerator`
7. `Clock`

## 14. 可直接转化的 Rust 编程任务

1. 定义 `Message`、`TurnState`、`ContinueReason`、`TerminalReason` 等核心数据结构。
2. 实现 `ConversationEngine::submit()`，负责 turn 入口装配与 transcript 预写。
3. 实现 `TurnExecutor::run()` 的 while-loop 骨架与阶段调度。
4. 接入 `ModelDriver` 的流式 reducer，并支持 withheld error 处理。
5. 实现 `ToolScheduler` 的分批、并发、串行与 context modifier 提交逻辑。
6. 实现 `StopHookRunner` 与 `StopHookOutcome`。
7. 实现 `TokenBudgetPolicy`、`MaxOutputRecoveryPolicy`、`ReactiveCompactPolicy`。
8. 补齐 transcript 顺序化写入和 compact boundary 补写。

## 15. 验收测试矩阵

至少应覆盖以下测试：

1. 正常单轮完成，无工具调用。
2. 模型输出工具调用，工具结果回注后继续第二轮。
3. 并发安全工具与非并发安全工具的批次切分正确。
4. Prompt-too-long 时先 collapse，再 reactive compact，再失败返回。
5. max_output_tokens 时先升级输出上限，再执行 continuation recovery。
6. abort 发生在流式阶段时，不遗留未配对 tool_use。
7. abort 发生在工具阶段时，回合能有序结束。
8. token budget continuation 与 diminishing returns stop 生效。
9. compact boundary 前的 transcript 补写可支撑 resume。

## 16. 当前已知缺口

本规格已经足够支撑 `agent-query` 骨架编码，但仍依赖后续文档补齐以下内容：

1. 消息 IR 与 transcript 精确模型。
2. 权限系统接口与决策规则。
3. 子代理与 forked agent 的隔离契约。
4. Session memory 与 compaction 的联动规格。
5. 编程工具能力层的统一接口和副作用分类规范。
