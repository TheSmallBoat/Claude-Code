# 后台任务与子代理规格

## 文档状态

- 状态：Draft
- 日期：2026-04-03
- 对应 Rust 目标模块：`agent-tasks`、`agent-subagents`

## 1. 目标

定义 Runtime Core 的任务与子代理运行时，使 Rust 基线版能够稳定复现：

1. 可观察的后台任务控制面。
2. 全功能子代理的隔离执行与恢复。
3. 内部 forked agent 的轻量执行模型。
4. sidechain transcript 驱动的 resume 机制。

## 2. 非目标

本规格明确不覆盖：

1. TUI 面板和背景任务对话框的展示细节。
2. workflow / dream / remote agent 的业务功能细节。
3. MCP、worktree、handoff classifier 的产品策略细节。

## 3. 设计原则

### 3.1 task 是控制面记录，不是执行器

Task registry 只负责：

1. 生命周期状态
2. 输出文件与通知
3. 观察与回收

实际 query loop 执行必须由独立 runner 负责。

### 3.2 forked agent 与 subagent 必须分离

至少要区分：

1. ephemeral/internal forked agent
2. user-visible/resumable subagent
3. backgrounded main-session query

这三者不能合并成一种“agent task”。

### 3.3 sidechain transcript 是恢复主轴

resume 必须依赖 transcript + metadata，而不是只依赖内存中的 task state。

### 3.4 状态切换必须先于昂贵后处理

后台 agent 到达 terminal state 时，必须先把 task state 变为 terminal，再跑 classifier、git cleanup、通知 embellishment 等昂贵逻辑。

### 3.5 并发 agent 需要 execution-local identity

当多个后台 agent 并发运行时，不能依赖共享全局状态标识当前 agent，必须使用 execution-local context。

## 4. 总体拆分

建议把这一层拆为四个子系统：

1. `TaskRegistry`
2. `ForkedAgentRunner`
3. `SubagentRuntime`
4. `SidechainStore`

其中：

1. `TaskRegistry` 面向 UI/SDK/notification。
2. `ForkedAgentRunner` 面向内部自动化。
3. `SubagentRuntime` 面向可见 agent 和 background session。
4. `SidechainStore` 面向 transcript / metadata / resume。

## 5. 核心数据模型

建议定义：

```rust
pub enum TaskStatus {
    Pending,
    Running,
    Completed,
    Failed,
    Killed,
}

pub enum TaskKind {
    LocalShell,
    LocalAgent,
    RemoteAgent,
    Workflow,
    Monitor,
    Dream,
    MainSessionBackground,
}

pub struct TaskRecord {
    pub id: TaskId,
    pub kind: TaskKind,
    pub status: TaskStatus,
    pub description: String,
    pub tool_use_id: Option<String>,
    pub start_time: Instant,
    pub end_time: Option<Instant>,
    pub output_file: PathBuf,
    pub output_offset: u64,
    pub notified: bool,
}

pub enum AgentRunMode {
    Foreground,
    Background,
    EphemeralFork,
    BackgroundedMainSession,
}

pub struct AgentTaskRecord {
    pub base: TaskRecord,
    pub agent_id: AgentId,
    pub agent_type: String,
    pub run_mode: AgentRunMode,
    pub is_backgrounded: bool,
    pub retain: bool,
    pub disk_loaded: bool,
    pub pending_messages: Vec<String>,
    pub progress: Option<AgentProgress>,
}

pub struct AgentRuntimeIdentity {
    pub agent_id: AgentId,
    pub parent_session_id: Option<SessionId>,
    pub execution_kind: AgentExecutionKind,
    pub subagent_name: Option<String>,
    pub invocation_kind: Option<InvocationKind>,
    pub invoking_request_id: Option<String>,
}
```

## 6. TaskRegistry 规范

### 6.1 职责

负责：

1. 注册任务
2. 更新任务状态
3. 合并 resume/replacement 场景的 UI-held 状态
4. terminal task eviction
5. 生成 task_started / task_notification 事件

### 6.2 关键语义

1. replacement register 必须保留 `retain`、`messages`、`disk_loaded`、`pending_messages`。
2. terminal task 只有在 `notified=true` 且 grace 期到期后才能回收。
3. task framework 本身不必统一所有完成通知逻辑，允许任务类型自带 terminal notification。

## 7. Task Output Store 规范

每个任务必须有稳定的 `output_file` 入口。

最低要求：

1. 输出目录按 session 隔离。
2. 目录路径在 session 生命周期内稳定，不能因 `/clear` 或 sessionId 再生成而漂移。
3. 输出支持增量读取与 size cap。
4. agent/main-session 任务允许把输出映射为 transcript symlink。

## 8. SidechainStore 规范

### 8.1 职责

为每个可恢复 agent 或 background session 维护：

1. 独立 transcript 文件
2. metadata 文件
3. sidechain message chain
4. resume 所需的 content replacement records

### 8.2 关键语义

1. initial messages 必须在 agent query loop 开始前写入 sidechain transcript。
2. 后续 message 应增量写入，并显式携带 `starting_parent_uuid`。
3. metadata 至少包含 `agent_type`、可选 `worktree_path`、可选 `description`。

## 9. AgentContextScope 规范

系统必须有 execution-local 的 agent identity 存储，至少能表达：

1. 当前 agentId
2. 上级 session/agent 关系
3. spawn 还是 resume
4. invoking request id

这层不能用共享 `AppState` 代替。

## 10. ForkedAgentRunner 规范

### 10.1 适用范围

用于内部自动化 fork，例如：

1. session memory extraction
2. compact summarize
3. 提示建议、内部技能或命令执行

### 10.2 关键要求

1. 必须接受 `CacheSafeParams`，以共享主线程 prompt cache。
2. 必须克隆或隔离所有可变上下文状态。
3. 可以选择不写 transcript。
4. 需要独立累计 usage。

建议接口：

```rust
pub trait ForkedAgentRunner {
    async fn run(&self, params: ForkedAgentParams) -> ForkedAgentResult;
}
```

## 11. SubagentRuntime 规范

### 11.1 职责

全功能 subagent runner 负责：

1. agent-specific prompt/context 构建
2. tool pool 解析与权限模式覆盖
3. skills / hooks / MCP client 生命周期
4. query loop 驱动
5. sidechain transcript / metadata 写入
6. finally cleanup

### 11.2 工具与权限隔离

必须支持：

1. 按 agent permission mode 重建工具池。
2. async agents 默认使用更严格的工具子集。
3. `allowed_tools` 能覆盖 session allow rules，但要保留 CLI/SDK 显式下发的更高层许可。
4. async agents 默认避免权限弹窗；sync/bubble agents 可继承父交互通道。

## 12. Async Agent Lifecycle 规范

后台 agent 生命周期必须经过统一外壳：

1. 启动可选 summarization sidecar。
2. 持续消费 subagent query stream。
3. 更新 progress tracker 与 task.progress。
4. 到达 terminal 时先切 `TaskStatus`，再做昂贵后处理。
5. 发出 terminal notification。
6. 释放 scoped skills、dump state 等副作用缓存。

建议接口：

```rust
pub trait AsyncAgentLifecycle {
    async fn run(&self, task_id: TaskId, stream: AgentStream) -> TerminalOutcome;
}
```

## 13. Foreground-to-Background Handoff 规范

同步 agent 需要支持中途 background。

正确语义应为：

1. 前台阶段先注册一个 foreground task record。
2. 主循环同时等待 `next_message` 与 `background_signal`。
3. background 发生时，先关闭 foreground iterator，使其 finally 执行。
4. 再用同一 `agent_id` 与新的 background shell 继续运行。
5. 对外立即返回 `async_launched`。

Rust 版不能简单用“把同一个 future 放后台继续跑”替代，因为资源清理和 context ownership 会出错。

## 14. MainSessionBackgroundRunner 规范

主会话后台化是单独的运行形态，不应在 Rust 中继续伪装成普通 local agent。

最低要求：

1. 使用独立 `task_id` 与 sidechain transcript。
2. 在背景化时先把当前 conversation snapshot 写入 sidechain transcript。
3. 后台 query 必须在 isolated transcript 中继续，不得写回主 transcript。
4. 需要支持 `foregrounded_task_id` 这类“切换主视图”语义。

## 15. ResumeService 规范

resume 入口必须做以下工作：

1. 读取 sidechain transcript + metadata。
2. 清理无法直接送回 API 的坏消息形态。
3. 重建 content replacement state。
4. 依据 metadata 恢复 worktree path、description、agent type。
5. 复用原 agentId 重新注册 task 并重启 async lifecycle。

特别规则：fork agent resume 需要恢复 parent cache-identical system prompt 与 tools；普通 agent resume 则按当前环境重建。

## 16. Progress / Summary / Notification 三分模型

必须明确分离三种东西：

1. progress stream：细粒度工具活动、token/tool-use 增长。
2. summary field：后台 summarizer 的短摘要，不应被普通 progress 覆盖。
3. terminal notification：单次完成/失败/停止通知。

这三者不能合并成一个 `status_message` 字段。

## 17. Cleanup Contract 规范

每个 subagent finally 至少必须清理：

1. agent-specific MCP clients
2. agent-scoped hooks
3. prompt-cache tracking state
4. cloned file-state cache
5. transcript subdir mapping
6. agent-scoped todos
7. 该 agent 派生出的 shell/monitor 任务

### 18.1 `agent-tasks`

1. `registry.rs`
2. `state.rs`
3. `output_store.rs`
4. `notifications.rs`
5. `eviction.rs`

### 18.2 `agent-subagents`

1. `fork_runner.rs`
2. `subagent_runner.rs`
3. `async_lifecycle.rs`
4. `background_session.rs`
5. `resume.rs`
6. `identity_scope.rs`
7. `sidechain_store.rs`

## 19. 可直接转化的 Rust 编程任务

1. 定义 `TaskKind`、`TaskStatus`、`TaskRecord`、`AgentTaskRecord`。
2. 实现 `TaskRegistry`，支持 register/update/evict/notify。
3. 实现 `TaskOutputStore`，支持稳定路径、增量偏移和 transcript symlink 模式。
4. 实现 `AgentContextScope`，为并发 agent 提供 execution-local identity。
5. 实现 `ForkedAgentRunner`，支持 cache-safe fork、isolated context、optional transcript。
6. 实现 `SubagentRuntime`，支持 prompt/context/tools/hooks/MCP 生命周期管理。
7. 实现 foreground-to-background handoff，保证相同 agentId 下的连续运行。
8. 实现 `MainSessionBackgroundRunner`，把主 query 后台化为 sidechain task。
9. 实现 `ResumeService`，从 transcript + metadata 恢复 agent。
10. 为 progress stream、summary update、terminal notification 建立三条独立通道。

## 20. 验收测试矩阵

至少应覆盖以下测试：

1. task replacement register 会保留 `retain/messages/disk_loaded/pending_messages`。
2. async agent 与主线程并发运行时，execution-local identity 不串号。
3. `runForkedAgent()` 修改 read-file state 不会污染 parent context。
4. `runAgent()` 会先写 initial messages 与 metadata，再进入 query loop。
5. sidechain transcript 的逐条写入能正确维持 parent chain continuity。
6. foreground agent 触发 background 后，会以同一 agentId 继续执行。
7. background main-session query 使用独立 transcript，因此 `/clear` 后仍可继续。
8. async agent completion 会先把 task 置为 terminal，再做 classifier/worktree cleanup。
9. resume 会重建 content replacement state，而不是重复发送旧 forkContextMessages。
10. async agent finally 会回收 agent-scoped hooks、todos、shell tasks 和 MCP clients。

## 21. 当前已知缺口

本规格已经足以支撑 `agent-tasks` 与 `agent-subagents` 的骨架设计，但仍需后续文档继续补齐：

1. `remote_agent`、`workflow`、`dream`、`monitor_mcp` 的实现级行为尚未逐个审计。
2. in-process teammate / swarm 的特殊协同语义仍未单列规格。
3. 任务输出文件的 shell/file 双后端实现还未单独抽成规格书。
