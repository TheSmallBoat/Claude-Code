# 高阶编程工作流补遗

## 文档状态

- 状态：Supplement Draft
- 日期：2026-04-03
- 对应 Rust 目标模块：`agent-code-workflows`

## 1. 目标

补足 `07-编程领域能力适配层规格` 中尚未展开的高阶工作流层，重点说明：

1. review / ultrareview
2. commit / commit-push-pr
3. verify
4. remote trigger / scheduled remote agents
5. ultraplan 一类远端规划流

这些能力如何建立在通用 adapters 之上，以及哪些仍然属于宿主/平台扩展而非 Runtime Core。

## 3. 第一判断：高阶工作流是“编排脚本层”，不是新的 Runtime Core 原语

这些工作流虽然直接面向用户，但实现模式高度一致：

1. 以 `PromptCommand` 或 `local-jsx` 命令作为入口。
2. 通过 prompt 文本预先组织多步操作顺序。
3. 借助 `allowedTools`、shell interpolation、远端 precondition 检查来约束执行。
4. 在需要长期运行时，转而委托给 `remote_agent` task 或其他后台执行形态。

因此 Rust 版不应把 review/commit/PR 本身做成 core crate，而应把它们建模为 `workflow descriptors + launchers`。

## 4. 工作流家族

### 4.1 本地仓库操作工作流

代表：`/commit`、`/commit-push-pr`

共同特征：

1. 入口都是 prompt 型命令。
2. 通过 `executeShellCommandsInPrompt()` 先把当前仓库状态、diff、branch、`gh pr view` 结果嵌入 prompt。
3. 明确声明一组受限 `allowedTools`，主要是 git / gh CLI 和少量协作工具。
4. 让模型在单回合内完成“分析状态 -> 生成提交文案 -> 调用 shell/tool”。

这类工作流的本质不是“新增 git adapter”，而是在 git adapter 之上附着一套 prompt protocol 和 safety protocol。

### 4.2 本地代码审查工作流

代表：`/review`

它本质上仍是本地 prompt workflow：

1. 通过 `gh pr list/view/diff` 获取上下文。
2. 让模型输出 review 结论。
3. 不引入新的 task 类型，也不要求远端环境。

因此它是最纯粹的“adapter orchestration workflow”。

### 4.3 远端代码审查工作流

代表：`/ultrareview`

其关键变化不是 prompt 内容，而是 launch path：

1. `/ultrareview` 是 `local-jsx` 入口，用于承载 overage / 资格确认界面。
2. 真正执行时会调用 `launchRemoteReview()`。
3. 该函数先检查 remote session preconditions，再 teleport 到远端环境。
4. 成功后注册 `RemoteAgentTask`，由本地 poller 把结果桥回当前 session。

因此 `ultrareview` 不是单纯“更强 review prompt”，而是一个跨本地会话与远端 CCR session 的复合工作流。

### 4.4 验证工作流

代表：bundled `/verify` skill

该工作流的特点是：

1. 不是一个新工具，而是一段高密度 skill 文本。
2. 正文由 bundled `SKILL.md` 与示例文档组成。
3. 目标是把“运行应用、验证效果、修正失败”这一类经验流程标准化。

这表明很多所谓“高阶编程工作流”在实现上其实只是 prompt catalog，而不是独立 runtime 组件。

### 4.5 远端计划工作流

代表：`/ultraplan`

关键特征：

1. 默认 instructions 来自 `utils/ultraplan/prompt.txt`。
2. ant-only 环境变量可以整体覆盖该 prompt 文本。
3. 实际执行依赖远端会话、浏览器审批与 `RemoteAgentTask` 的状态追踪。

其价值在于证明：高阶工作流既可能由 markdown 资产驱动，也可能由 `.txt` 资产驱动；真正稳定的是 launch protocol。

### 4.6 计划内远端自动化工作流

代表：`/schedule` skill + `RemoteTriggerTool`

这里体现了另一种更平台化的 workflow：

1. workflow 文本负责引导用户配置 cron、repo、environment、connectors。
2. 真正的动作交给 `RemoteTriggerTool` 调后端 API 完成 `list/get/create/update/run`。
3. 触发器最终生成“以后再启动远端 agent”的自动化计划，而不是立即在当前主对话中展开完整 coding loop。

这类 workflow 更接近“编程平台自动化编排层”。

## 5. 统一抽象

建议把高阶工作流统一建模为：

```rust
pub enum WorkflowExecutionKind {
    InlinePrompt,
    LocalToolOrchestration,
    ForkedAgent,
    RemoteSession,
    ScheduledRemote,
}

pub struct WorkflowDescriptor {
    pub name: String,
    pub description: String,
    pub execution_kind: WorkflowExecutionKind,
    pub required_tools: Vec<String>,
    pub required_capabilities: Vec<String>,
    pub prompt_asset_ids: Vec<String>,
    pub preconditions: Vec<WorkflowPrecondition>,
}
```

其中 `execution_kind` 比“命令名”更稳定，因为它直接描述 runtime 需要走哪条执行管线。

## 6. 最小运行协议

无论是 commit、review 还是 remote session workflow，都建议复用以下协议：

1. 先做 precondition 检查。
2. 再收集工作区 / repo / auth / remote environment 上下文。
3. 装配 workflow prompt 资产。
4. 根据 `execution_kind` 选择：
   1. 直接注入主对话。
   2. 转为 forked agent。
   3. 注册 remote task 并异步桥回结果。
   4. 调平台 API 创建未来触发器。
5. 把最终结果或通知通过 transcript/task notification 反馈给主会话。

## 7. 与 task plane 的耦合点

### 7.1 本地工作流通常不需要新 task 类型

`commit`、`review`、`verify` 这类工作流主要停留在：

1. prompt 注入
2. tool orchestration
3. 单回合或少量回合主线程执行

### 7.2 远端工作流必须落到 task plane

`ultrareview`、`ultraplan`、scheduled remote 这类工作流会跨越主会话生命周期，因此必须借助：

1. `remote_agent` task
2. output file / task notification
3. sidecar metadata
4. resume 恢复

这说明 workflow 层和 task 层的边界是：

1. workflow 负责“定义怎么发起”。
2. task 负责“发起后如何持续观察和恢复”。

## 8. 对 Rust 版的建议

### 8.1 保持 workflow 与 adapters 解耦

建议把 workflow 实现为独立目录或 crate：

1. `review_workflow`
2. `commit_workflow`
3. `verification_workflow`
4. `remote_plan_workflow`
5. `scheduled_remote_workflow`

它们只能依赖：

1. repo adapter
2. shell/tool adapter
3. prompt asset assembler
4. task launcher

### 8.2 不要把具体 prompt 文案写死在 core

workflow 应依赖 prompt asset ids，而不是直接依赖静态字符串常量。

### 8.3 远端工作流要做成 host extension

因为它们依赖：

1. OAuth
2. remote environment
3. teleport / CCR
4. GitHub app / repo access

这些都不属于基线 runtime core。

## 9. 已知不完整之处

当前 restored tree 有几类能力不能当作成熟主链抽象：

1. `commands/workflows/` 在恢复树中缺失，说明 workflow scripts 并非当前稳定主轴证据。
2. browser 相关运行时在当前树中没有形成统一的 workflow 模块，不能在 Rust 版中预设成熟接口。
3. 很多平台化远端能力更像 host product extension，而不是 coding runtime 本体。

因此这份补遗的作用是“限定边界”，而不是把所有产品工作流都纳入 Phase 1。

## 10. 最小测试矩阵

1. 本地 git workflow 应验证 prompt 前置状态采集与 allowed-tools 绑定。
2. local review 与 remote review 应验证 launch path 分流是否正确。
3. remote workflow 应验证 precondition 失败时返回可注入的用户可见解释，而不是直接崩溃。
4. scheduled remote workflow 应验证 prompt 资产与 API tool 的组合是否稳定。
5. workflow 不存在或 feature 缺失时，应 fail-closed 为“不提供该工作流”，而不是暴露空壳命令。

## 11. 结论

高阶编程工作流不是 Runtime Core 的新层级，而是：

1. prompt asset catalog
2. adapter orchestration
3. optional task launcher
4. host/platform capability gate

把这四者分开，Rust 版就能在不复制现有产品外壳的前提下，保留真正可迁移的 workflow 语义。
