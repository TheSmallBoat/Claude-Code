# 任务扩展类型补遗

## 文档状态

- 状态：Supplement Draft
- 日期：2026-04-03
- 对应 Rust 目标模块：`agent-tasks`、`agent-subagents`、可选扩展 `agent-remote-tasks`

## 1. 目标

补足 `06-后台任务与子代理规格` 中尚未细化的 task 扩展类型，明确当前恢复树里：

1. 哪些 task 变体已经有实装语义。
2. 哪些只是保留的扩展槽位或 stub。
3. Rust 版应该如何在保持可扩展的同时避免把 stub 误当成熟能力。

## 2. 证据源

本补遗主要基于以下代码：

1. `TS-Source/tasks/types.ts`
2. `TS-Source/tasks/RemoteAgentTask/RemoteAgentTask.tsx`
3. `TS-Source/tasks/InProcessTeammateTask/types.ts`
4. `TS-Source/tasks/DreamTask/DreamTask.ts`
5. `TS-Source/tasks/LocalShellTask/guards.ts`
6. `TS-Source/tasks/LocalWorkflowTask/LocalWorkflowTask.ts`
7. `TS-Source/tasks/MonitorMcpTask/MonitorMcpTask.ts`
8. `TS-Source/utils/background/remote/remoteSession.ts`

## 3. 当前 task 联合类型

`tasks/types.ts` 给出的 `TaskState` / `BackgroundTaskState` 联合类型为：

1. `LocalShellTaskState`
2. `LocalAgentTaskState`
3. `RemoteAgentTaskState`
4. `InProcessTeammateTaskState`
5. `LocalWorkflowTaskState`
6. `MonitorMcpTaskState`
7. `DreamTaskState`

这说明 task plane 在现有代码里已经明确是“开放联合”，而不是只有 local shell / local agent 两种。

## 4. 已有实装的扩展类型

### 4.1 `local_bash`

`LocalShellTaskState` 的关键字段表明它不只是一个 subprocess handle：

1. `type: 'local_bash'`
2. `isBackgrounded`
3. `agentId`
4. `kind?: 'bash' | 'monitor'`
5. `lastReportedTotalLines`
6. `completionStatusSentInAttachment`

含义是：

1. 同一个底层 task family 已覆盖普通 bash 和 monitor 两种 UI / 语义变体。
2. task 可以归属于某个 agent，而不只是主线程。
3. 背景化与输出增量跟踪已经内建在该 state shape 中。

### 4.2 `remote_agent`

这是当前恢复树里最完整的扩展任务类型。其 state 明确携带：

1. `remoteTaskType`
2. `sessionId`
3. `command` / `title`
4. `todoList` / `log`
5. `isLongRunning`
6. `pollStartedAt`
7. `isRemoteReview` / `reviewProgress`
8. `isUltraplan` / `ultraplanPhase`
9. `remoteTaskMetadata`

还具备以下关键行为：

1. sidecar metadata 持久化到 session storage。
2. `restoreRemoteAgentTasks()` 在 resume 后重建本地 task 并重新开始 polling。
3. `pollRemoteSessionEvents()` 驱动远端状态同步。
4. completion checker 允许按 `remoteTaskType` 插入特定完成判定。

因此 `remote_agent` 已经不是普通 task subtype，而是一个小型“远端会话观察运行时”。

### 4.3 `in_process_teammate`

`InProcessTeammateTaskState` 的关键字段包括：

1. `identity` 子对象
2. `awaitingPlanApproval`
3. `permissionMode`
4. `messages`
5. `pendingUserMessages`
6. `isIdle`
7. `shutdownRequested`
8. `onIdleCallbacks`
9. `lastReportedToolCount` / `lastReportedTokenCount`

再结合 `TEAMMATE_MESSAGES_UI_CAP` 可以看出：

1. 这是同进程 teammate/worker 语义，不是 forked external runner。
2. 它强调 identity、plan approval、mailbox/zoom transcript、idle lifecycle。
3. task state 只是 UI mirror；完整执行上下文还依赖运行时侧的 execution-local state。

### 4.4 `dream`

`DreamTaskState` 不是另一个通用 agent runtime，而是 auto-dream 的可视化 task surface：

1. `phase: 'starting' | 'updating'`
2. `sessionsReviewing`
3. `filesTouched`
4. `turns`
5. `priorMtime`

它的职责更偏向：

1. 让后台 consolidation 子代理在 footer/task dialog 中可见。
2. 为 kill 提供 lock rollback。
3. 保留最近若干 turn 供用户查看。

因此 `dream` 是“对既有后台运行的 UI surfacing”，而不是一条独立的执行主链。

## 5. 当前仅有 stub 或占位的扩展类型

### 5.1 `local_workflow`

当前 `LocalWorkflowTaskState = Record<string, unknown>`，`isLocalWorkflowTask()` 恒为 `false`。

这说明：

1. runtime 想为本地 workflow 预留 task family。
2. 但恢复树里尚无可落地的状态形状和执行逻辑。

### 5.2 `monitor_mcp`

当前 `MonitorMcpTaskState` 同样只是占位类型，type guard 恒为 `false`。

这说明：

1. MCP monitor 被设计为潜在 task family。
2. 但目前没有足够证据将其视作 Rust 基线版必须实现的成熟功能。

## 6. 设计含义

### 6.1 task kind 必须保持开放式扩展

Rust 版不能把 task plane 收缩回固定的两三个变体。

至少需要支持：

1. 基线 task families
2. 远端扩展 task families
3. 同进程协作扩展 task families
4. UI surfacing-only task families

### 6.2 但不能把所有扩展都并入基线核心

当前证据表明更合理的拆法是：

1. `agent-tasks` 保存 registry、status、output、notification 等通用框架。
2. `agent-remote-tasks` 提供 `remote_agent` 相关实现。
3. `agent-team-runtime` 提供 `in_process_teammate`。
4. `dream` 之类 UI surfacing task 可放在宿主层或 feature crate。

### 6.3 stub 类型应明确标记为保留槽位

对于 `local_workflow`、`monitor_mcp`，Rust 版的正确策略不是胡乱补全，而是：

1. 在类型系统里保留 extension slot。
2. 在注册表中允许后续实现注入。
3. 在 Phase 1 默认不启用。

## 7. Rust 建模建议

建议把 task 种类拆成“核心种类 + 扩展种类”：

```rust
pub enum CoreTaskKind {
    LocalShell,
    LocalAgent,
}

pub enum TaskExtensionKind {
    RemoteAgent,
    InProcessTeammate,
    Dream,
    LocalWorkflow,
    MonitorMcp,
}

pub struct TaskRecord {
    pub id: TaskId,
    pub status: TaskStatus,
    pub core_kind: Option<CoreTaskKind>,
    pub extension_kind: Option<TaskExtensionKind>,
    pub output_file: PathBuf,
    pub notified: bool,
}
```

或者更进一步，使用扩展注册表：

```rust
pub trait TaskExtension {
    fn kind(&self) -> &'static str;
    fn can_restore(&self) -> bool;
    fn on_register(&self, record: &mut TaskRecord);
    fn on_kill(&self, record: &TaskRecord) -> TaskKillOutcome;
}
```

## 8. 每类扩展的最低不变量

### 8.1 `remote_agent`

1. 必须有 sidecar metadata。
2. 必须可 resume。
3. 必须有 polling loop 或等价的远端状态同步器。
4. terminal 通知必须能桥回主会话。

### 8.2 `in_process_teammate`

1. 必须有 execution-local identity。
2. 必须支持 idle / shutdown / pending user messages。
3. UI mirror transcript 必须与真实运行时 transcript 分离。

### 8.3 `dream`

1. 只能视为 UI surface。
2. kill 时必须能做 lock rollback。
3. 不要求成为独立的通用 agent execution primitive。

### 8.4 `local_bash` 的 `monitor` 变体

1. `kind` 必须保留，因为 monitor 不是单独 task type，而是同一家族的显示和语义分支。
2. 不能因为名字不同就拆成第二套 shell runtime。

## 9. 恢复与持久化建议

当前证据表明：

1. 不是所有扩展任务都需要完整持久化。
2. 需要恢复的关键是 `remote_agent`。
3. `in_process_teammate` 更偏会话内存态与运行态耦合，不宜盲目跨会话恢复。
4. `dream` 更接近短生命周期 UI task，不宜承诺强恢复语义。

所以 Rust 版应把“可恢复性”做成 task capability，而不是所有 task 的默认属性。

## 10. 最小测试矩阵

1. `remote_agent` 应验证 metadata 写入、resume 重建、polling 完成、404 清理。
2. `in_process_teammate` 应验证 identity、plan approval、idle callback、message cap。
3. `dream` 应验证 phase 切换、filesTouched 聚合、kill rollback。
4. `local_bash` 应验证 `bash` 与 `monitor` 两种 kind 的共用框架。
5. `local_workflow` / `monitor_mcp` 应验证默认未实现时不会误判为有效 task。

## 11. 结论

当前恢复树中的 task plane 已经证明：

1. task 不是单一执行器，而是一个开放的控制面注册表。
2. `remote_agent` 和 `in_process_teammate` 是真正的扩展任务家族。
3. `dream` 是 UI surfacing 型扩展。
4. `local_workflow` 和 `monitor_mcp` 目前仍应视为保留槽位。

Rust 基线版应该保留这种开放扩展能力，但只实现有充足证据的家族，不要把 stub 硬凑成主链能力。
