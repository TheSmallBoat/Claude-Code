# 压缩与记忆子系统规格

## 文档状态

- 状态：Draft
- 日期：2026-04-03
- 对应 Rust 目标模块：`agent-compact`、`agent-memory`

## 1. 目标

定义 Runtime Core 的长上下文治理子系统，使 Rust 基线版能够稳定复现：

1. 多阶段上下文瘦身策略。
2. session memory 的后台提取与 compaction 消费。
3. compact 结果的结构化重建。
4. compact 后共享状态清理与 transcript 连续性恢复。

## 2. 非目标

本规格明确不覆盖：

1. compact prompt 的最终文案细节。
2. context collapse 的完整实现细节。
3. UI 上如何展示 compact summary。
4. 具体 memory 文件内容模板的产品化演化。

## 3. 设计原则

### 3.1 先做便宜瘦身，再做昂贵摘要

长上下文治理必须优先尝试：

1. tool result budget replacement
2. microcompact
3. 其他投影式折叠
4. 最后才是 full compact / reactive compact

### 3.2 compact 后必须恢复“工作集”，而不是只保留 summary

compact 的输出必须能恢复：

1. 最近保留消息
2. 最近读过的关键文件
3. 运行中的后台任务状态
4. 计划文件、技能内容、增量工具说明

### 3.3 transcript 连续性靠边界元数据维持

compact boundary 不是展示消息，而是持久化与恢复的结构锚点。

### 3.4 记忆分层必须分开建模

至少区分：

1. nested memory
2. relevant memories
3. session memory

三者不能被合并成单一 memory subsystem，否则职责会混乱。

## 4. 总体阶段顺序

建议定义一个统一的 `LongContextGovernor`，在每轮 query 中按如下顺序工作：

1. 对超大 tool result 施加预算替换。
2. 执行 microcompact。
3. 执行其他投影式上下文折叠。
4. 判断是否触发 proactive compaction。
5. 若触发，先尝试 session memory compaction，再尝试 full compact。
6. 若模型调用真实返回 prompt-too-long 或 media-size rejection，则执行 reactive recovery。
7. compact 成功后执行 cleanup，并用 `CompactionResult` 重建上下文。

## 6. microcompact 规范

### 6.1 职责

`MicrocompactEngine` 负责在不生成 summary 的前提下，优先收缩旧工具结果占用。

### 6.2 最低能力要求

必须支持两类策略：

1. cold-cache 时的内容清空策略。
2. warm-cache 时的 cache-edit 策略。

### 6.3 约束

1. 仅允许对配置好的可压缩工具结果生效。
2. 默认只在主线程使用 cache-edit 路径。
3. 必须能够返回额外的 API-layer edit 指令，而不一定直接改 message 内容。

建议接口：

```rust
pub trait MicrocompactEngine {
    async fn apply(
        &self,
        messages: Vec<RuntimeMessage>,
        ctx: &ToolContextSnapshot,
        query_source: &QuerySource,
    ) -> MicrocompactOutcome;
}

pub struct MicrocompactOutcome {
    pub messages: Vec<RuntimeMessage>,
    pub pending_cache_edits: Option<PendingCacheEdits>,
}
```

## 7. AutoCompactController 规范

### 7.1 职责

负责：

1. 计算有效上下文窗与 auto-compact threshold。
2. 根据 query source / feature mode 决定是否跳过。
3. 维持 consecutive failure circuit breaker。

### 7.2 语义要求

必须支持：

1. `effective_window = context_window - reserved_summary_output`
2. `auto_threshold = effective_window - buffer`
3. circuit breaker：达到阈值后停止继续 autocompact

### 7.3 递归保护

以下 query source 不得再次进入 proactive compaction：

1. `compact`
2. `session_memory`
3. 其他会污染主线程共享状态的内部 fork source

## 8. Full CompactionExecutor 规范

### 8.1 职责

在需要生成 summary 时：

1. 运行 pre-compact hooks。
2. 调用 summarize 模型。
3. 处理 compact 请求本身的 prompt-too-long 重试。
4. 组装 post-compact attachments 与 hooks。
5. 产出稳定的 `CompactionResult`。

### 8.2 summarize 输入治理

至少应支持：

1. 去除媒体块或将其替换为 marker。
2. 剔除会在 compact 后重新注入的附件。
3. 在需要时按 API round group 截断最旧历史重试。

### 8.3 cache-sharing

实现上建议支持两种 summarize 路径：

1. 使用 cache-safe params 的 forked summarize。
2. 不依赖 parent cache 的 fallback summarize。

第一种更优，第二种是可靠降级。

## 9. CompactionResult 组装顺序

`build_post_compact_messages()` 的稳定顺序必须固定为：

1. boundary marker
2. summary messages
3. messages to keep
4. attachments
5. hook results

这是 transcript loader、resume、继续 query 的共同契约，不能改成可选顺序。

## 10. Preserved Segment 规范

### 10.1 目标

当 compact 需要保留一段 tail 时，系统不能重写整段 transcript，只能记录最小 relink 元数据。

### 10.2 语义

boundary metadata 必须能够表达：

1. preserved segment 的 head
2. preserved segment 应该接到哪个 anchor 后面
3. preserved segment 的 tail

### 10.3 Rust 建议接口

```rust
pub struct BoundaryMetadata {
    pub preserved_segment: Option<PreservedSegment>,
    pub discovered_tools: Vec<String>,
}

pub trait PreservedSegmentRelinker {
    fn relink(&self, transcript: &mut TranscriptStore);
}
```

## 11. SessionMemoryManager 规范

### 11.1 职责

后台维护会话记忆文件，并向 compaction 提供可消费的摘要源。

### 11.2 执行方式

必须满足：

1. 只在主线程触发。
2. 顺序执行，不允许并发 extraction。
3. 使用隔离的 subagent context。
4. 仅允许编辑目标 session memory 文件。

### 11.3 阈值驱动

默认应支持三个阈值：

1. 初始化上下文规模阈值。
2. 两次更新之间的上下文增长阈值。
3. 两次更新之间的最小 tool call 数。

### 11.4 状态更新规则

successful extraction 后必须：

1. 记录当时的上下文 token 规模。
2. 若最后一个 assistant turn 没有 tool call，则更新 `last_summarized_message_id`。
3. 清除 `extraction_started_at`。

## 12. Session Memory File 规范

session memory 文件必须被视为结构化记忆账本，而不是自由文本日志。

最低要求：

1. 有固定 section 模板。
2. 更新时必须保留模板骨架。
3. 有总长度预算与单 section 长度预算。
4. compaction 读取时可对超长 section 做截断，但要显式标记。

## 13. SessionMemoryCompactor 规范

### 13.1 职责

优先用 session memory 替代 full compact summary，从而减少一次昂贵 summarize API 调用。

### 13.2 执行前提

必须先：

1. 等待进行中的 extraction 完成，但要有 timeout 和 stale 检测。
2. 读取 session memory 文件。
3. 若文件不存在或仍是空模板则返回 `None`。

### 13.3 preserved tail 选择规则

不能简单保留最后 N 条，而必须满足：

1. 至少保留最小 token 数。
2. 至少保留最小数量的 text-block messages。
3. 不超过最大 token cap。
4. 不跨越最近 compact boundary。
5. 不拆开 tool_use / tool_result 对。
6. 不拆开同一 assistant `message.id` 的 thinking / tool_use 分片。

## 14. RelevantMemoryPrefetcher 规范

### 14.1 职责

根据当前用户请求，在不阻塞主轮采样的前提下异步找回相关 memory files，并在工具执行后择机注入。

### 14.2 语义要求

必须满足：

1. prefetch 启动一次即可，不在每个等待点重复触发。
2. collect 点必须 zero-wait；未 settle 则直接跳过并留给下一 iteration。
3. 每轮 recall 文件数有限。
4. 每文件内容量和整个 session 的累计 recall 量都有限。
5. 已 surfacing 或已读入上下文的 memory file 不得重复注入。

建议接口：

```rust
pub trait RelevantMemoryPrefetcher {
    fn start(&self, input: &str, state: &MemoryRecallState) -> MemoryPrefetchHandle;
    async fn collect_if_ready(
        &self,
        handle: &mut MemoryPrefetchHandle,
        read_state: &mut FileStateCache,
    ) -> Vec<Attachment>;
}
```

## 15. Nested Memory 规范

nested memory 属于稳定规则层，不属于 relevant memory recall。

最低要求：

1. 使用非驱逐集合防止反复注入同一规则文件。
2. 同时记录到 read-state，支持跨轮 dedupe。
3. 若注入内容是原文件的 partial view，必须显式标记，避免后续 Edit 基于不完整内容工作。

## 16. Reactive Compact Driver 规范

### 16.1 当前证据等级

由于恢复树中的 `reactiveCompact.ts` 只有 stub，以下仅能作为接口级契约，而不能视为实现级细节已证实。

### 16.2 必须保留的接口语义

1. 只有真实 API overflow / media-size failure 才能触发。
2. prompt-too-long 时应先让 cheaper recovery path 处理一次。
3. media-size recovery 走 strip-retry，不应假设 collapse 能解决。
4. 同一轮只允许尝试一次，防止死循环。
5. 成功后也必须产出标准 `CompactionResult` 并执行统一 cleanup。

## 17. PostCompactCleaner 规范

compact 成功后必须统一清理以下共享状态：

1. microcompact 状态
2. user context cache
3. memory-file cache
4. system prompt sections cache
5. classifier / speculative permission 状态
6. session message cache

同时必须允许按 query source 区分 main-thread cleanup 与 subagent-safe cleanup，避免子代理 compact 误伤主线程共享状态。

### 18.1 `agent-compact`

2. `microcompact.rs`
3. `autocompact.rs`
4. `compact_executor.rs`
5. `preserved_segment.rs`
6. `cleanup.rs`

### 18.2 `agent-memory`

1. `session_memory/state.rs`
2. `session_memory/extractor.rs`
3. `session_memory/compactor.rs`
4. `memory_recall/relevant_prefetch.rs`
5. `memory_recall/nested_memory.rs`

## 19. 可直接转化的 Rust 编程任务

1. 定义 `AutoCompactTracking`、`SessionMemoryState`、`CompactionResult`、`PreservedSegment`。
2. 实现 `MicrocompactEngine` 的 cold-cache 与 cache-edit 双路径。
3. 实现 `AutoCompactController` 的阈值计算、抑制条件与 circuit breaker。
4. 实现 `CompactionExecutor`，支持 summarize 重试和 post-compact reconstruction。
5. 实现 `SessionMemoryManager` 的阈值判断、文件更新、严格限权子代理执行。
6. 实现 `SessionMemoryCompactor` 的 preserved tail 选择算法与 resumed-session 分支。
7. 实现 `RelevantMemoryPrefetcher` 的异步 recall、zero-wait collect 与 dedupe。
8. 实现 `PostCompactCleaner` 与 transcript relink。
9. 为 reactive compact 预留 trait/interface，并先以 no-op 或 fallback adapter 承接。

## 20. 验收测试矩阵

至少应覆盖以下测试：

1. 冷 cache 场景下 microcompact 会清空旧工具结果，但保留最近工作集。
2. cached microcompact 仅在主线程且模型支持时启用。
3. proactive autocompact 在 `compact` / `session_memory` query source 下不会递归触发。
4. autocompact 连续失败达到阈值后会熔断。
5. full compact 产出的消息顺序固定为 boundary -> summary -> keep -> attachments -> hooks。
6. preserved segment relink 后 transcript 链连续，且不会丢失 kept tail。
7. session memory extraction 只有在 token threshold 满足时才触发。
8. session memory extraction 不会在最近 assistant turn 仍含 tool calls 时推进 `last_summarized_message_id`。
9. session memory compaction 不会拆开 tool_use / tool_result 对，也不会拆开同 `message.id` 的 thinking 分片。
10. relevant memory prefetch 未完成时 collect 点零等待跳过，完成后才注入。
11. compact 后 cleanup 不会把子代理 compaction 的副作用扩散到主线程共享状态。
12. reactive compact 接口最多尝试一次，并能与 withheld 413 流程正确衔接。

## 21. 当前已知缺口

本规格已经足以支撑 `agent-compact` 与 `agent-memory` 的骨架设计。此前尚未闭环的部分，现已由补遗文档单列：

1. `10-任务扩展类型补遗` 补足了 compact 后 async task reconstruction 的边界。
2. `11-reactive compact 恢复补遗` 补足了真实 overflow recovery 的协议级语义。

如果后续继续外扩，本模块仍值得单列深化的主题主要是：

1. context collapse 与 compact 的完整耦合关系。
