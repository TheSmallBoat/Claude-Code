# ADR-006：分布式运行时与多代理协同

## 状态

Proposed

## 日期

2026-04-02

## 背景

现有审计显示，成熟 Code Agent 不能只停留在单终端、单会话、单代理模式。远程控制、后台持续运行、跨会话自治和多代理协同已经构成系统长期演化方向。如果这部分不在架构阶段定义清楚，后续会出现大量旁路实现和语义分叉。

## 决策

新一代系统在架构上原生支持以下四类运行形态：

1. 本地主会话 agent
2. forked/background agent
3. remote/bridge attached session
4. coordinator-worker 多代理协同

它们统一复用同一 agent kernel，不允许每种模式拥有独立执行语义。

## 决策理由

### 理由一：高阶能力本质上是同一内核的不同宿主

Bridge、Remote、KAIROS、Coordinator 的差异主要体现在 transport、权限、生命周期和 UI，而非 query 本身。

### 理由二：多代理难点在角色边界，不在并发数量

Coordinator 的核心价值是职责分离、结果回流和 prompt 自包含，而不是起更多 worker。因此必须在运行时中显式建模角色边界。

### 理由三：后台与自治必须可观测、可恢复

持续运行体如果缺少 task state、心跳、锁、回滚和 notification 机制，系统会迅速退化为不可控后台脚本。

## 方案细节

### 统一运行形态模型

```rust
pub enum RuntimeMode {
    LocalInteractive,
    LocalHeadless,
    BackgroundAgent,
    RemoteAttached,
    RemoteViewer,
    Coordinator,
    Worker,
    Assistant,
}
```

### 角色约束

#### Coordinator

- 不直接承担重度执行工作
- 以任务分配、结果综合和用户沟通为主
- 只能通过受限工具集与 worker 交互

#### Worker

- 使用统一 query kernel 执行任务
- 通过结构化 task notification 回流结果
- 权限与上下文边界独立于 coordinator

### 远程 transport 约束

1. transport 负责认证、续签、重连、心跳。
2. transport 不定义新的 query 语义。
3. remote message 必须进入统一消息 IR。

### 持续运行体约束

1. 背景任务必须有显式状态与 output artifact。
2. auto-dream、cron、proactive loop 必须视为内核能力，不得作为隐藏脚本拼接。

## 被拒绝的备选方案

### 方案 A：分布式模式各自实现自己的执行栈

拒绝原因：最终将导致本地、远程、后台、多代理行为不一致，且维护成本不可控。

### 方案 B：Coordinator 直接共享 worker 全部上下文

拒绝原因：会模糊角色边界，降低 prompt 自包含性，且不利于替换/恢复 worker。

### 方案 C：把后台自治逻辑放在 shell/cron 外部脚本层

拒绝原因：会丧失与消息流、权限、任务状态和记忆系统的一致性。

## 后果

### 正面后果

1. 本地、后台、远程和多代理将共享统一内核。
2. 后续扩展 Bridge、Assistant、Coordinator 成本显著降低。
3. 分布式能力更容易做故障注入与恢复测试。

### 负面后果

1. 初期需要设计更多 runtime mode 和 role 边界。
2. transport 与 task runtime 的接口设计要求更高。

## 实施建议

1. 第一阶段先实现 LocalInteractive 与 BackgroundAgent。
2. 第二阶段实现 Coordinator 与 Worker 角色模型。
3. 第三阶段再实现 RemoteAttached 与 Bridge transport。
