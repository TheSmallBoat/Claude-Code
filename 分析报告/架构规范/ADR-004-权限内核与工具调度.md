# ADR-004：权限内核与工具调度

## 状态

Proposed

## 日期

2026-04-02

## 背景

Code Agent 不是被动阅读系统，而是具有文件修改、命令执行、网络访问、远程扩展调用等能力的主动执行系统。若权限和工具调度没有进入内核，系统将无法在企业或生产环境中可信运行。

## 决策

新一代系统将权限评估、危险规则分析和工具批调度统一纳入执行内核。

### 权限内核必须负责

1. 规则装载与合并
2. 规则来源区分
3. 工具输入级决策
4. 危险规则静态识别
5. 后台代理权限行为控制
6. 自动模式约束

### 工具调度器必须负责

1. 工具并发安全分类
2. 只读批并行
3. 副作用批串行
4. 上下文修改顺序保证
5. 执行结果与 transcript 对齐

## 决策理由

### 理由一：权限是运行语义，不是 UI 行为

权限模式会直接影响工具能否执行、是否等待用户、后台代理如何运行，因此必须由统一内核决策。

### 理由二：危险授权应在执行前被识别

像 Bash 全通配、PowerShell 解释器前缀、Agent 工具过宽授权这类规则，一旦放入自动模式，相当于绕过安全评估。必须显式做静态风险分析。

### 理由三：工具并发与安全强相关

如果不区分只读和副作用工具，系统虽然“更并发”，但状态和 transcript 将不可预测，且权限提示与恢复会更加混乱。

## 方案细节

### 核心抽象

```rust
pub enum PermissionMode {
    Default,
    Ask,
    Auto,
    Bypass,
    AcceptEdits,
    Bubble,
}

pub trait PermissionEvaluator {
    fn evaluate(&self, request: &PermissionRequest) -> PermissionDecision;
}

pub trait RuleRiskAnalyzer {
    fn analyze(&self, rules: &[PermissionRule]) -> Vec<RiskyRule>;
}

pub trait ToolScheduler {
    fn plan(&self, calls: &[ToolCall]) -> SchedulePlan;
}
```

### 调度约束

1. `ReadOnly` 工具允许批并发。
2. `Mutating` 工具必须串行。
3. `UnknownSafety` 工具按保守策略处理。
4. 同一批次的 context modifier 应按确定顺序应用。

## 被拒绝的备选方案

### 方案 A：让每个工具自己决定权限与并发

拒绝原因：这会让系统失去全局可治理性，且难以统一审计和测试。

### 方案 B：一律串行执行所有工具

拒绝原因：会显著损失只读类工具的吞吐，削弱大代码库探索性能。

### 方案 C：只在宿主层弹窗时做权限判断

拒绝原因：后台代理、远程控制和自动模式都不能依赖宿主 UI 才获得正确语义。

## 后果

### 正面后果

1. 安全行为稳定且可验证。
2. 工具并发性能与确定性可兼得。
3. 后台 agent 与前台 agent 可共享同一安全模型。

### 负面后果

1. 内核实现会更复杂。
2. 工具开发者需要提供更完整的元数据。

## 实施建议

1. 优先完成 `PermissionMode`、`PermissionDecision`、`SchedulePlan`。
2. 再完成 Bash/PowerShell/Agent 类危险规则分析。
3. 最后补全远程权限响应与后台代理策略。
