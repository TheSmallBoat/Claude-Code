# 存储-观测-恢复实现设计

## 文档状态

- 状态：Proposed
- 日期：2026-04-02
- 对应上游：模块接口清单、全部 ADR 的实现支撑面

## 目标

定义 transcript、session snapshot、task snapshot、compaction artifact、extension metadata 等关键对象的持久化、观测与恢复策略，确保系统不是一次性 demo，而是可恢复、可排障、可演进的 runtime。

## 存储对象清单

建议最小持久化集合包括：

- transcript log
- session metadata
- query checkpoint
- task snapshot
- compaction artifact
- memory catalog index
- transport session metadata

## 存储抽象

建议把存储能力拆成多个小接口，而不是单一大 store：

```rust
pub trait SessionStore {
    fn save_session(&self, session: &SessionSnapshot) -> Result<(), StoreError>;
    fn load_session(&self, id: &SessionId) -> Result<SessionSnapshot, StoreError>;
}

pub trait TaskStore {
    fn save_task(&self, task: &TaskSnapshot) -> Result<(), StoreError>;
    fn load_task(&self, id: &TaskId) -> Result<TaskSnapshot, StoreError>;
}
```

## 观测模型

建议同时提供三类观测输出：

1. 结构化日志：用于排障与审计。
2. 指标：用于运行统计、延迟、错误率、预算命中率。
3. 事件流：用于宿主 UI、远程观察者、debug 工具订阅。

关键指标至少包括：

- query latency
- tool latency
- permission decision counts
- compaction trigger counts
- session recovery success rate
- remote reconnect success rate

## 恢复边界

系统应至少支持以下恢复场景：

1. 进程重启后恢复 session 列表。
2. 中断回合恢复到可继续 checkpoint。
3. 后台 task 重新装载并继续执行。
4. 远程 transport 断连后自动重连。
5. compaction artifact 丢失后的降级恢复。

## Recovery Orchestrator

建议引入独立恢复协调器，而不是把恢复逻辑散落在各模块初始化路径中：

```rust
pub trait RecoveryOrchestrator {
    fn recover_sessions(&self) -> Result<Vec<RecoveredSession>, RecoveryError>;
    fn recover_tasks(&self) -> Result<Vec<RecoveredTask>, RecoveryError>;
}
```

## 一致性策略

- transcript append 应优先保证顺序一致性
- session snapshot 可采用最终一致，但必须携带版本号
- task state 写入必须具备幂等恢复能力
- compaction artifact 与 transcript range 必须可对账

## 调试建议

实现阶段应默认保留：

- query state transition trace
- tool call audit trail
- permission decision reason
- compaction decision trace
- transport reconnect trace

这些信息应既可写入日志，也可在 debug host 中订阅显示。

## 最小实施顺序

1. 先实现 transcript 和 session snapshot 存储。
2. 再实现 query checkpoint。
3. 然后实现 task snapshot 与恢复。
4. 最后接入 transport recovery 和统一指标。
