# 工具-权限-任务执行设计

## 文档状态

- 状态：Proposed
- 日期：2026-04-02
- 对应上游：ADR-004、模块接口清单

## 目标

定义 tool registry、permission evaluator、task runtime 的实现边界，确保系统既能高效执行能力调用，又不会把高风险副作用散落到 UI 或宿主层。

## 三类执行单元

系统中应明确区分三类执行单元：

1. 工具调用：单回合内、短生命周期、由模型显式发起。
2. 后台任务：跨回合、可挂起恢复、通常与自治能力相关。
3. 宿主动作：由 host 直接执行的界面或交互动作，不属于工具或任务语义。

## Tool Registry

工具注册表建议公开如下能力：

```rust
pub trait ToolRegistry {
    fn list(&self) -> Vec<ToolDescriptor>;
    fn resolve(&self, name: &str) -> Result<Arc<dyn ToolExecutor>, ToolLookupError>;
}
```

注册表内的 `ToolDescriptor` 至少应包含：

- 名称与描述
- 输入 schema
- side effect class
- 默认超时
- 是否允许后台使用
- 默认权限级别

## Tool Execution Pipeline

推荐执行路径如下：

```text
+------------------------+
| Tool Call Requested    |
+------------------------+
           |
           v
+------------------------+
| Risk Classification    |
+------------------------+
           |
           v
+------------------------+
| Permission Evaluation  |
+------------------------+
           |
           v
+------------------------+
| Tool Execution         |
+------------------------+
           |
           v
+------------------------+
| Result Normalization   |
+------------------------+
```

## Permission Kernel

权限系统必须内核化，建议由 `agent-permissions` 提供统一接口：

```rust
pub trait PermissionEvaluator {
    fn evaluate_tool_call(&self, req: &ToolPermissionRequest) -> PermissionDecision;
    fn evaluate_task_action(&self, req: &TaskPermissionRequest) -> PermissionDecision;
}
```

决策类型应至少覆盖：

- allow
- deny
- ask-user
- allow-with-constraints

## 任务运行时

`agent-tasks` 不应只是工具循环的延长线，而应拥有自己的状态模型：

```rust
pub enum TaskState {
    Pending,
    Running,
    WaitingForInput,
    WaitingForSchedule,
    Retrying,
    Completed,
    Failed,
    Cancelled,
}
```

每个 task 必须具备：

- `TaskId`
- owner session
- spawn reason
- permission profile
- durable state snapshot
- output artifact references

## 隔离原则

- tool executor 通过受控 `ToolContext` 执行，不直接访问 runtime 全局状态
- task runtime 通过消息接口与 query runtime 通信，不共享可变内存
- 高风险工具执行需支持超时、取消、审计日志与资源上限

## 错误边界

建议分离三类错误：

- `ToolExecutionError`
- `PermissionError`
- `TaskRuntimeError`

这些错误不得在 UI 层被重新发明语义，而应由 kernel 统一规范化。

## 最小实现顺序

1. 先实现只读工具和最小 permission evaluator。
2. 再实现可变更工具与 ask-user 分支。
3. 然后实现 task state 持久化。
4. 最后把自治型后台任务接入统一 task runtime。
