# 运行时启动与装配设计

## 文档状态

- 状态：Proposed
- 日期：2026-04-02
- 对应上游：ADR-001、模块接口清单

## 目标

定义新一代 Rust Code Agent 的 workspace 布局、crate 分层、启动路径、composition root 与 host adapter 装配方式，确保系统从一开始就按“内核优先、宿主后置”的方式落地。

## Rust workspace 建议结构

建议采用单 workspace、多 crate 的结构：

```text
workspace/
  crates/
    agent-message/
    agent-context/
    agent-query/
    agent-tools/
    agent-permissions/
    agent-tasks/
    agent-extensions/
    agent-transport/
    agent-runtime/
    host-cli/
    host-tui/
    host-sdk/
  apps/
    code-agent-cli/
    code-agent-daemon/
```

## 分层要求

- `agent-runtime` 作为组合根，负责把其它 `agent-*` crate 装配为可运行系统
- `host-*` 只依赖 `agent-runtime` 暴露的稳定接口，不直接穿透到更深模块
- `apps/` 下的二进制项目只做参数解析、配置加载和 runtime 启动

## 启动链路

推荐启动主线如下：

```text
+------------------------+
| App Entry              |
| main.rs / cli.rs       |
+------------------------+
           |
           v
+------------------------+
| Config Loader          |
| files / env / flags    |
+------------------------+
           |
           v
+------------------------+
| Runtime Builder        |
| compose services       |
+------------------------+
           |
           v
+------------------------+
| Host Adapter           |
| CLI / TUI / SDK        |
+------------------------+
           |
           v
+------------------------+
| Session Bootstrap      |
| create or resume       |
+------------------------+
```

## Composition Root 责任

`agent-runtime` 中应存在唯一的 composition root，建议暴露如下结构：

```rust
pub struct RuntimeBuilder {
    config: RuntimeConfig,
    services: ServiceOverrides,
}

pub struct Runtime {
    pub query_engine_factory: Arc<dyn QueryEngineFactory>,
    pub transcript_store: Arc<dyn TranscriptStore>,
    pub task_runtime: Arc<dyn TaskRuntime>,
    pub extension_registry: Arc<dyn ExtensionRegistry>,
    pub transport_manager: Arc<dyn TransportManager>,
}
```

## 配置装配原则

- 配置源统一收敛为 `RuntimeConfig`
- flags、环境变量、配置文件先归一化，再进入 builder
- 任何模块不得在内部直接读取环境变量
- 实验开关不得散落在 crate 内部，统一进入 config 或 feature gate registry

## Session Bootstrap

session 启动阶段建议拆成以下步骤：

1. 解析启动模式：new、resume、headless、remote-attached。
2. 生成或恢复 `SessionId`。
3. 加载 transcript、session metadata、task metadata。
4. 创建 `SessionRuntimeHandle`。
5. 将 handle 交给 host adapter 驱动。

建议公开的宿主入口：

```rust
pub trait RuntimeFacade {
    fn create_session(&self, req: CreateSessionRequest) -> Result<SessionHandle, RuntimeError>;
    fn resume_session(&self, session_id: &SessionId) -> Result<SessionHandle, RuntimeError>;
    fn list_sessions(&self, filter: SessionFilter) -> Result<Vec<SessionSummary>, RuntimeError>;
}
```

## 依赖注入策略

建议分三层注入：

1. 基础设施层：clock、fs、http client、process runner、kv store。
2. 领域服务层：query engine、context builder、permission evaluator、task runtime。
3. 宿主层：renderer、input source、notification sink、interactive approval handler。

测试时应允许替换上述三层依赖，避免 runtime 与外部环境强耦合。

## 并发模型建议

- session 作为并发隔离边界
- 每个 session 有独立 event loop 和 mailbox
- tool execution、transport I/O、task runtime 通过异步任务并发执行
- 状态写入应通过单写者原则归并，避免多处争抢 transcript 与 session snapshot

## 最小落地顺序

1. 建立 workspace 与 crate skeleton。
2. 先实现 `agent-message` 与 `agent-runtime` 的最小闭环。
3. 接入 `host-cli`，做最小命令行交互验证。
4. 再补 `agent-query`、`agent-context`、`agent-tools`。
5. 最后接 `host-tui`、`host-sdk` 与 transport。
