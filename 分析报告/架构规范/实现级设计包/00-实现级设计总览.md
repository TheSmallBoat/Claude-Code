# 实现级设计总览

## 文档状态

- 状态：Proposed
- 日期：2026-04-02
- 作用：把“审计报告”与“架构规范”进一步下沉为可进入编码阶段的实现级设计说明

## 目标

本设计包用于回答一个更具体的问题：当实现团队开始用 Rust 落地新一代 Code Agent 时，应该如何把既有架构决策展开为 crate、模块、状态机、运行时组件、数据边界和工程落地步骤。

因此，这一层文档不再讨论“是否应该这样设计”，而是讨论“按既定 ADR，应该怎样实现”。

## 与上两层文档的关系

完整文档体系分为三级：

1. 审计报告：回答“现有系统实际上是如何工作的”。
2. 架构规范：回答“新系统必须固定哪些结构性决策”。
3. 实现设计：回答“工程团队应如何把这些决策落成代码与模块”。

边界约束如下：

- 审计报告提供事实证据与迁移启发
- 架构规范提供 ADR、依赖方向和模块边界
- 实现设计提供 crate 组织、关键结构体、生命周期、调度方式、恢复策略和实施顺序

## 适用范围

本设计包覆盖以下内容：

- Rust workspace 与 crate 分层
- runtime bootstrap 与 composition root
- query 状态机与消息管线
- context、memory、compaction 的实现流程
- tools、permissions、tasks 的调度与隔离
- extensions、transport、multi-agent 的装配方式
- storage、observability、recovery 的实现骨架

本设计包不覆盖以下内容：

- 具体 UI 像素级设计
- 某个模型供应商 SDK 的最终选型
- 某个发布平台的 CI/CD 细节
- 语言绑定层的最终 API 形式

## 推荐阅读顺序

建议按以下顺序进入实现阶段：

1. 先读 [01-运行时启动与装配设计](./01-运行时启动与装配设计.md)，确定 workspace 和 composition root。
2. 再读 [02-查询状态机与消息管线设计](./02-查询状态机与消息管线设计.md)，确定最核心的执行内核。
3. 然后读 [03-上下文-记忆-压缩实现设计](./03-上下文-记忆-压缩实现设计.md)，补齐上下文预算和记忆体系。
4. 再读 [04-工具-权限-任务执行设计](./04-工具-权限-任务执行设计.md)，锁定能力执行与安全边界。
5. 然后读 [05-扩展-传输-多代理实现设计](./05-扩展-传输-多代理实现设计.md)，接入平台化与分布式能力。
6. 最后读 [06-存储-观测-恢复实现设计](./06-存储-观测-恢复实现设计.md)，补齐可运维性与恢复路径。

## 实施原则

### 原则一：先做单机稳定闭环，再做远程与自治

第一阶段必须先完成本地单会话闭环：输入、上下文装配、模型流、工具执行、消息落盘、恢复。

### 原则二：先固化消息与状态，再固化外层接口

任何 host API、SDK API、transport 协议都必须建立在稳定的消息 IR 与 query 状态机之上。

### 原则三：可恢复性优先于功能堆叠

每新增一个运行能力，都必须同时回答：如何落盘、如何恢复、如何重试、如何观测。

### 原则四：实现包不得反向突破 ADR

如果实现过程中发现当前设计无法容纳某需求，应回到 ADR 层修订，而不是在实现文档里绕过依赖方向。

## 编码前检查项

进入具体 crate 开发前，每个实现项都应先回答：

1. 它属于哪个 `agent-*` 或 `host-*` crate。
2. 它的状态归谁拥有，如何持久化。
3. 它产生什么事件，消费什么事件。
4. 它失败时由谁恢复，恢复到哪个边界。
5. 它是否需要新的配置、权限、观测字段。

## 交付预期

实现团队读完本设计包后，应能够直接开展以下工作：

- 初始化 Rust workspace 与 crate 清单
- 建立 query loop 的最小可运行版本
- 定义 transcript、session state、task state 的持久化模型
- 建立 tool execution、permission evaluation、task runtime 的基本执行骨架
- 规划 extensions、transport、remote session 的接入顺序
