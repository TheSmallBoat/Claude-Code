# ADR-001：内核与宿主分层

## 状态

Proposed

## 日期

2026-04-02

## 背景

现有 Code Agent 系统同时支持 CLI、REPL、后台任务、远程控制、远程会话 viewer、多代理 worker 等多种形态。若新一代系统继续按“先做 CLI，再往里面塞各种模式”的方式构建，最终会得到一个强耦合且难以演进的宿主程序，而不是可复用的 agent runtime。

Rust 重构的关键机会在于：把原本分散在入口、UI、工具、后台代理和远程逻辑之间的共用运行语义提升为统一内核。

## 决策

新一代 Code Agent 采用“内核优先、宿主适配器后置”的架构分层。

### 内核层职责

内核层负责：

- 消息模型与转录规范
- 查询状态机与事件驱动执行
- 工具调度与任务调度
- 上下文构建、记忆与压缩
- 权限策略与风险分析
- 会话状态与恢复语义

### 宿主层职责

宿主层负责：

- CLI 参数解析
- TUI 或桌面 UI 展示
- 用户交互输入输出
- 终端、编辑器、浏览器等平台适配
- 远程 transport 端的收发与会话管理

### 架构边界

宿主层不得直接定义新的运行时语义，只能调用内核提供的显式接口。新的运行模式必须通过内核的统一状态机与消息模型接入，而不是在宿主层旁路实现。

## 决策理由

### 理由一：统一运行语义

主线程 agent、后台 agent、远程 session、本地 REPL 的核心执行规律是一致的，区别只在于上下文来源、权限策略、transport 和 UI 行为。内核化能避免重复实现。

### 理由二：提升可测试性

将状态机与消息处理从 UI/CLI 中剥离后，可以对核心行为做 golden replay、property-based testing 和 deterministic replay。

### 理由三：支持多宿主演进

未来若需要支持桌面端、IDE 内嵌、server mode 或 WebAssembly，只要宿主层遵守统一接口，内核无需重写。

### 理由四：降低分布式复杂度

Bridge、remote viewer、多代理协同等分布式能力本质上只应包装 transport 和身份管理，而不应改变 agent 内核。如果宿主与内核耦合过深，分布式能力会复制并扭曲核心语义。

## 方案细节

### 推荐 crate 分层

1. `agent-message`
2. `agent-context`
3. `agent-query`
4. `agent-tools`
5. `agent-permissions`
6. `agent-tasks`
7. `agent-extensions`
8. `agent-transport`
9. `agent-runtime`
10. `host-cli`
11. `host-tui`
12. `host-sdk`

### 依赖方向

```text
+------------------+ 
| host-cli         |
+------------------+
          \
           \
+------------------+      +----------------------+
| host-tui         | ---> | agent-runtime        |
+------------------+      +----------------------+
           /                  |
          /                   +--> +------------------+ ---> +----------------------+
+------------------+          |    | agent-query      |      | agent-message        |
| host-sdk         | ---------+    +------------------+      +----------------------+
+------------------+          |
                              +--> +------------------+ ---> +----------------------+
                              |    | agent-context    |      | agent-message        |
                              |    +------------------+      +----------------------+
                              |
                              +--> +------------------+ ---> +----------------------+
                              |    | agent-tools      |      | agent-message        |
                              |    +------------------+      +----------------------+
                              |
                              +--> +------------------+
                              |    | agent-permissions|
                              |    +------------------+
                              |
                              +--> +------------------+ ---> +----------------------+
                              |    | agent-tasks      |      | agent-message        |
                              |    +------------------+      +----------------------+
                              |
                              +--> +------------------+
                              |    | agent-extensions |
                              |    +------------------+
                              |
                              +--> +------------------+ ---> +----------------------+
                                   | agent-transport  |      | agent-message        |
                                   +------------------+      +----------------------+
```

### 强制约束

1. `host-*` 不能被 `agent-*` 反向依赖。
2. `agent-transport` 不能持有 `host-*` 状态对象。
3. `agent-query` 不能直接依赖 UI 概念。
4. `agent-tools` 不能自行读写全局配置单例。
5. 所有宿主对内核的通信必须使用显式 `Request` / `Event` / `Command` 接口。

## 被拒绝的备选方案

### 方案 A：先做一个单体 CLI，再逐步抽模块

拒绝原因：这会把所有语义先固化在 CLI 中，后续拆分成本高，而且极易形成“宿主层定义内核”的反向依赖。

### 方案 B：让 transport 层直接驱动工具和任务

拒绝原因：这会让 Bridge/Remote 等分布式能力拥有不同于本地会话的执行语义，破坏内核一致性。

### 方案 C：把 UI store 当作主状态容器

拒绝原因：UI 状态容器不是长期可演化的运行时状态模型，难以支持 deterministic replay 和多宿主复用。

## 后果

### 正面后果

1. 内核具备强复用性。
2. 宿主实现可以并行推进。
3. 远程模式和本地模式更容易保持语义一致。
4. 后续做桌面端或 IDE 插件时成本更低。

### 负面后果

1. 前期接口设计成本提高。
2. 宿主与内核之间需要明确定义更多中间类型。
3. 某些原本在单体程序中很方便的隐式共享状态将被禁止。

## 实施建议

1. 第一阶段只实现内核最小闭环：消息、query、tools、permissions。
2. CLI 作为第一个宿主接入，但不得下沉新的内核语义。
3. 分布式能力只在本地宿主稳定后接入。
