# ASCII 图例规范

## 文档状态

- 状态：Proposed
- 日期：2026-04-02
- 适用范围：`分析报告/` 与 `分析报告/架构规范/` 中全部 ASCII 流程图、状态图、分层图、依赖图

## 目标

本规范用于统一整套报告中的 ASCII 图书写方式，避免同类图形在标题、框体、箭头、分支、回环、注释上的表达漂移。

本规范约束的是“图的写法”，不是“图表达的架构内容”。任何新增或修改的 ASCII 图，都应优先遵循本文件，而不是临时发明新样式。

## 总体约定

### 约定一：统一使用 `text` 代码块

所有 ASCII 图必须放在 Markdown 的 `text` 代码块中。

### 约定二：框体统一使用单层直角边框

标准框体写法如下：

```text
+----------------------+
| Module Name          |
+----------------------+
```

约束如下：

- 使用 `+` 作为四角与横向连接点
- 使用 `-` 作为水平边框
- 使用 `|` 作为垂直边框
- 不使用 Unicode 线框字符
- 同一张图内，同层级框体宽度应尽量一致

### 约定三：标题写在框内第一行

如果框内只有一个语义单元，标题直接写在唯一内容行中。

```text
+----------------------+
| agent-runtime        |
+----------------------+
```

如果框内还要补充角色、子项或说明，标题放在第一行，补充说明放在后续行。

```text
+----------------------+
| Context Plane        |
| prompt / memory /    |
| compaction / cache   |
+----------------------+
```

### 约定四：命名保持与正文术语一致

图中的模块名、状态名、阶段名必须与正文中的术语保持一致，不要为了凑宽度而改名，也不要在同一套文档里混用近义词。

## 箭头约定

### 纵向推进

表示主流程向下推进时，统一写法如下：

```text
+----------------------+
| Prepare Input        |
+----------------------+
          |
          v
+----------------------+
| Build Context        |
+----------------------+
```

约束如下：

- 竖线统一使用 `|`
- 箭头尖统一使用 `v`
- 上下框之间至少留一行连接区
- 主干流程优先使用纵向表达

### 横向依赖或指向

表示模块依赖、消息流向或分支指向时，统一使用 `--->`。

```text
+------------------+ ---> +----------------------+
| agent-query      |      | agent-message        |
+------------------+      +----------------------+
```

约束如下：

- 横向箭头统一使用 `--->`
- 不使用 `=>`、`==>`、`---->`、Unicode 箭头
- 箭头左右至少保留一个空格

### 分支引出

从一个主节点引出多个分支时，统一使用 `+-->`。

```text
   |
   +--> +------------------+
   |    | agent-tools      |
   |    +------------------+
   |
   +--> +------------------+
        | agent-tasks      |
        +------------------+
```

约束如下：

- 分支起点统一使用 `+-->`
- 平级分支应垂直对齐
- 不在同一层混用 `+->`、`|->`、`\-->`

## 状态图约定

### 状态框

状态机中的状态统一使用与模块图相同的矩形框，不使用圆角、椭圆或 Mermaid 风格起止点。

```text
+-------------------+
|   Stream Model    |
+-------------------+
```

### 起点与终点

状态机的终止节点统一写为 `[END]`，仅在确实需要表达终止时使用。

```text
          |
          v
        [END]
```

如果图只表达循环，不强调结束，可省略终点。

### 条件分叉

状态机中的条件分叉写在连线上，不额外发明菱形判断框。

```text
      | terminal stop reason
      v
+-------------------+
|    No Tool Use    |
+-------------------+
```

或：

```text
      | assistant emits tool_use
      v
+-------------------+
|  Schedule Tools   |
+-------------------+
```

约束如下：

- 条件文字写在线路附近，不写进目标框标题
- 条件短语应描述“为什么走这条边”，而不是重复状态名

### 回环

回环统一使用显式回指说明，不画复杂折线迷宫。

推荐写法：

```text
+-------------------+               +-------------------+
|   Execute Tools   |               |  Append Results   |
+-------------------+               +-------------------+
          |                                   |
          +-----------------------------------+
                      loop back to Stream Model
```

约束如下：

- 回环文字统一使用 `loop back to ...`
- 如果回环路径较长，优先用文字标注目标，而不是堆叠多段折线
- 一张图内回环说明风格必须一致

## 分层图约定

### 主干层次

分层架构图优先采用“自上而下主干 + 侧向能力分支”的布局。

```text
+--------------------------------------------------------+
| Bootstrap Plane                                         |
+--------------------------------------------------------+
                         |
                         v
+--------------------------------------------------------+
| Control Plane                                           |
+--------------------------------------------------------+
```

### 侧向平面

与主干平级但不属于主流程的平面，统一从主干节点侧向引出。

```text
   +--> +--------------------------------------+
   |    | Extension Plane                      |
   |    +--------------------------------------+
```

### 治理类约束

如果某一层是“约束所有其它层”的治理平面，不强制画成立体包围关系，统一写成“所有平面指向该治理框”的显式表达。

```text
Bootstrap  ----+
Control    ----+
Execution  ----+--> Governance Plane
Context    ----+
Extension  ----+
Distribution --+
```

## 排版约定

### 宽度

- 同一张图中，同层级框体宽度应尽量一致
- 如果不同框名称长度差异过大，以“视觉整齐优先、不过度拉长全图”为准
- 不要求整套报告所有图全局绝对等宽，但要求每张图内部一致

### 缩进

- 主干连接线使用固定缩进，避免手工漂移
- 平级分支的起点列应一致
- 说明文字相对其所解释的箭头或回环对齐

### 空白

- 图前后各留一个空行
- 框体之间至少保留一行连接区
- 不在同一图内混用过密和过疏的留白

## 不推荐写法

以下写法在整套报告中应避免：

- 把图退化成普通项目列表
- 同一图内混用多种箭头风格
- 用大量折线模拟几何图，导致 Markdown 中难以阅读
- 使用 Mermaid 语法残片作为 ASCII 图的一部分
- 在图里塞入过长句子，导致框体被异常拉宽

## 推荐工作流

新增或修改图时，建议按以下顺序处理：

1. 先确定图的类型：流程图、状态图、分层图、依赖图。
2. 再确定主干方向：优先纵向，必要时横向。
3. 然后统一同层框宽。
4. 最后补条件标注、回环说明和图下注释。
