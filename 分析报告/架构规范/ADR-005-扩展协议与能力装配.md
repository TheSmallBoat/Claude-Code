# ADR-005：扩展协议与能力装配

## 状态

Proposed

## 日期

2026-04-02

## 背景

新一代 Code Agent 若要长期演进，不可能把所有能力都做成内建代码。Claude Code 现有审计已证明，MCP、skills、plugins 和 workflows 构成了可扩展能力平面。缺少统一的扩展协议层，会导致平台能力碎片化。

## 决策

新一代系统采用“统一能力描述 + 多来源装配 + 单一治理入口”的扩展架构。

### 扩展来源

1. Built-in capabilities
2. MCP servers
3. Skills
4. Plugins
5. Workflows

### 统一装配目标

所有扩展来源最终都必须映射为标准能力描述：

- `ToolDescriptor`
- `CommandDescriptor`
- `ResourceDescriptor`
- `PromptDescriptor`

## 决策理由

### 理由一：运行时抽象必须统一

来源不同不应导致运行时对象模型不同，否则调度、权限、提示装配和观测都无法统一。

### 理由二：扩展必须进入治理框架

任何扩展都必须接受权限、安全、策略、观测和缓存约束，而不能绕过主运行时。

### 理由三：协议优先于脚本注入

对于高风险执行系统，优先选择可解析、可声明、可限制的扩展定义，而不是任意宿主脚本执行。

## 方案细节

### MCP 子系统

MCP 作为一级协议子系统，负责：

- transport 连接
- session/auth 管理
- external capability discovery
- tool/resource/prompt 标准化
- error 与 session expiry 处理

### Skill 子系统

Skill 作为轻量 prompt 级扩展，负责：

- Markdown/frontmatter 解析
- allowed-tools 声明
- model/effort/execution context 声明
- 命令级导出

### Plugin 子系统

Plugin 作为可分发打包扩展，负责：

- plugin source 与 metadata 管理
- command/skill 发现
- 配置替换与变量注入
- 策略兼容性检查

## 被拒绝的备选方案

### 方案 A：每种扩展来源都直接接入 runtime 各自逻辑

拒绝原因：最终会形成多套不兼容的能力模型与治理分支。

### 方案 B：插件允许直接执行任意宿主代码

拒绝原因：会显著破坏权限、安全、可审计性与跨宿主一致性。

### 方案 C：只做 MCP，不做 skills/plugins

拒绝原因：MCP 适合外部系统集成，不适合承载轻量 prompt 资产与项目级知识封装。

## 后果

### 正面后果

1. 平台扩展能力清晰。
2. 不同扩展来源可统一进入权限与观测体系。
3. 技能、插件和 MCP 能在同一系统中长期并存。

### 负面后果

1. 需要维护额外的标准化层。
2. 某些来源特有能力需要通过扩展字段传递。

## 实施建议

1. 首先实现 `CapabilityDescriptor` 统一模型。
2. 其次实现 MCP transport + normalization。
3. 再实现 skill frontmatter parser。
4. 最后实现 plugin package layer 与 workflows。
