# ADR-002：消息模型与查询状态机

## 状态

Proposed

## 日期

2026-04-02

## 背景

Code Agent 的执行闭环不是单次模型请求，而是“消息流驱动的状态机”。用户输入、模型流式输出、工具调用、任务回流、系统通知、远程消息，最终都需要进入同一条语义主线。若消息模型不统一，query loop 无法稳定；若状态机不显式，恢复、回放、压缩和多代理都将变得脆弱。

## 决策

新一代系统采用统一消息 IR，并以显式查询状态机作为唯一的回合执行内核。

### 消息 IR 必须覆盖的类型

1. `UserMessage`
2. `AssistantMessage`
3. `SystemMessage`
4. `ProgressMessage`
5. `ToolResultMessage`
6. `TaskNotificationMessage`
7. `CompactBoundaryMessage`
8. `RemoteControlMessage`

### Query State Machine 必须覆盖的状态

1. `PreparingInput`
2. `BuildingContext`
3. `Sampling`
4. `DispatchingTools`
5. `AwaitingToolResults`
6. `Compacting`
7. `Stopping`
8. `Completed`
9. `Failed`
10. `Aborted`

### 统一原则

所有执行路径，包括本地主回合、forked subagent、background agent、remote session replay，都必须复用同一个 query 内核，只允许在上下文、权限、transport 和宿主表现层存在差异。

## 决策理由

### 理由一：消息是跨模块唯一稳定边界

命令、工具、任务、transport、memory、UI 都会变化，但消息流是最适合作为统一交换媒介的结构。

### 理由二：状态机可显式表达 continue 语义

agent 的本质不是“得到答案”，而是“在工具结果回注后继续推理”。只有显式状态机才能把这种 continue 语义做成稳定行为。

### 理由三：支持压缩、恢复和重放

统一消息 IR 可以支持：

- transcript 持久化
- compact boundary 标记
- sidechain transcript
- deterministic replay

### 理由四：便于适配不同模型供应商

不同模型 SDK 的流式事件格式可以不同，但统一消息 IR 可以把它们收敛到上层运行时可理解的稳定结构。

## 方案细节

### 建议的核心类型

```rust
pub struct SessionId(pub String);
pub struct MessageId(pub String);
pub struct TurnId(pub String);

pub enum Message {
    User(UserMessage),
    Assistant(AssistantMessage),
    System(SystemMessage),
    Progress(ProgressMessage),
    TaskNotification(TaskNotificationMessage),
    CompactBoundary(CompactBoundaryMessage),
}

pub enum QueryState {
    PreparingInput,
    BuildingContext,
    Sampling(SamplingState),
    DispatchingTools(ToolDispatchState),
    Compacting(CompactionState),
    Stopping(StopState),
    Completed(QueryResult),
    Failed(QueryError),
    Aborted,
}
```

### 建议的事件源

1. 用户输入事件
2. 模型流式事件
3. 工具执行结果事件
4. 权限决策事件
5. 压缩完成事件
6. 任务通知事件
7. 远程 transport 事件
8. 中断与超时事件

### 关键约束

1. 任何工具结果都必须重新进入消息流，不能旁路修改状态后跳过 transcript。
2. 状态转移必须通过 reducer 或显式 transition API 实现。
3. 宿主层不能自行拼装 assistant message 并绕过 normalization。
4. 任何 compact 操作都必须留下边界标记。

## 被拒绝的备选方案

### 方案 A：直接以 SDK 事件为内部主模型

拒绝原因：SDK 事件是供应商或 transport 绑定的，不适合作为内核抽象。

### 方案 B：只保留 user/assistant 两类消息

拒绝原因：task notification、compact boundary、system diagnostic、progress 都是运行时关键语义，不能被压平为普通文本。

### 方案 C：以协程流程代替显式状态机

拒绝原因：协程流程虽可运行，但难以做恢复、调试、回放和跨宿主一致行为验证。

## 后果

### 正面后果

1. Query 内核会更稳定、更可验证。
2. 多种运行模式更容易共用同一执行语义。
3. transcript、compact、replay、恢复将获得统一基础。

### 负面后果

1. 需要设计较多中间类型。
2. 某些简单路径会显得比脚本式写法更“重”。

## 实施建议

1. 先完成消息 IR 和 normalizer。
2. 再完成 query reducer。
3. 再将 tool orchestration、permission 和 compact 挂载到状态机。
