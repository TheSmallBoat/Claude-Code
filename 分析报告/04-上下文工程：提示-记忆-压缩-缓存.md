# 04. 上下文工程：提示、记忆、压缩、缓存

## 为什么上下文工程是系统核心

对 Code Agent 而言，模型能力并不会直接等价于系统能力。系统真正的上限，往往由“上下文工程”决定。Claude Code 在这方面投入极深，其核心不是单个 prompt，而是一整套上下文生命周期系统。

可以用一句话概括：

> Claude Code 把上下文当作受预算约束、可压缩、可回收、可记忆、可缓存的运行时资源来管理。

这正是新一代 Rust Code Agent 应该重点继承的部分。

## 上下文并非单一 prompt，而是多层叠加

[src/context.ts](../src/context.ts)、[src/utils/queryContext.ts](../src/utils/queryContext.ts) 与 [src/constants/systemPromptSections.ts](../src/constants/systemPromptSections.ts) 共同表明，最终送入模型的 system prompt 前缀来源于多个层次：

1. 默认系统提示
2. 用户上下文
3. 系统上下文
4. 额外 working directory / CLAUDE.md 注入
5. coordinator 或 memory mechanics 等模式化附加上下文
6. append/custom system prompt

### 设计意义

这样做的好处不是“灵活”，而是分离不同信息的生命周期：

- 有些信息跨会话长期稳定
- 有些信息只在当前项目有效
- 有些信息只在当前模式有效
- 有些信息会破坏 cache，必须显式标记

这说明上下文构建在 Claude Code 中是一个编排问题，而不是字符串拼接问题。

## System Prompt Section 是显式缓存单元

[src/constants/systemPromptSections.ts](../src/constants/systemPromptSections.ts) 提供了重要启示：

- prompt section 可按 section 名缓存
- cacheBreak section 与普通 section 被显式区分
- `/clear` 与 `/compact` 会重置 section 缓存状态

这种设计意味着系统并不只关心“提示内容是什么”，还关心“提示内容什么时候会改变缓存键”。

### 对 Rust 的直接启示

未来应把 prompt section 视为：

- 可独立计算
- 可独立缓存
- 可独立判定是否影响 cache key

的结构化对象，而不是随手拼接的文本块。

## 记忆体系是分层的

Claude Code 的记忆体系至少包含四层。

### 第一层：项目静态记忆

通过 CLAUDE.md 及相关 memory 文件，把项目规范、工作方式、常识性事实注入模型上下文。

### 第二层：相关记忆检索

[src/memdir/findRelevantMemories.ts](../src/memdir/findRelevantMemories.ts) 展示了系统并不是每次把所有记忆都塞进上下文，而是先扫描 header，再用 side query 选择最相关文件。这是“稀疏记忆装配”的设计，而不是“全量记忆灌入”。

### 第三层：Session Memory

[src/services/SessionMemory/sessionMemory.ts](../src/services/SessionMemory/sessionMemory.ts) 负责为当前会话自动维护一份结构化摘要。

它的关键机制包括：

- 初始化阈值
- 更新阈值
- 工具调用阈值
- 安全提取时机判断
- forked subagent 异步生成摘要

其本质是：

> 把当前长会话压缩成一份可继承、可再读、可继续演化的中间记忆体。

### 第四层：Auto Memory / Auto Dream

[src/memdir/memdir.ts](../src/memdir/memdir.ts) 与 [src/services/autoDream/autoDream.ts](../src/services/autoDream/autoDream.ts) 说明，系统还有更长期的自动记忆层：

- 自动维护 memory 目录
- 提供 MEMORY.md 入口索引
- 通过 Auto Dream 定期整合多会话信号
- 采用锁文件、时间门槛、会话数门槛和任务面板整合

这已经不是“聊天记录摘要”，而是长期知识资产管理系统。

## Auto Dream 的工程价值

从架构角度，Auto Dream 很值得单独强调。

它意味着系统承认以下现实：

1. 会话会积累长期知识
2. 长期知识不能只靠主会话即时维护
3. 记忆整合本身应是一个后台 agent 任务
4. 记忆整合需要锁、回滚、节流和显式可观测性

这与传统“把对话都存数据库”的做法完全不同。Claude Code 选择的是一种更接近“知识仓整理”的模型。

## 压缩体系不是单一 compact，而是分层治理

[src/services/compact/autoCompact.ts](../src/services/compact/autoCompact.ts) 说明系统至少具备以下几种上下文治理能力：

- token warning state
- auto compact
- manual compact
- reactive compact
- session memory compaction 优先尝试
- context collapse 互斥控制
- prompt cache break detection 协同

### 关键工程思想

1. 压缩不是最后一刻的紧急止血，而是持续预算管理。
2. 压缩并不是单算法，而是多策略协调。
3. 上下文治理要与模型上下文窗口、最大输出 token、预算阈值联动。
4. 压缩后必须伴随 cache、message、memory 等后处理。

### 这对 Rust 重构意味着什么

应该把 compact 设计为策略框架，而不是单个函数：

- `SessionMemoryCompact`
- `ConversationCompact`
- `ReactiveCompact`
- `ContextCollapse`

通过统一接口协调。

## Prompt Cache 稳定性是隐形核心能力

虽然仓库中到处是功能代码，但从 `QueryEngine`、`query.ts`、`forkedAgent.ts`、`services/api/claude.ts` 可看到另一个更深层设计目标：

> 尽可能让相同语义的请求生成相同或兼容的 cache-safe prompt 前缀。

具体体现为：

- query config 快照化
- system prompt section 缓存
- forked agent 复用 cache-safe params
- tool result replacement state 复制
- custom prompt 与 memory mechanics 的显式注入控制

这说明系统对长期运行成本极为敏感，也说明上下文工程直接关系到产品经济性。

## 上下文工程的五个设计原则

从实际代码中，可以提炼出五条稳定原则。

### 原则一：上下文是预算化资源

系统处处围绕 token 窗口、warning threshold、blocking limit、summary reserve 做设计。未来 Rust 内核应原生支持 token-aware scheduling。

### 原则二：上下文必须分层

静态事实、当前项目、当前会话、长期记忆、自治整理结果，不应混在一起管理。

### 原则三：压缩必须可恢复、可观测

压缩不是黑盒。要能知道为什么触发、压掉了什么、是否需要后续恢复或回放。

### 原则四：记忆生成应异步化

Session Memory 和 Auto Dream 都说明，把记忆维护从主交互路径中解耦是必要的。

### 原则五：缓存稳定性本身是一级工程目标

如果未来 Rust 版本忽略 cache-safe prefix 稳定性，性能和成本都会大幅倒退。

## Rust 重构建议

### 建议拆出的核心组件

1. `prompt-sections`
2. `context-builder`
3. `memory-index`
4. `memory-selector`
5. `session-memory-engine`
6. `dream-engine`
7. `compaction-engine`
8. `cache-key-stabilizer`

### 建议的内部抽象

- `PromptSection`
- `ContextLayer`
- `MemoryArtifact`
- `CompactionPolicy`
- `BudgetState`
- `CacheSafePrefix`

### 优先级建议

如果资源有限，Rust 化优先顺序应是：

1. token/budget 计算
2. message normalization 与 cache-safe prefix
3. compaction state machine
4. session memory / auto dream

## 本章结论

Claude Code 的上下文工程不是附属优化，而是系统真正的“长期智能维持层”。如果只重写 query loop，不重写上下文工程，新系统最多只是一个更快的工具调用器，不会成为真正成熟的 Code Agent。
