# 05. 执行与安全：权限、并发、隔离、恢复

## 本章中心结论

Claude Code 的安全性不是依赖模型自觉，也不是依赖 UI 层一个确认弹窗，而是依赖一个贯穿执行链的治理体系：

- 权限模式
- 规则装载
- 危险规则识别
- 并发调度分级
- 任务隔离
- worktree 与远程模式隔离
- 中断、重试、心跳、锁与恢复

从第一性原理看，这一层决定了系统是否真的可以在真实工程环境中长期运行。

## 权限不是功能开关，而是执行内核

[src/utils/permissions/permissionSetup.ts](../src/utils/permissions/permissionSetup.ts) 显示，权限系统并不只是“allow/deny”几个选项，而是一个策略编排中心。

### 权限系统要解决的真实问题

1. 模型是否可以触发某类工具
2. 工具在何种输入条件下可被放行
3. 自动模式下哪些规则会绕过安全审查
4. 背景代理能否触发交互式权限提示
5. 父会话与子代理的权限边界如何隔离
6. CLI、settings、policy、session 等多个来源的规则如何合并

因此，权限系统更接近 capability kernel，而不是配置项。

## 权限模式是运行语义的一部分

从 `ToolPermissionContext` 与权限 setup 逻辑看，权限模式至少影响以下方面：

- 是否自动放行
- 是否必须询问
- 是否可用 auto mode
- 是否允许 bypass
- 是否避免 permission prompts
- 是否在显示对话框前等待自动检查

这说明 permission mode 不只是 UX 选择，而是 query loop 与 agent/task 行为的重要输入参数。

### 对 Rust 的启示

建议把 permission mode 纳入显式执行状态，而不是做成工具层的附属布尔值。

## 危险规则识别是非常成熟的设计

`permissionSetup.ts` 对 Bash、PowerShell、Agent 工具做了危险规则识别，其根本逻辑是：

> 阻止用户通过过宽的 allow rule 绕过自动模式或分类器的安全门槛。

这个设计非常关键，因为它承认一个事实：

- 真正危险的不是“工具本身”
- 而是“能绕过策略评估的宽泛授权模式”

未来 Rust 版本也必须具备“规则级静态风险分析”，而不是只做运行时 permission check。

## 并发策略是显式调度，而不是 incidental async

[src/services/tools/toolOrchestration.ts](../src/services/tools/toolOrchestration.ts) 清楚说明了系统对工具并发的治理原则：

- 并发安全的只读工具成批执行
- 具有副作用或不安全的工具串行执行

这比“默认全并发”或“默认全串行”都更高级，因为它利用了两者优点：

- 读取与搜索操作获得吞吐
- 写入与环境改变操作保留确定性

### 为什么这不是小优化

对于 Code Agent，工具并发策略直接影响：

- 文件一致性
- 用户可理解性
- 权限提示顺序
- transcript 可重放性
- 错误恢复难度

因此并发策略应被建模为 runtime scheduling policy。

## 子代理与后台任务通过隔离控制风险

[src/utils/forkedAgent.ts](../src/utils/forkedAgent.ts) 与 [src/tools/AgentTool/runAgent.ts](../src/tools/AgentTool/runAgent.ts) 说明，子代理运行并不是主上下文的裸共享，而是带有隔离策略的派生上下文。

隔离内容包括：

- abort controller
- read file cache
- UI 控制权
- permission prompt 行为
- local denial tracking
- tool result replacement state

### 背景代理的关键约束

对于 async/background agents，系统默认倾向：

- 避免交互式权限提示
- 通过 root AppState 通道维护任务级基础设施
- 将结果以 notification 回注主会话

这是非常典型的“后台执行必须可审计但不可失控”的设计思路。

## 任务隔离与输出隔离

[src/Task.ts](../src/Task.ts) 与 [src/tasks/LocalAgentTask/LocalAgentTask.tsx](../src/tasks/LocalAgentTask/LocalAgentTask.tsx) 表明，任务输出并不只是内存对象，还会绑定磁盘输出文件和通知路径。

这意味着系统安全设计考虑了两个维度：

- 运行时状态隔离
- 观测与回放隔离

对新系统而言，这很重要，因为后台任务若只有内存状态，崩溃恢复和审计都会变弱。

## 恢复能力是执行安全的一部分

Claude Code 并不把恢复看成“附加特性”。从多个模块看，恢复能力直接嵌入执行模型：

- `withRetry` 和 non-streaming fallback
- session activity keepalive
- bridge token refresh
- lock file rollback
- remote reconnect
- session transcript flush

这说明系统非常清楚：

> 在真实工程环境中，失败不是异常，而是常态的一部分。

## 会话活动心跳体现了运行时成熟度

[src/utils/sessionActivity.ts](../src/utils/sessionActivity.ts) 的 refcount keepalive 模型很值得注意。它通过 API 调用和 tool execution 两类活动引用计数维持心跳，而不是粗暴地“每隔 N 秒发心跳”。

这种设计更成熟，因为它把 keepalive 绑定到真实工作负载，而不是盲目绑定到进程存活。

## 远程和桥接安全设计并未下沉到业务层

[src/bridge/bridgeApi.ts](../src/bridge/bridgeApi.ts)、[src/bridge/bridgeMain.ts](../src/bridge/bridgeMain.ts)、[src/bridge/remoteBridgeCore.ts](../src/bridge/remoteBridgeCore.ts) 展现出完整的远程安全机制：

- OAuth access token
- worker JWT
- trusted device token
- 401 refresh
- safe ID validation
- reconnect and epoch refresh

这里的重要架构价值在于：

- 协议安全并不散落在业务逻辑里
- transport 层承担了绝大部分身份、重连、续签责任

Rust 重构应保留这一分层，把认证/续签/重连放在 transport abstraction，而不是放到 query 内核。

## Feature Gate、USER_TYPE、Policy 是三层治理

虽然 [docs/07-feature-gates.md](../docs/07-feature-gates.md) 已有说明，但从工程角度它更应被视为安全治理与产品治理结构：

1. 编译时 feature 决定能力是否存在
2. USER_TYPE 决定不同构建身份边界
3. GrowthBook/Policy 决定运行时是否开放

这种三层治理的价值是：

- 最高风险功能可在构建时彻底裁掉
- 内部能力与外部能力可共仓共架构而不共暴露面
- 企业策略和灰度发布可在运行时快速收敛风险

## Rust 重构建议

### 必须独立成内核的模块

1. `permission-kernel`
2. `rule-analyzer`
3. `tool-scheduler`
4. `task-isolation`
5. `retry-and-recovery`
6. `transport-auth`

### 关键 trait 建议

- `PermissionEvaluator`
- `RuleRiskAnalyzer`
- `ConcurrencyClassifier`
- `TaskSandboxPolicy`
- `RetryPolicy`
- `SessionKeepalive`

### 关键数据结构建议

- `PermissionContext`
- `PermissionRuleSet`
- `ToolScheduleBatch`
- `TaskLifecycle`
- `RecoveryState`
- `AuthSession`

## 常见重构误区

1. 把权限逻辑塞进 UI 提示流程。
2. 把并发安全判断写死为工具白名单，而不是能力特征。
3. 让后台代理和前台代理共享完全相同的权限策略。
4. 把恢复机制视为外围工具，而不是内核状态的一部分。

## 本章结论

Claude Code 的可用性来自工具和模型的结合，但它的可部署性、可长期运行性与企业可接受性，来自这一整套执行治理机制。未来 Rust 重构若不先建立权限、并发、隔离、恢复四位一体的执行安全内核，很容易得到一个功能上“像” Claude Code、工程上却远不如它的系统。
