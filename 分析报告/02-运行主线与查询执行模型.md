# 02. 运行主线与查询执行模型

## 为什么这一章最重要

重构 Code Agent 时，最容易犯的错误是把系统理解成“一个调用模型 API 的 CLI”。Claude Code 的实际运行模式不是一次请求，而是一个持续闭环：

> 输入归一化 -> 上下文装配 -> 流式采样 -> 工具检测 -> 工具执行 -> 结果回注 -> 继续采样 -> 持久化与恢复

这个闭环才是系统的主引擎。没有它，就不会有真正的 agent 行为。

## 从进程启动到可交互状态

### 阶段一：CLI 分流与 fast path

[src/entrypoints/cli.tsx](../src/entrypoints/cli.tsx) 不是简单的 `main()` 包装器，而是承担入口路由器角色。

它做的事情包括：

- `--version` 这类零依赖快速路径
- MCP、Chrome host、daemon worker、bridge server 等特定模式的提前分流
- 只在必要时才加载完整 CLI 栈

这说明系统从入口处就按“多运行模式”设计，而不是单一交互路径。

### 阶段二：环境初始化

[src/entrypoints/init.ts](../src/entrypoints/init.ts) 把进程环境初始化为 agent 可运行环境。

其职责包括：

- 配置系统与受管环境变量应用
- 代理、mTLS、证书与 API 预连接
- 遥测与 GrowthBook 初始化
- 远程策略与 managed settings 预加载
- cleanup registry 注册

这一步的本质，是把“一个普通进程”变成“一个具备组织策略、网络能力、遥测和恢复能力的 agent host”。

### 阶段三：会话设置

[src/setup.ts](../src/setup.ts) 负责把抽象配置转换为具体会话上下文。

关键动作：

- 设置 cwd 与 project root
- hooks 配置快照
- worktree 创建与模式切换
- session memory 和 background jobs 初始化
- sink 和版本锁初始化

这是运行时从全局初始化进入“当前会话语义”的关键边界。

### 阶段四：主界面与状态树组装

[src/main.tsx](../src/main.tsx) 完成 REPL、headless、assistant、remote-control 等模式的最终装配：

- 装配命令与工具集合
- 初始化插件、技能、MCP、LSP
- 创建 AppState store
- 装配 REPL 与交互组件
- 分配不同模式下的运行逻辑

因此 `main.tsx` 的职责不是“渲染 UI”，而是“把运行时各平面粘合为一个工作系统”。

## 单次回合的执行链

### 总体状态机

```text
+-------------------+
|   Prepare Input   |
+-------------------+
                    |
                    v
+-------------------+
|   Build Context   |
+-------------------+
                    |
                    v
+-------------------+
|   Stream Model    |
+-------------------+
            |         |
            |         +------------------------------+
            |                                        |
            | terminal stop reason                   | assistant emits tool_use
            v                                        v
+-------------------+               +-------------------+
|    No Tool Use    |               |  Schedule Tools   |
+-------------------+               +-------------------+
                    |                                   |
                    v                                   v
+-------------------+               +-------------------+
|    Stop Hooks     |               |   Execute Tools   |
+-------------------+               +-------------------+
                    |                                   |
                    v                                   v
+-------------------+               +-------------------+
|      Persist      |               |  Append Results   |
+-------------------+               +-------------------+
                    |                                   |
                    v                                   |
                [END] <-------------------------------+
                                            loop back to Stream Model
```

### 1. QueryEngine 负责会话级编排

[src/QueryEngine.ts](../src/QueryEngine.ts) 的职责不是直接推理，而是负责“一个会话中的 query 生命周期管理”。

它维护以下会话级状态：

- mutableMessages
- abortController
- permissionDenials
- totalUsage
- readFileState
- turn-scoped skill/memory 跟踪集合

其核心方法 `submitMessage()` 做的事情包括：

- 固化本轮执行配置
- 构造 `ToolUseContext`
- 获取 system prompt / userContext / systemContext
- 执行用户输入归一化
- 调用 `query()` 进入主循环
- 汇总 usage、permission denial 和 transcript 持久化结果

因此 QueryEngine 是“会话级协调器”。

### 2. 输入预处理先于模型推理

[src/utils/processUserInput/processUserInput.ts](../src/utils/processUserInput/processUserInput.ts) 和相关模块表明，真正进入模型之前系统会做大量前处理：

- slash command 解析
- 附件展开
- pasted content 展开
- ultraplan 关键词触发
- hook 执行与阻断
- user message 标准化

这一步的意义是：模型看到的输入并不是原始终端字符串，而是结构化后的“会话事件”。

### 3. query() 是真正的回合内核

[src/query.ts](../src/query.ts) 中的 `query()` 与 `queryLoop()` 才是系统最核心的 agent loop。

它的核心思想是：

- 把本轮执行拆成一系列 iteration
- 每次 iteration 都围绕消息、工具上下文、压缩状态、预算状态进行状态转移
- 只有当模型不再要求工具执行，或达到终止条件时，整个回合才结束

这与普通聊天 API 最本质的区别在于：

- 聊天 API 一次返回文本
- agent loop 一次返回“可能继续执行”的状态

### 4. query config 被快照化

[src/query/config.ts](../src/query/config.ts) 将某些运行时门控和会话配置在 query 入口一次性快照，避免在同一轮执行中出现中途漂移。

这体现出一个重要工程原则：

> 对单轮 agent 执行而言，配置稳定性和可重现性比“实时读取最新全局状态”更重要。

Rust 重构时，这一点应该保留为不可变 `QueryConfig`。

### 5. 模型调用是流式事件源，而不是单次响应

[src/services/api/claude.ts](../src/services/api/claude.ts) 中 `queryModelWithStreaming()` 与内部 `queryModel()` 说明，模型层向上暴露的是事件流。

事件包括但不限于：

- stream request start
- stream delta
- assistant message
- system api error

这让上层可以：

- 实时显示输出
- 捕捉中间 usage
- 对错误和 fallback 做细粒度恢复
- 在工具调用边界截断并切换执行模式

### 6. 工具调用不是附属功能，而是回合的中间状态

当 assistant message 中出现 `tool_use` 时，query loop 不会把它视为终点，而会将其视为中间状态：

- 提取工具调用块
- 调用 `runTools()`
- 把工具结果变成消息再追加回消息流
- 进入下一轮采样

这说明“工具执行”在架构上属于推理回路内部，而不是外部插件。

### 7. 工具调度具有并发语义

[src/services/tools/toolOrchestration.ts](../src/services/tools/toolOrchestration.ts) 会先将工具调用分批：

- 并发安全的只读批次并行运行
- 非只读或不安全调用串行运行

这说明系统并不是简单按顺序执行工具，而是在构建一个轻量调度器。它的调度准则以“上下文安全”和“副作用隔离”为核心，而不是以工具类型枚举为核心。

### 8. 工具上下文在执行中可被修改

工具执行返回的不只是消息，还可能返回 context modifier。也就是说，工具执行会改变后续回合的上下文视角。

这是一条非常关键的设计信号：

> 工具不是纯函数，它们可能改变 agent 的执行环境。

因此 Rust 运行时需要显式建模“执行上下文突变”，不能把 tool call 简化成 `input -> output`。

### 9. 回合结束后仍有 stop hooks 和持久化阶段

Query loop 结束后并不是立刻结束回合，还会进入：

- stop hooks
- transcript record
- session storage flush
- usage 和 metrics 汇总

这说明回合结束也是一个工程阶段，而不是单点事件。

## 子代理如何复用主循环

### createSubagentContext 保留内核一致性

[src/utils/forkedAgent.ts](../src/utils/forkedAgent.ts) 中的 `createSubagentContext()` 说明，子代理不是另一套实现，而是在父上下文基础上构造隔离上下文：

- 可选择共享或派生 abort controller
- 默认克隆 file state cache
- 默认隔离 UI 回调
- 可共享或隔离 AppState 写入能力
- 重新生成 query tracking chain

这表明系统设计者追求的是“同一 agent kernel，多种执行隔离级别”。

### runForkedAgent 复用 query() 内核

`runForkedAgent()` 直接使用同一个 `query()` 进行子代理执行，并在外层增加：

- sidechain transcript 记录
- usage 累加
- cache-safe param 复用
- message 监听

换言之，主线程和子代理的差别主要在上下文隔离和持久化策略，而不是在核心执行算法上。

### runAgent 进一步封装 agent 级策略

[src/tools/AgentTool/runAgent.ts](../src/tools/AgentTool/runAgent.ts) 在 `query()` 之上再包装 agent 特定逻辑：

- agent model 选择
- worker 工具池解析
- permission mode 调整
- transcript 分组和恢复
- async 与 sync agent 的交互差异

这说明“Agent”只是 query kernel 的一个高阶应用层，而不是另一个推理引擎。

## 运行时不变量

从执行链看，可以抽取出八条对 Rust 重构极其重要的不变量。

1. 单次用户请求不等于单次模型调用，而是一个多 iteration 回路。
2. message log 是整个系统的主事实来源，工具与任务都围绕它增量演化。
3. 工具执行结果必须以消息形式回注回路，而不是旁路更新状态后悄悄继续。
4. query config 在单轮执行中应保持稳定，不应被中途动态门控扰动。
5. 子代理必须复用主循环，而不是再造另一套不一致的执行栈。
6. 工具调度必须显式区分只读并发与副作用串行。
7. 中断、失败、fallback、non-streaming recovery 都属于同一运行时状态机的一部分。
8. 回合结束后仍有必须执行的后处理阶段。

## 对 Rust 重构的直接启示

### 不要先写 CLI，先写内核状态机

正确顺序应该是：

1. Message IR
2. QueryConfig
3. QueryState
4. Tool scheduler
5. Stream event reducer
6. Persistence hooks

CLI、TUI、HTTP、desktop 只是这个内核的宿主层。

### 建议把 query loop 做成显式 reducer

目前 TypeScript 实现虽清晰，但仍偏命令式。Rust 重构可以把它提升为：

- `State`
- `Config`
- `Event`
- `Transition`

通过 reducer 统一建模，收益包括：

- 更强的可测试性
- 更容易做 deterministic replay
- 更适合持久化与崩溃恢复
- 更容易支持多宿主执行

## 本章结论

Claude Code 的真正核心不在命令，不在 UI，而在这个可以持续自我推进的 query loop。任何新一代 Rust Code Agent，如果不能先把这条主循环设计好，就不可能具备 Claude Code 这类系统的真实能力密度。
